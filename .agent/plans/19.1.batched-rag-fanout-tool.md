# 19.1 Batched RAG Fan-Out Tool

⚠️ **Medium** — single new tool + instruction tweak + recall bump. Risk concentrated in tool routing (orchestrator must pick `dispatch_questions` over N parallel `rag_specialist` calls reliably). Single-pass executable.

## Problem (empirical + architectural)

**Empirical (verified 2026-05-02 in Vertex Agent Builder against Südi-Areal corpus):**
- Single-question call returned `Bausumme = 39'114'000` and named all four TP-Leiter (Pascal Ryser, Thomas Kieliger, Silvia Bucher, Luca Nosetti).
- Same Agent Builder, same corpus, same model with 11 batched questions returned `"Bausumme nicht genannt"` and `"namentlicher Projektleiter nicht genannt"`. Facts that ARE retrievable on focused queries vanish under batching.
- Thinking trace on the batched call shows ~20 near-identical "structuring response" iterations — model spinning on output formatting after one shallow retrieval, not 11 distinct retrieval rounds.

**Architectural (per `google/adk-python` `vertex_ai_rag_retrieval.py`):**
- `VertexAiRagRetrieval` on Gemini 2+ injects native `types.Tool(retrieval=Retrieval(vertex_rag_store=...))` into the LlmAgent's `GenerateContentConfig`.
- Native grounding does **one server-side retrieval per `generate_content` call**, against a single query string the model auto-derives from the prompt. Top-K chunks come back, blended across whatever topics the prompt covered.
- Multi-question prompts → one fused fuzzy query → top-K window diluted across topics → per-question recall collapses. Raising `similarity_top_k` cannot fix this — it's still one query.

**Why our existing fan-out doesn't save us:**
- `CHAT_ORCHESTRATOR_INSTRUCTION` already specifies parallel fan-out (Schritt 0, Beispiele A–F, "12 parallele rag_specialist-Aufrufe in DERSELBEN Antwort").
- `streaming_agent_tool.py` already supports per-call grounding chunk append with global idx offsets (`agent_grounding_chunks`, `_annotate_with_grounding_supports(idx_offset=existing_count)`).
- BUT: the fan-out depends on Gemini 2.5 Flash emitting **parallel function calls** in one model turn. Empirically (Google AI forum thread on Gemini 2.5/3 Flash), the model often serialises tool calls or — worse — batches the multi-question prompt into a single `rag_specialist(request="<all 11 questions>")` call, which then fails the same way Agent Builder's batched call fails.

## Goal

Make multi-question chat turns deterministically fan out into N independent `rag_specialist` runs via a single explicit tool call. Each sub-question gets its own focused retrieval, its own thinking budget, its own citation indices. The Frage X / Antwort X format the user wants for templated batches becomes a natural by-product of N independent retrievals.

## Out of scope

- `question_planner` agent emitting a typed DAG (rag vs web vs both per node) — tracked as project memory `project_planner_agent_followup`. This plan is the explicit-list stepping stone.
- `projektanalyse_v2` — different code path, must NOT auto-escalate (memory `feedback_no_v2_autoescalation`).
- `StreamingAgentTool` flattening (line 140-142). Current behaviour — strip `p.thought` from final tool_result, capture thoughts separately into `agent_trace` for the activity panel — is correct for fan-out. Do not touch.
- `web_researcher` is unaffected. It stays a per-question tool.

## Tasks

### T1 — `dispatch_rag_questions` FunctionTool

**File:** new `backend/app/adk/dispatch_rag_questions_tool.py`.

Build a `FunctionTool` that takes a list of in-sich-geschlossene questions and runs N parallel `rag_specialist` invocations.

Signature:

```python
async def dispatch_rag_questions(
    questions: list[str],
    tool_context: ToolContext,
) -> dict:
    """Dispatch N independent RAG questions in parallel.

    Use this whenever the user turn contains 2+ distinct factual questions.
    Each entry in `questions` MUST be self-contained (no pronouns, no
    cross-references). The tool runs each question through rag_specialist
    concurrently with its own session, retrieval, and citation budget.

    Returns: {"answers": [{"question": str, "answer": str}, ...]}
    Per-question grounding chunks are appended to parent state under
    `agent_grounding_chunks` in dispatch order, with [N] markers in each
    answer text already offset to global indices.
    """
```

Implementation requirements:
- Resolve the corpus-bound `rag_specialist` factory the same way `chat_orchestrator` does — accept a `rag_specialist_factory: Callable[[], LlmAgent]` at module construction so we don't re-bind the corpus per call.
- For each question, construct a fresh `Runner` with its own `InMemorySessionService` session (mirror `StreamingAgentTool.run_async` lines 93-112). Sessions MUST be independent so concurrent retrieval state cannot race.
- Use `asyncio.gather(*[_run_one(q, idx) for idx, q in enumerate(questions)], return_exceptions=True)` with `return_exceptions=True` so a single sub-question failure does not poison the whole batch. On exception, return `{"question": q, "answer": "<Fehler beim Abruf — bitte erneut versuchen.>"}`.
- Per sub-call, accumulate grounding chunks into a **local** list keyed by sub-call order, then append to `tool_context.state["agent_grounding_chunks"]` in deterministic order (sub-call 0 → 1 → ... → N-1) so global `idx_offset` math stays correct. Annotate each sub-call's answer text with `[N]` markers using `_annotate_with_grounding_supports` BEFORE its chunks are appended (same invariant as `StreamingAgentTool` line 153-159).
- Per sub-call, propagate `agent_trace` entries via `_capture_activity` so the activity panel shows N parallel rag_specialist branches in dispatch order.
- Return shape: `{"answers": [{"question": str, "answer": str}, ...]}` — a list of pairs the orchestrator can stitch into Frage X / Antwort X.

**Wiring:** add to `make_chat_orchestrator` in `backend/app/adk/agents.py`:

```python
tools=[
    StreamingAgentTool(
        agent=rag_specialist,
        propagate_grounding_metadata=True,
    ),
    make_dispatch_rag_questions_tool(rag_specialist),  # NEW
    StreamingAgentTool(agent=web_researcher),
    run_projektanalyse_v2_tool,
],
```

Pass the same `rag_specialist` instance to both the `StreamingAgentTool` (single-question path) and the dispatch tool (multi-question path) so the corpus-bound retrieval tool is constructed once per orchestrator instance.

**Validation:**
- Unit test `backend/tests/test_dispatch_rag_questions_tool.py`:
  - Mock `rag_specialist` to return `f"Antwort {i}"` after `random.uniform(0.05, 0.2)s`. Call `dispatch_rag_questions` with 5 questions. Assert wallclock < 0.5s (parallelism check).
  - Assert returned `answers` list preserves input order.
  - Assert one sub-call raising `RuntimeError` does NOT abort others; resulting answer string contains the fallback marker.
  - Assert `agent_grounding_chunks` global ordering matches sub-call order when each mock returns 2 chunks.

## Risk register

1. **Orchestrator picks the wrong tool** (calls `rag_specialist` 11 times in parallel anyway, or worse, calls `rag_specialist(request="<all 11 questions>")`). Mitigation: T2 worked examples B and D explicitly route 2+ → `dispatch_rag_questions`; T4 e2e test asserts the tool call shape. If Flash still mis-routes, fallback is a server-side "if `rag_specialist.request` contains a numbered list or 2+ `?` markers, error and tell the model to use dispatch" guardrail in `chats.py`. Track separately if it bites.
2. **Race on `tool_context.state` parallel writes.** `asyncio.gather` runs sub-calls concurrently. They all want to append to `agent_grounding_chunks` and `agent_trace`. Mitigation: each `_run_one` accumulates LOCALLY, gathers complete, then the dispatch tool serially merges results into parent state in deterministic order. No concurrent state mutation.
3. **Citation index drift across the dispatch + a follow-up `rag_specialist` call in the same turn.** `StreamingAgentTool._annotate_with_grounding_supports(idx_offset=existing_count)` already handles this for serial calls. The dispatch tool must use the SAME state key (`agent_grounding_chunks`) and the SAME offset math, which T1 enforces. Validation: T1 unit test checks global ordering.
4. **Latency regression at top_k=20.** Vertex RAG p50 retrieval is ~250-400ms; doubling top_k typically adds 100-200ms. For the single-question path this is acceptable; for `dispatch_rag_questions` it's parallel so wallclock is unchanged. Worst case is rolling back to top_k=15 if user complains about p95 chat latency.
5. **DSQ 429 storms during fan-out.** 11 concurrent retrievals share the same DSQ pool. Existing `_HTTP_OPTIONS` retry (1s/2s/4s exp backoff, 4 attempts) absorbs this — verified working post-region-switch (commit 3cc6f47). No new mitigation needed; if it bites, add per-corpus concurrency cap as follow-up.
