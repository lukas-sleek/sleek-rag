# 14. Agentic Retrieval — Tool-Driven Search Replaces Regex Router

## Complexity: 🔴 Complex

Architectural shift. The current chat path regex-routes the user query *before* the LLM ever sees it (`retrieval.py:60`), then stuffs all hits into the prompt. The model has no agency over what gets retrieved, "im Dokument A" is dropped on the floor because there's no regex for filename hints, and the meta SSE frame is emitted before the model decides anything — so 14 unfiltered chunks render as citation chips while the answer is still being generated.

After this plan: the model decides what to retrieve via a `search_chunks` tool with structured filters. Backend executes the tool, feeds results back, model answers. Citations meta frame is emitted *after* the answer text, and only contains chunks the model actually used as evidence.

---

## Goal

Replace the regex retrieval router in `_send_message_stream` (`backend/app/routers/chats.py`) with a multi-turn agent loop:

1. Inject a **per-project file inventory** into the system prompt so the model can resolve "Dokument A" → `file_id` natively.
2. Expose a single **`search_chunks` tool** with structured filters (`query`, `file_ids`, `page`, `figure_label`, `section`, `block_type`, `top_k`).
3. Run an **agent loop** (cap 3 iterations) that streams Gemini, intercepts tool calls, executes them, and feeds results back.
4. Emit the **citations meta frame after** the model finishes — citations contain only chunks returned by tool calls, not pre-LLM guesses.
5. Delete the four regex branches in `retrieval.retrieve()`. Their SQL helpers stay and become tool implementations.

---

- Plans 10–13 complete. Existing `document_chunks`, `chunk_images`, RPCs (`match_chunks`, `chunks_by_heading_prefix`) all stay.

## Architectural decisions

### File inventory in system prompt

Project files are listed once per turn in the system prompt — cheap, deterministic, prompt-cache-friendly. Format:

```
Verfügbare Dokumente in diesem Projekt:
- file_id=abc12345  HO_Teil_A_Südi-Areal-Infrastruktur_def.pdf  (8 Seiten)
- file_id=def67890  HO_Teil_B_Südi-Areal-Infrastruktur_def.pdf  (12 Seiten)
- ...

Verwende file_ids in `search_chunks`, wenn der Nutzer ein Dokument namentlich nennt.
```

Use the **8-char file_id prefix** in the visible inventory to keep tokens down; resolve back to full UUID in the tool handler.

                },
                "figure_label": {

                },

### Tool result format (returned as `tool` role message)

```json
{
  "results": [
    {
      "ref": 1,
      "chunk_id": "uuid",
      "file_id": "abc12345",
      "filename": "HO_Teil_A_Südi-Areal-Infrastruktur_def.pdf",
      "page_start": 5,
      "page_end": 5,
      "figure_label": null,
      "block_type": "paragraph",
      "similarity": 0.87,
      "excerpt": "3.4 ZUSCHLAGSKRITERIEN — Es wird erklärt..."
    }
  ]
}
```

`ref` is a 1-indexed reference for the model to cite as `[1]`, `[2]`, …

### Tool dispatch logic (inside the handler)

The single tool internally routes by which params are filled — same SQL helpers as today, just reordered:

| Filter combination | Path |
|---|---|
| `figure_label` set | `_by_figure_label` (exact match), then vector-rank within hits |
| `page` set | `_by_page`, then **rank by `(page_end == page) DESC, (page_end - page_start) ASC, similarity DESC`** — exact-page chunks beat narrow-span chunks beat wide-span chunks |
| `section` set | `_by_heading`, then vector-rank |
| only `query` | pure vector via `match_chunks` |

`file_ids` (when present) becomes a `WHERE file_id IN (...)` filter applied to all branches. `block_type` likewise.

**Always compute the query embedding once** and use it to rank hits in every branch. Eliminates the hardcoded-1.0 placeholder scores; LangSmith traces show real similarity for every result.

### Page ranking rule (Domain note)

Document AI emits `page_start` and `page_end`. When a chunk truly belongs to a single page, `page_start == page_end`. A chunk with `page_start=1, page_end=14` "happens to overlap" page 5 but is rarely *about* page 5. The page tool's ranking enforces:

1. Exact match (`page_start == p AND page_end == p`) — score boost +1.0
2. Narrow span (`page_end - page_start <= 1`) — kept as-is
3. Wide span (>1 page) — sorted last by similarity within that bucket

### Agent loop

```
build_messages (system + file inventory + history + user)
tools = [search_chunks, run_projektanalyse, run_projektanalyse_v2]
collected_tool_results = []   # for the final citations meta frame

for iteration in 1..MAX_TOOL_ITERATIONS (=3):
    stream = gemini.chat.completions.create(messages, tools, stream=True)
    parts, tool_calls = consume_stream(stream)

    if no tool_calls:
        # plain text response, stream deltas to client as we go
        emit done frame
        emit meta frame with collected_tool_results
        persist assistant message
        return

    if tool_calls includes run_projektanalyse{,_v2}:
        existing handoff (already covered)
        return

    for tc in tool_calls (search_chunks):
        result = execute_search_chunks(tc.arguments)
        collected_tool_results.extend(result["results"])
        messages.append({"role": "assistant", "tool_calls": [tc]})
        messages.append({"role": "tool", "tool_call_id": tc.id, "content": json.dumps(result)})

# loop cap reached without final answer:
emit warn frame, persist warn message
```

`MAX_TOOL_ITERATIONS = 3` to prevent runaway. Each iteration's Gemini call streams to the client only on the *final* iteration (when no more tool calls arrive). Earlier iterations are silent — model is "thinking".

### SSE frame timeline (new)

```
1. (immediate)  none — frontend already shows loading dots
2. (per delta)  data: {"type":"delta","content":"..."}
3. (after final)  data: {"type":"meta","citations":[...]}
4. (after meta)  data: {"type":"done","message_id":"..."}
```

Note: meta is now **after** all deltas. Frontend must handle meta arriving last (it currently handles meta arriving first — small change).

### Citation source

`citations` is built from `collected_tool_results`, deduped by `chunk_id`, in the order they appeared. This means **only the chunks the model actually retrieved** become chips — not 14 unfiltered hits.

### What gets deleted

- `retrieval.retrieve()` — the regex router function (`backend/app/retrieval.py:59-92`). Replaced by the tool handler.
- The four regex constants `_PAGE_RE`, `_FIGURE_RE`, `_SECTION_RE`, `_VISUAL_RE`.

### What stays

- `_by_figure_label`, `_by_page`, `_by_heading`, `_by_vector`, `_attach_images` — all repurposed as tool sub-implementations.
- `RetrievedChunk` dataclass, `to_citation()` — used by the tool result builder.
- Projektanalyse v1/v2 — they keep using `retrieve()` for now. **Migrate them in a follow-up plan**; out of scope here.
- Existing SQL RPCs (`match_chunks`, `chunks_by_heading_prefix`) — unchanged.

### Backwards compat

- Frontend SSE handler reorders trivially (citations after deltas instead of before).
- Empty-content+citations bug from earlier today disappears organically.
- Old chats in `chat_messages` keep their `citations` JSON — read path unchanged.

### T2. `search_chunks` tool definition + handler

**File**: new `backend/app/tools/search.py` (create `tools/__init__.py` empty)

- Define `SEARCH_CHUNKS_TOOL` constant (the schema above).
- Function `execute_search_chunks(*, args: dict, project_id: str, user_id: str) -> dict` that:
  - Resolves 8-char `file_ids` prefixes back to full UUIDs (or returns empty if no match).
  - Always computes `query` embedding via `gemini_client().embeddings.create(...)`.
  - Routes by which optional filter is present (figure_label / page / section / pure-vector), applies `file_ids` and `block_type` filters when present.
  - Applies the page-ranking rule for `page` lookups (exact-match boost, narrow-span preference, wide-span demotion).
  - Caps results at `top_k` (default 8, range-clamped 1-20).
  - Calls `_attach_images` to populate `image_path`.
  - Returns `{"results": [...]}` with `ref` indices.

### T3. Agent loop in `_send_message_stream`

**File**: `backend/app/routers/chats.py`

Replace the body of `_send_message_stream` (currently `chats.py:194-355` after the error-handling work) with:

1. Persist user message (unchanged).
2. Load history (unchanged).
3. Build inventory block via T1.
4. Build initial messages: `[system_with_inventory, *history, {"role":"user","content":text}]`.
5. Loop up to `MAX_TOOL_ITERATIONS = 3`:
   - Call Gemini with tools `[SEARCH_CHUNKS_TOOL, PROJEKTANALYSE_TOOL, PROJEKTANALYSE_V2_TOOL]`.
   - Consume stream:
     - If `run_projektanalyse{,_v2}` triggered → existing handoff (unchanged).
     - If `search_chunks` tool calls → execute each, append assistant message with `tool_calls`, append `tool` message with result. **Do NOT stream deltas to the client during this iteration.** Continue loop.
     - If pure text → stream deltas to client (`type:delta` frames). Break loop after `done`.
6. After break: emit `meta` frame with `citations` built from accumulated `collected_tool_results` (deduped by `chunk_id`). Persist assistant message with `citations`. Emit `done` with `message_id`.
7. If loop hits cap without text response: emit user-friendly warn delta, emit `done`, persist warn message.

Existing error-wrapping (try/except around `.create()` and stream iteration) stays — same vendor-neutral message on failure.

**Test**: integration test with test user's seeded project.
- "was steht auf seite 5 im dokument a?" → expect tool call with `file_ids=["<A's prefix>"]` and `page=5`, expect ≤3 citations all from Document A, expect answer text.
- "Hallo" → expect no tool call, plain answer, empty citations.
- Tool storm guard: ensure cap is hit gracefully if the model loops.

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| +1 LLM round-trip per turn (latency) | File inventory + system prompt are cache-stable; only the user-message tail is fresh. Gemini 2.5 Flash hot-cache hits on prefix. Real impact: ~1-2s additional first-token latency on agent-loop turns; turns that don't need retrieval are unchanged. |
| Model fails to call tool when it should | Tool description aggressively pushes "Rufe dieses Tool auf, sobald die Frage Inhaltliches aus den Dokumenten betrifft." Validate with test #4 above. |
| Model calls tool with wrong file_id prefix | Resolver returns empty result; second iteration the model sees "results: []" and either retries with different filter or answers without that doc. |
| Loop runs 3 times unnecessarily | Iteration cap saves us. Each iteration logged via LangSmith for debugging. |
| Existing chat history breaks | None — `chat_messages` schema unchanged. Only the per-turn pipeline changes. |
| Projektanalyse path uses old retrieve() that we're keeping | Yes — explicit in T6. Tracked for plan 15. |
