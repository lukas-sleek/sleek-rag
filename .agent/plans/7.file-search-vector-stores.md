# 7. File Search + OpenAI Vector Stores

⚠️ **Medium** — Throwaway code by design (M2 replaces with pgvector).

## Context

`ProjectFilesModal` (374 lines) is a fully styled drag-and-drop UI that today does nothing — `SAMPLE_FILES` from `fixtures.ts` populates the list. To complete M1's "managed RAG" promise, we wire it to OpenAI Vector Stores: each project gets a vector store on first upload; uploaded files become OpenAI File objects attached to that store; the chat endpoint from sub-plan 6 enables `file_search` tool with the project's vector store id so retrieval flows end-to-end. M2 will rip this out and replace with pgvector.

## Goal

User uploads a PDF in `ProjectFilesModal` → row appears with `status=pending` → flips to `indexed` once OpenAI finishes processing → asking a question about the file content in chat returns an answer that cites the document.

## Prerequisites

- Sub-plan 6 landed (chat streaming works).
- OpenAI key has Vector Stores access (it does by default for paid accounts).

## Design Notes

- **Lazy vector store creation:** Don't pre-create on project create. On first file upload, if `projects.openai_vector_store_id` is null, create one (`vector_stores.create(name=project.name)`) and write the id back.
- **Upload flow:** Frontend `POST /api/projects/{id}/files` (multipart) → backend uploads to `files.create(purpose="assistants")` → attaches to vector store via `vector_stores.files.create(vector_store_id, file_id=...)` → row inserted with `status="pending"` and the OpenAI `file_id`.
- **Status sync:** Vector store file processing is async. Either poll `vector_stores.files.retrieve(...)` from the frontend on a 3s interval until `status="completed"`, or have the upload route block via `vector_stores.files.create_and_poll(...)` and only return when indexed. **Recommend create_and_poll for M1** — simpler, no realtime, fits "managed" framing. M2 introduces Supabase Realtime for non-blocking status.
- **file_search tool wiring:** In `chats.py` send_message, if the project has an `openai_vector_store_id`, attach:
  ```python
  tools=[{"type": "file_search"}],
  attachments=None,  # vector store is attached to the conversation
  ```
  Actually for Responses API, pass via `tools=[{"type":"file_search","vector_store_ids":[vs_id]}]`. Check current SDK signature when implementing.

## Tasks

### 7.1 Files router

Create `backend/app/routers/files.py`:

- `GET /api/projects/{project_id}/files` — list rows from `project_files` for this project + user.
- `POST /api/projects/{project_id}/files` — multipart upload. Steps:
  1. Verify project belongs to user.
  2. If `openai_vector_store_id` null, create vector store and update project row.
  3. Stream upload to `client.files.create(file=(filename, content), purpose="assistants")`.
  4. Call `client.vector_stores.files.create_and_poll(vector_store_id, file_id=openai_file.id)` (blocking until indexed or failed).
  5. Insert `project_files` row with `status` from poll result.
  6. Return the row.
- `DELETE /api/projects/{project_id}/files/{file_id}` — remove from vector store, delete OpenAI file, delete row.

Wire into `main.py`.

**Validation:**
```
curl -H "Authorization: Bearer $TOKEN" -F "file=@README.md" \
  http://localhost:8000/api/projects/<id>/files
```
Returns a row with `status="indexed"` (or `"failed"` with reason). Subsequent GET shows it.

### 7.4 Enable file_search in chats router

In `backend/app/routers/chats.py`, before calling `responses.stream`:

```python
project = supabase().table("projects").select("openai_vector_store_id").eq("id", chat["project_id"]).single().execute().data

    tools.append({"type": "file_search", "vector_store_ids": [project["openai_vector_store_id"]]})

with openai_client().responses.stream(

    input=[{"role": "user", "content": body.text}],
    conversation=chat["openai_thread_id"],
```

## Validation

1. Open `ProjectFilesModal` on a fresh project → empty list.
2. Drop a small PDF → optimistic row → flips to `indexed` after a few seconds.
3. Reopen modal → list persists across page reloads.
4. Send a chat message: "Summarise the document I uploaded." → response references the doc content.
5. Delete the file → row disappears, follow-up chat question can no longer cite it.
6. Second user uploads to their project → first user's vector store is unaffected (RLS).
7. `tools=[{"type":"file_search",...}]` is omitted when no files are uploaded yet (don't pass an empty vector store).
