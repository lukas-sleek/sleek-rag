# 17.5 Parallel Tool Dispatch + Rerank Metadata + Phrase-Aware FTS + Page Defrag

## Complexity: ⚠️ Medium

Plan 17.4 fixes the rater + retry-target gaps (P0). This plan ships P1 — the four mechanical retrieval fixes that target chunk-boundary loss, structural-chunk burial in rerank, phrase-search false-negatives, and the sequential-execution disincentive that makes the model avoid emitting parallel tool calls.

These four are independent; each has its own task and test. Order doesn't matter inside this plan, though F5 (parallel dispatch) is the cheapest and highest-value sit-back-and-watch change.

This plan only ships if 17.4's UAT shows the rater-rubric/outline-first changes alone don't close the gap to ≥9/11 — specifically when failures cluster around Q5 (Bausumme Total chunk-boundary), Q11 (phrase-search false-negative), Q3/Q5/Q10 (rerank burying short fact chunks), or Q2/Q6 (aggregation under-recall).

---

## Background — what 17.4 doesn't touch

17.4 fixes WHEN to retry and WHAT the rater greenlights. It does NOT change:

- **R4 — sequential tool execution** in `chats.py:615-639`: when Gemini emits multiple parallel `tool_calls` in one response, the loop dispatches them in a serial `for slot in retrieval_calls: await asyncio.to_thread(...)`. The model has no speed incentive to emit parallel calls — the latency is the same as sequential ones, so it usually emits 1-2 then waits. Affects Q2/Q4/Q6.
- **R5 part 1 — rerank blind to chunk structure**: Vertex Ranking sees only 280-char excerpts (`search.py:471`), no `block_type`, `heading_path`, `figure_label`, or `page_start`. Table-headline rows and figure captions are short and lose token-density battles to verbose paragraphs. Affects Q3 (Abb.15 caption), Q5 (Tabelle-2 Total row), Q10 (concrete-Mandate sentence).
- **R5 part 2 — FTS doesn't do phrase matching**: `match_chunks_hybrid:89` uses `plainto_tsquery('german', p_query)` which breaks multi-word queries into OR-of-tokens. A query like `"Ist in einer späteren Phase zu Detaillieren"` becomes `'ist'|'spät'|'phas'|'detailli'` — matches anything with any of those words, ranks pure phrase hits no higher than incidental keyword hits. Affects Q11.
- **R5 part 3 — chunk-boundary loss on multi-chunk facts**: when the answer to "Was ist die Bausumme?" is the headline row of Tabelle 2 but the model retrieves the per-Etappe sub-rows from the same table, there's no cheap way to fetch the rest of the table without re-searching. `read_section` with a heading prefix may include the headline if the heading_path matches, but the model has to know this. Affects Q5 (Tabelle 2), Q10 (Mandate-Liste paragraph).

---

## Goal

After this plan, against the same 11-question UAT (assumed already ≥9/11 from 17.4):

- **Aggregation answer recall** (Q2/Q6): completeness improves because the model sees parallel calls actually parallelize (2-3× wall-clock for 3 calls vs sequential), creating a behavioral incentive to fan out. Concrete: ≥3 distinct entities surfaced for "welche Bauherren" / "welche Drittprojekte" without needing the rater nudge to fire.
- **Structural chunks survive rerank** (Q3/Q5/Q10): chunks of `block_type ∈ {table, figure, list}` reranked with structural prefix can land in the top-K alongside paragraph chunks, even when shorter. Q5 specifically: Tabelle-2 headline row appears in retrieved chunks, not just sub-rows.
- **Phrase queries hit phrase matches** (Q11): a 4+ token quoted/phrase-shaped query routes through `phraseto_tsquery` and the load-bearing chunk surfaces in the top-15.
- **Page-level defrag works**: when `read_section` is called on a section spanning a multi-chunk table, the model can opt into "give me everything on these pages in document order" and the headline row + sub-rows arrive together.

### T1 — F5: Parallel tool dispatch via `asyncio.gather`

**File**: `backend/app/routers/chats.py`

The current loop body (around `chats.py:615-639` in pre-17.4 numbering — adjust per actual line after 17.4 lands):

```python
for slot in retrieval_calls:
    # ... resolve executor by name ...
    result = await asyncio.to_thread(executor, args=args, ...)
    # ... append to assistant_tool_calls / tool_messages ...
```

becomes:

```python
async def _dispatch_one(slot, ref_offset_local: int) -> tuple[dict, list[RetrievedChunk], int, str]:
    """Dispatch a single tool call. Returns (tool_message_dict, chunks_added,
    refs_consumed, tool_call_id). Each call gets a SEPARATE ref_offset slice
    so refs across parallel calls don't collide — the offsets are reconciled
    after gather() completes."""
    # ... existing per-tool dispatch logic, but pass ref_offset_local ...
    result = await asyncio.to_thread(executor, args=args, ref_offset=ref_offset_local, ...)
    # ... return tuple ...

# Pre-allocate ref_offset slices: each call gets a 50-ref window. Reconciled
# down to actual usage after gather. (Top_k is 25 max per search_chunks; 20
# per read_section; outline returns no refs. 50 is a safe upper bound per
# call.)
SLICE = 50
slices = [ref_offset + i * SLICE for i in range(len(retrieval_calls))]

results = await asyncio.gather(
    *(_dispatch_one(slot, sl) for slot, sl in zip(retrieval_calls, slices)),
    return_exceptions=False,  # let one tool's exception propagate; existing
                              # error path handles it
)

# Compact the ref windows back to contiguous numbering:
final_results = []
running_offset = ref_offset
for tool_msg, chunks_added, refs_consumed, tc_id in results:
    # Renumber refs in the tool result JSON from sliced to running.
    # (Each result["results"][i]["ref"] was assigned ref_offset_local + i + 1;
    # we rewrite to running_offset + i + 1.)
    rewritten = _rewrite_refs(tool_msg, from_base=tool_msg_base, to_base=running_offset)
    final_results.append(rewritten)
    collected_chunks.extend(chunks_added)
    running_offset += refs_consumed
ref_offset = running_offset
```

Key details:

1. **Ref isolation**: each parallel call gets a non-overlapping ref slice during execution, then results are renumbered to contiguous space after gather. Avoids races on the shared `ref_offset` counter.
2. **Order preservation**: the gather output preserves call order, so re-numbering is deterministic — `[1]` always means the first call's first chunk.
3. **Citation accumulator (`collected_chunks`)** stays single-threaded — extended after gather completes, in call order.
4. **Error handling**: `return_exceptions=False` keeps the current exception-propagates-up behavior. If one of three parallel calls raises, the whole iteration falls through to the existing exception handler in `chats.py`. (We could go softer with `return_exceptions=True` and log per-call failures, but that's a 17.6 polish.)
5. **The structured-error path from 17.3 T2 still fires**: if all parallel calls return `error` envelopes, `turn_errors == turn_call_count` triggers the TOOL-RETRY system message and force-tool. Behavior unchanged for the all-errored case.

**Test** (extend `test_chat_force_tool.py`): mock 3 parallel `search_chunks` tool calls in one Gemini response; assert all 3 executors are invoked concurrently (use `asyncio.Event` per executor to verify start-before-finish overlap); assert final refs are contiguous `[1..N1+N2+N3]` in call order; assert `collected_chunks` length matches the sum.

### T2 — F6: Rerank chunk-type prefix

**File**: `backend/app/ranking_client.py`, `backend/app/tools/search.py`, `backend/app/retrieval.py` (small)

**Subtask T2a**: Compose a structured prefix per chunk before sending to Vertex Ranking. The prefix encodes block_type + a short heading-path tail + figure_label when present:

```python
def _format_for_rerank(chunk: RetrievedChunk) -> str:
    """Render a chunk for the Vertex Ranking API document field. Adds
    structural metadata as a leading bracket-tagged context so the
    semantic-ranker can score short structural chunks (table headlines,
    figure captions, list items) on equal footing with verbose paragraphs.

    Format:
      [TYPE | SECTION | LABEL] body…

    Where TYPE ∈ {ABSATZ, TABELLE, ABBILDUNG, AUFZÄHLUNG, ÜBERSCHRIFT}.
    SECTION is the last entry of heading_path (max 60 chars). LABEL is
    figure_label if present, else omitted.

    Example:
      [TABELLE | 4.2 Grobkostenschätzung | Tabelle 2] CHF 39'114'000 (exkl. MwSt.)
    """
    type_map = {
        "paragraph": "ABSATZ",
        "table": "TABELLE",
        "figure": "ABBILDUNG",
        "list": "AUFZÄHLUNG",
        "heading": "ÜBERSCHRIFT",
    }
    type_tag = type_map.get((chunk.block_type or "").lower(), "ABSATZ")
    section_tag = ""
    if chunk.heading_path:
        section_tag = chunk.heading_path[-1][:60]
    label_tag = chunk.figure_label or ""
    head_parts = [type_tag]
    if section_tag:
        head_parts.append(section_tag)
    if label_tag:
        head_parts.append(label_tag)
    head = "[" + " | ".join(head_parts) + "]"
    return f"{head} {chunk.content}"
```

`heading_path` and `block_type` come from 17.4 T2's migration `0016` and the `RetrievedChunk` field additions. This task assumes those landed.

**Subtask T2b**: Replace the call site in `tools/search.py` where rerank input is constructed:

```python
# Before:
scored = ranking_client.rank(
    query=query,
    documents=[c.content for c in chunks],
    top_n=top_k,
)

# After:
scored = ranking_client.rank(
    query=query,
    documents=[_format_for_rerank(c) for c in chunks],
    top_n=top_k,
)
```

The chunk's `c.content` (full body) goes through the formatter. Vertex Ranking sees the prefix + body; ordering is on the (chunk_index, score) pair returned, which we map back to the original `chunks[idx]` — so no downstream changes.

**Subtask T2c**: The 280-char excerpt that the AGENT sees in the tool result (`search.py:471`) stays unprefixed — that's the human-readable preview. Only the rerank input gets the prefix.

**Test** (extend `test_search_hybrid.py` / new `test_rerank_prefix.py`):

### T3 — F7: Phrase-aware FTS path

**Files**: `supabase/migrations/0017_phrase_fts.sql` (new), `backend/app/tools/search.py`

**Subtask T3a (migration)**: New SQL function `match_chunks_phrase(p_query text, p_project_id uuid, p_file_ids uuid[], p_top_k int)` running `phraseto_tsquery('german', p_query)` against the existing `content_tsv` column. Returns the same shape as `match_chunks_hybrid` for drop-in compat (id, file_id, project_id, content, page_start, page_end, figure_label, block_type, filename, heading_path, chunk_index, vec_similarity=0, fts_rank, rrf_score=fts_rank). Vector cols stay 0 since this path is FTS-only.

```sql
create or replace function public.match_chunks_phrase(
  p_query text,
  p_project_id uuid,
  p_file_ids uuid[] default null,
  p_top_k int default 15
)
returns table (
  -- … same columns as match_chunks_hybrid …
) language sql stable as $$
  select c.id, c.file_id, c.project_id, c.content, c.page_start, c.page_end,
         c.figure_label, c.block_type, pf.filename, c.heading_path,
         c.chunk_index,
         0::float as vec_similarity,
         ts_rank_cd(c.content_tsv, phraseto_tsquery('german', p_query)) as fts_rank,
         ts_rank_cd(c.content_tsv, phraseto_tsquery('german', p_query)) as rrf_score
  from public.document_chunks c
  join public.project_files pf on pf.id = c.file_id
  where c.project_id = p_project_id
    and (p_file_ids is null or c.file_id = any(p_file_ids))
    and c.content_tsv @@ phraseto_tsquery('german', p_query)
  order by fts_rank desc
  limit greatest(1, coalesce(p_top_k, 15));
$$;
```

Apply via `mcp__supabase__apply_migration`.

**Subtask T3b (`tools/search.py`)**: Add a phrase-detection heuristic inside `execute_search_chunks`:

```python
def _looks_like_phrase_query(query: str) -> bool:
    """Heuristic: queries that look like a verbatim phrase search benefit
    from phraseto_tsquery routing. Triggers on:
      - Quoted strings: query starts and ends with " or '
      - 4+ tokens AND the tokens form a coherent phrase (no commas, no |)
    Conservative — false-negatives just degrade to hybrid retrieval, not a
    hard miss."""
    q = query.strip()
    if (q.startswith('"') and q.endswith('"')) or (q.startswith("'") and q.endswith("'")):
        return True
    tokens = q.split()
    if len(tokens) >= 4 and "," not in q and "|" not in q and "/" not in q:
        return True
    return False
```

When the query looks phrase-shaped:

1. Run `match_chunks_phrase` first. If it returns ≥1 row, RRF-merge those rows with the hybrid pool (same merge logic 17.x already uses for synonym expansion). The phrase rank gets a strong RRF weight — it's already filtered to actual phrase matches.
2. If `match_chunks_phrase` returns 0 rows, fall through to the existing hybrid path (current behavior).
3. Rerank still runs on the merged pool — same pipeline as today.

The model doesn't need a new tool or parameter. The router decides based on the query shape. (This is one of the few "decide-for-the-model" moves we keep — phrase shape is unambiguous and doesn't need agent reasoning.)

**Test** (extend `test_search_hybrid.py`):
- Query `"Ist in einer späteren Phase zu Detaillieren"` (quoted) → assert `_looks_like_phrase_query` returns True; assert `match_chunks_phrase` RPC is called; assert results merge with hybrid.
- Query `Bausumme` (single word) → assert phrase path is NOT taken.
- Query `Bauherren beteiligt sind welche` (4+ tokens but list-shaped) → still phrase-routed (we accept the false-positive; degrades to one extra cheap RPC).
- `match_chunks_phrase` returns 0 rows → falls through to hybrid, no error.

### T4 — F8: Page-defrag helper on `read_section`

**File**: `backend/app/tools/section.py`, `backend/app/tools/__init__.py`

Add an optional boolean parameter `include_page_neighbors` to `read_section`. When set:

- Pre-condition: the call has resolved to a non-empty chunk set via the existing path (section / page range / heading prefix).
- Post-condition: for every distinct `page_start` in the resolved set, fetch ALL chunks on that page (in `chunk_index` order) via a small RPC `chunks_on_page(p_file_id, p_page)`. Replace the resolved set with the page-expanded set, deduped by `chunk_id`, capped at the existing 30-chunk hard limit.

**Subtask T4a**: New RPC in `0017_phrase_fts.sql` (same migration since both are page-level helpers):

```sql
create or replace function public.chunks_on_page(
  p_file_id uuid,
  p_page int
)
returns table (
  -- same columns as chunks_in_range …
) language sql stable as $$
  select c.id, c.file_id, c.project_id, c.content, c.page_start, c.page_end,
         c.figure_label, c.block_type, pf.filename, c.heading_path,
         c.chunk_index
  from public.document_chunks c
  join public.project_files pf on pf.id = c.file_id
  where c.file_id = p_file_id
    and c.page_start <= p_page and c.page_end >= p_page
  order by c.chunk_index;
$$;
```

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| Parallel dispatch races on shared state (collected_chunks, ref_offset) | Pre-sliced ref windows during execution + sequential reconciliation after gather. `collected_chunks` is extended single-threaded post-gather. |
| Vertex Ranking treats the `[TYPE \| SECTION]` prefix as content noise and degrades scoring | Prefix is structured, short (≤ 80 chars), and at the start where rankers typically weight context. Worst case: rerank quality unchanged; structural chunks still benefit from the heading-path-as-context signal. |
| `phraseto_tsquery` German config doesn't tokenize colloquial-domain phrases well (Bahn-/SIA-jargon) | Phrase path is opt-in via shape heuristic AND falls through to hybrid on 0 rows. If German config is the wrong dictionary for some queries, the existing path picks them up. |
| `include_page_neighbors=true` blows the rater + agent context budget (large pages with 20+ chunks) | 30-chunk hard cap stays; `_RATER_MAX_CHUNKS=30` (17.3 T5) caps rater rendering. Worst case: rater sees a page-defragged set, decides sufficient or insufficient based on what's there. No infinite token growth. |
| Phrase routing hits the hybrid table for short prose queries that happen to be 4 tokens | 4-token threshold + `,` / `\|` / `/` filters keep the trigger conservative. False-positives cost one extra cheap RPC; results merge into hybrid pool, no quality loss. |
| Migration 0017 conflicts with parallel migration work | Coordinate with 17.4 (which lands 0016). 17.5 must land AFTER 0016 since `match_chunks_phrase` depends on `heading_path`/`chunk_index` columns being in the return shape (consistent shape across both phrase + hybrid functions). |
| Parallel async dispatch surfaces a Gemini OpenAI-compat shim bug we previously dodged via sequential dispatch | The shim's known issues are around tool_calls in messages history (handled in 17.x). Dispatching parallel CALLS is at the executor layer, after the model emitted them. No new shim path triggered. Add an integration-style test that drives 3 parallel calls through `_send_message_stream` and asserts no shim error. |
| `_format_for_rerank` produces strings that exceed Vertex Ranking's per-doc length limit | Vertex limit is generous (32k chars per doc). Our chunks max at ~2k chars; prefix adds ≤ 80. Safe. |
