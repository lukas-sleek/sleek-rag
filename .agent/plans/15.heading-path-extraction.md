# 15. Heading-Path Extraction — Fix `heading_path[]` So Section Filters Stop Lying

## Complexity: ⚠️ Medium

Single-file ingestion change plus a one-time SQL backfill. Low blast radius (chunks already exist; we're rewriting one column). The risk is in correctness: if the parser is too greedy, every chunk ends up with a wrong section path; if too strict, we miss cases Layout Parser already gave us for free. Both regressions are observable and fixable on a single re-ingest pass.

> **Renumbering note**: Plan 14 reserved slot 15 for the Projektanalyse-v1 → `search_chunks` migration. That work is now plan 16; this plan claims slot 15 because it's a higher-priority data-quality fix surfaced during plan 14's UAT.

---

## Goal

Make the `heading_path[]` column on `document_chunks` actually contain the chapter heading hierarchy, so the `search_chunks(section="1.3")` filter resolves deterministically against ingestion data instead of fighting with content-regex fallbacks.

After this plan:

1. New chunks get `heading_path[]` populated from the markdown heading lines Document AI prepends to `chunk.content` (the data we currently throw away).
2. New chunks ALSO get in-text section markers ("1.3 FRAGEN" run-on text Layout Parser failed to mark up) appended to `heading_path[]` so semi-broken parses are still queryable.
3. All existing chunks get backfilled in place via a SQL migration — no Document AI re-call, no re-embed.
4. The `match_chunks_filtered` RPC's section branch becomes a clean `heading_path` lookup; the content-regex fallback stays as a belt-and-suspenders cushion for chunks the parser couldn't recover anything from.

---

## Background — what we found in plan 14 UAT

Plan 14 added the `search_chunks` agent loop with a `section` filter. The filter routes through `chunks_by_heading_prefix` / `match_chunks_filtered`, which match against `document_chunks.heading_path[]`. Test query "was steht in abschnitt 1.3?" returned only the C1 form chunks; Teil A's section 1.3 ("FRAGEN") was missing.

Root cause traced to two issues stacked:

**Our bug — `backend/app/workers/ingest.py:262`:**

```python
heading_path = (
    [h.text for h in chunk.page_headers if h.text]
    if chunk.page_headers
    else None
) or None
```

We populate `heading_path[]` from `chunk.page_headers`. Per the proto definition ([v1beta3 document.proto, lines 1087–1093](https://github.com/googleapis/googleapis/blob/master/google/cloud/documentai/v1beta3/document.proto)), `page_headers` is the **running page banner** ("Gemeinde Hochdorf, Entwicklung Südiareal …"), not the chapter hierarchy. So `heading_path` ends up containing the same string for every chunk in the file — useless for section filtering.

The actual ancestor heading chain comes from `include_ancestor_headings=True` (set in `_layout_options` line 131), which Document AI delivers by **prepending markdown heading lines to `chunk.content`**. The proto has no separate field for it. We get the data; we just discard it during column mapping.

**Document AI bug — Layout Parser sometimes drops in-text headings:**

In Teil A, section "1.3 FRAGEN" appears as run-on paragraph text (`Hochdorf1.3 FRAGENDie Fragen sind …`). Document AI never detected it as a heading, so it's not in the prepended chain either. No amount of clean-up on our end can recover what Layout Parser didn't mark; we have to detect inline section markers ourselves with a regex during ingestion.

Confirmed against authoritative sources: [v1beta3 `Document.proto` ChunkedDocument.Chunk schema](https://github.com/googleapis/googleapis/blob/master/google/cloud/documentai/v1beta3/document.proto), [Document AI release notes (2026, Layout Parser v1.6)](https://docs.cloud.google.com/document-ai/docs/release-notes), [Vertex AI RAG Engine layout-parser integration docs](https://docs.cloud.google.com/vertex-ai/generative-ai/docs/rag-engine/layout-parser-integration). Schema is unchanged in the 2026 Gemini 3 Flash / Pro versions — `page_headers` will keep meaning "page banner."

## Architectural decisions

### What goes in `heading_path[]`

A flat array of strings, ordered shallow → deep. Each entry is one heading. Example for Teil A page-2 chunk:

```python
heading_path = [
    "VERFAHRENSBESTIMMUNGEN TEIL A",
    "1 AUFTRAGGEBER",
    "1.1 OFFIZIELLER NAME UND ADRESSE DES AUFTRAGGEBERS",
    "1.2 EINGABESTELLE",
    "1.3 FRAGEN",                # recovered via in-text regex
    "1.4 EINREICHUNG DES ANGEBOTES",
]
```

Two extraction paths run on every chunk:

**Path A — leading markdown headings (clean case):** consume consecutive lines at the top of `chunk.content` matching `^(#+)\s+(.+?)\s*$`. These are the ancestor headings Document AI prepended. Strip them from the array's stored form (no `#` prefix, no leading/trailing whitespace) but keep them in `chunk.content` unchanged so the LLM still sees the structure when it reads the excerpt.

**Path B — in-text section markers (recovery case):** scan `chunk.content` for tokens matching `(?m)(?:^|(?<=\D))(\d+(?:\.\d+)+)\s+([A-ZÄÖÜ][A-ZÄÖÜa-zäöüß /-]{2,})` (a numeric section like `1.3` or `3.4.2`, followed by an uppercase-led title of ≥3 chars). Each unique match becomes an entry like `"1.3 FRAGEN"`. Dedup against Path A (a heading that already came in clean shouldn't be re-added).

The regex is intentionally narrow:
- Requires at least one dot in the number (`1.3`, not `5`) — single-digit sections are too noisy and almost always show up as proper headings anyway.
- Requires uppercase-led title — filters out `"page 1.3 mio CHF"` and decimal numbers in body text.
- Requires preceding non-digit (`(?<=\D)` or start of line) — kills "31.3 sec" matches.
- Title cap of ~64 chars (final regex) — prevents matching huge run-on paragraphs as a single "title."

### Where the parser lives

New module `backend/app/ingest_headings.py`. Single public function `extract_heading_path(content: str) -> list[str] | None`. Pure-Python, no I/O, easy to unit-test. Imported once in `workers/ingest.py:_persist_chunks`.

### Column type

`heading_path` is already `text[]`. No schema change. The migration only rewrites values.

### Backfill strategy

**Option 1: re-ingest every file.** Heavy — costs Document AI calls + re-embedding ($) + worker time + downtime per file. Rejected.

**Option 2: SQL update from existing `content`.** Translates the same Python parser into a Postgres function and updates every row. Free, fast, deterministic. **Chosen.**

The Postgres function lives in the same migration that introduces the new column-population code so prod and dev land in sync. It uses Postgres regex (`~` operator) — equivalent to the Python parser, with a small set of test cases asserted as part of the migration.

If the Postgres regex turns out to disagree with the Python parser on edge cases later, the worst case is one re-ingest. The Python parser is the source of truth for new chunks; the SQL is the one-shot backfill.

### `match_chunks_filtered` RPC update

After the backfill lands, every section's content is reachable via `heading_path`. The current "match by content's `# 1.3` line" and "regex `(^|\D)1\.3\s*[A-ZÄÖÜ]`" branches in `match_chunks_filtered` become belt-and-suspenders — useful only for chunks whose `heading_path` couldn't be filled (because Document AI dropped the heading entirely AND the in-text regex didn't catch it either). Keep them, but `heading_path` becomes the primary path.

No new migration for the RPC — leave the existing 0011 in place. Once we trust `heading_path` we can simplify in plan 17 if we feel like it.

### What we don't do here

- We don't change `chunk.content`. The leading `#`-prefixed heading lines stay in the chunk text so the model sees structural context when it reads excerpts. Removing them would be a separate decision (might help / hurt embedding quality; not worth diffing in this plan).
- We don't touch embeddings. Existing embeddings keep their semantic value. The change is purely metadata.
- We don't migrate Projektanalyse v1 to `search_chunks` — that's plan 16.

## Tasks

### T1. New parser module — `backend/app/ingest_headings.py`

Public function:

```python
def extract_heading_path(content: str) -> list[str] | None:
    """Pull the chapter hierarchy from a Document AI Layout Parser chunk."""
```

Behavior:
- Returns `None` for empty / whitespace-only input.
- Path A: peel consecutive lines from the start matching `^(#+)\s+(.+?)\s*$`. Stop at the first non-heading line. The captured group-2 text becomes one entry per line. Whitespace stripped.
- Path B: regex-scan the remaining body for inline section markers per the pattern in "Architectural decisions" above. Append unique matches that aren't already in Path A.
- If both paths yield nothing, return `None` (preserves current "no heading" semantics — the column is nullable).

**Tests** (`backend/tests/test_ingest_headings.py`):

- Clean prepended chain → array matches the `#`-stripped lines (e.g. C1's `# 1.3 ZUSÄTZLICHE ANGABEN…` → `["1.3 ZUSÄTZLICHE ANGABEN BEI BIETERGEMEINSCHAFTEN / SUBUNTERNEHMERN"]`).
- Multi-level chain (`# A\n## 1\n### 1.1`) → 3 entries in order.
- Teil A run-on case (`### 1.2 EINGABESTELLE\n\nPostadresse…6280 Hochdorf1.3 FRAGENDie Fragen…`) → array contains `"1.2 EINGABESTELLE"` and `"1.3 FRAGEN"`.
- Body-only chunk (no leading `#`, no inline marker) → returns `None`.
- False-positive guard: `"page 1.3 mio CHF"` and `"Tel. 31.3 sec"` do NOT produce entries.
- Dedup: heading present both as `# 1.3 FRAGEN` at top and as inline `1.3 FRAGEN` later → one entry.

### T3. Backfill migration — `supabase/migrations/0012_backfill_heading_path.sql`

Postgres function `public.extract_heading_path(p_content text) returns text[]` mirroring T1's Python logic, plus a one-shot UPDATE statement that rewrites every row in `document_chunks`.

Structure:

```sql
create or replace function public.extract_heading_path(p_content text)
returns text[] language plpgsql immutable as $$
declare
  result text[] := array[]::text[];
  rest text := coalesce(p_content, '');
  m text[];
  -- ... see T1 for the regex spec
begin
  -- Path A: peel leading markdown heading lines
  loop
    -- regex_match the first line for ^#+\s+(.+?)\s*$
    -- if match: append, advance rest past the line; else break
  end loop;

  -- Path B: scan remainder for inline section markers
  for m in select regexp_matches(rest, '...', 'gm') loop
    -- dedup against result
  end loop;

  if array_length(result, 1) is null then
    return null;
  end if;
  return result;
end;
$$;

update public.document_chunks
   set heading_path = public.extract_heading_path(content);
```

The UPDATE will touch every row in the project (~thousands across our test data). Wrap in a transaction and accept the table-scan cost — this is one-time migration data, not query path.

**Test**: after applying, query a known Teil A chunk and assert `heading_path` contains both `'VERFAHRENSBESTIMMUNGEN TEIL A'` and `'1.3 FRAGEN'`. Spot-check 3 chunks total: clean C1 case, run-on Teil A case, no-heading chunk (page-1 logo description) → `null`.

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| In-text regex (Path B) over-fires on body text with decimal numbers | Title-cap pattern requires uppercase-led, ≥3 chars after the number. Plan-14 UAT data is the test set; we run T1's "false-positive" tests against it. |
| In-text regex misses real headings with lowercase titles or non-Western characters | Document the observed corpus is German/Swiss; if Italian/French/etc. titles appear later, extend the character class. Out of scope to design for languages we haven't seen. |
| Postgres regex behaves differently from Python regex on edge cases | T3 includes assertions that compare a fixed set of inputs across both. Mismatch → either fix the SQL or fall back to a Python-side backfill script (still no Document AI cost). |
| Backfill UPDATE locks the chunks table | Single-transaction UPDATE on a few thousand rows is fine. If the corpus grows past ~100k chunks we'd batch it; not a concern at current scale. |
| Frontend / chat behavior changes unexpectedly | None — `heading_path` is read only by the SQL filter, which already worked for the cases it matched. Worst case: better recall on `section=` queries; nothing breaks. |
