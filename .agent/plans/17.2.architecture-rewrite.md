# 17.2 Architecture Rewrite — Description-Driven Routing, Structured Errors, Sufficiency Check

## Complexity: 🔴 Complex

Plan 17 shipped three retrieval tools. Plan 17.1 stacked regex routers + workflow trees + per-turn system-message injection on top to coerce tool selection. Both made answer quality WORSE per the live UAT (citations dropped, agent now hands off to user, hallucinated totals).

Per Google's documented agent-design guidance (cited inline below), the right architecture is **description-driven routing**: tools describe their boundaries, the model picks via AUTO mode, errors are structured directives, sufficiency is checked by a prompted autorater. Pre-LLM regex routers, hidden coercion, and long workflow trees are all **anti-patterns** in this architecture.

This plan rips out the coercion layer, restructures the prompt, fixes the tool-result envelope, makes `projektanalyse_v2` a visible escalation path, and adds a Reasoning-Agent sufficiency check as the final-iteration gate.

**Failure modes** (from architecture-explore agent):

1. **Hidden coercion the agent can't reason about**: query expansion (`_should_expand` regex) and rerank both happen inside `search_chunks` invisibly. The agent sees the merged/reranked output and treats it as ground truth. If rerank buries the right chunk (Q3 Kieliger, Q5 Bausumme table), the agent has no way to ask for a different view.
2. **Brittle workflow tree**: WORKFLOW A asks the model to classify into `aggregation | point | smalltalk | scope-out`. The boundary is undefined ("Was ist die Bausumme?" looks point-shaped but is aggregation-shaped per the regex). Plan 17.1 added the regex injection as a "belt-and-suspenders" — the comment in code admits the static prompt wasn't firing.
3. **Outline-first rule fragmented across 3 places**: Workflow B step 5, Workflow C step 2, `read_section` description. None enforce it. Q3 mode-2 (`read_section(section="6")` with no prior outline) violated the rule three places told the model to follow.
4. **Projektanalyse v2 is invisible as an escalation**: the tool exists but neither its description nor the system prompt mentions delegating to it when the per-tool deep-dive can't answer. Q3 Projektleiter is exactly the case where v2 (full corpus) finds the answer trivially.
5. **Error envelope misroutes**: `{"results": [], "system_note": "TOOL-FEHLER (kein Nutzerfehler) … NIEMALS den Nutzer nach einer Suchanfrage fragen"}` — the model parses "Nutzerfehler" + "NIEMALS" and emits a confused user-facing handoff anyway. The directive is ignored because it's a content string, not a typed `functionResponse` payload.
6. **The "STOPPE sobald belegt" softening over-fires**: plan 17.1's last patch told the model to stop after one good search hit. The model interpreted this as "stop without citing" — Q1, Q2, Q4, Q7, Q8 all dropped citations after this patch.

## Google's documented architecture — what we're matching

All quotes from official Google sources, fetched 2026-04-28:

- **Tool descriptions are the routing layer.** [Gemini API function-calling guide](https://ai.google.dev/gemini-api/docs/function-calling): *"The model relies on these to choose the correct function and provide appropriate arguments."*
- **AUTO mode is the recommended default.** [Vertex function-calling intro](https://docs.cloud.google.com/vertex-ai/generative-ai/docs/multimodal/function-calling): *"In most cases this is the desired behavior when you want the Gemini model to use information from the prompt to determine if it should call a function, and which function it should call."*
- **Active tool set should be ≤10–20.** Gemini API guide flags that "too many tools can increase the risk of selecting an incorrect or suboptimal tool."
- **Aggregation = parallel function calls.** Gemini API guide: *"Parallel function calling lets you execute multiple functions at once and is used when the functions are not dependent on each other. You don't need to return the function_result objects in the same order that the function_call objects were received. The Gemini API maps each result back to its corresponding call using the id."*
- **Errors are natural-language directives the model can act on.** Gemini API guide: *"Implement robust error handling in your functions to gracefully handle unexpected inputs or API failures. Return informative error messages that the model can use to generate helpful responses."*
- **Sufficiency = prompted autorater (SCA).** [Vertex cross-corpus retrieval](https://docs.cloud.google.com/vertex-ai/generative-ai/docs/rag-engine/cross-corpus-retrieval): *"Reasoning Agent: Evaluates if the retrieved contexts are sufficient to answer the query. If insufficient, it generates feedback to trigger another retrieval loop based on Sufficient Context Awareness (SCA)."* [Google Research SCA post](https://research.google/blog/deeper-insights-into-retrieval-augmented-generation-the-role-of-sufficient-context/): *"The best performing method we tried is a prompted Gemini 1.5 Pro, without any fine-tuning."*
- **No regex pre-routing in front of the LLM.** Implicit but consistent across every Google routing surface (Vertex Cross-Corpus Planning Agent, Agent Builder agent cards, A2A protocol) — all are description-driven, not pattern-driven.

### T2. Slim system prompt

**File**: `backend/app/routers/chats.py`

Replace the entire `CHAT_SYSTEM_PROMPT` with a tight ~300-token block that contains ONLY:

```
ROLLE: Du bist ein technischer RAG-Assistent für Schweizer
Bahn-/Ingenieurprojekt-Ausschreibungen. Du beantwortest Fragen
ausschließlich anhand der hochgeladenen Projektdokumente.

VERHALTEN:
- Sprache: Deutsch.
- Tools: nutze die verfügbaren Retrieval-Tools, um Belege zu finden.
  Lies die Tool-Beschreibungen — sie sagen dir, wann welches Tool
  passt. Bei Aggregations-Fragen (z.B. "welche Bauherren") darfst du
  mehrere Tool-Aufrufe parallel emittieren.
- Du formulierst Suchanfragen und Filter immer selbst aus der Frage
  des Nutzers. Du fragst NIEMALS den Nutzer nach Suchbegriffen,
  Synonymen, Quellen oder einer Suchanfrage zurück. Wenn ein Tool
  einen Fehler oder leere Treffer liefert, korrigiere den Aufruf
  selbst und versuch es erneut.
- Zitate: jeder belegte Satz/Aufzählungspunkt bekommt die `ref`-Nummer
  aus dem Tool-Ergebnis in eckigen Klammern, z.B. [1] oder [3]. Refs
  sind über alle Retrieval-Tools eines Turns durchnummeriert.
- Scope-Fallback: wenn die Beschaffung nur SIA-Phasen 21/31 umfasst
  und der Nutzer nach Bauprojekt (32/41) oder Ausführung (51+) fragt,
  antworte: "Nicht Teil dieser Beschaffung — der Auftragsumfang
  umfasst nur [konkrete Phasen]." (KEIN "nicht gefunden"-Fall.)
- Smalltalk und Meta-Fragen ohne Tool-Aufruf kurz beantworten.
```

That's it. No WORKFLOW tree, no HEURISTIKEN table, no regex magnets, no aggregation list. The tool descriptions (T3) carry the rest.

`PROJEKTANALYSE_INSTRUCTIONS` stays appended as before — but its content gets simplified in T4.

### T3. Rewrite tool descriptions with explicit boundaries

**Files**: `backend/app/tools/search.py`, `backend/app/tools/outline.py`, `backend/app/tools/section.py`

Each tool description gets restructured to encode three things:
1. **Purpose** — what the tool does in one sentence.
2. **When to use me** — concrete trigger cases.
3. **When NOT to use me / use sibling X instead** — boundary cases.

`SEARCH_CHUNKS_TOOL` description:

```
Globale, gerankte Suche über alle Projektdokumente. Returns the top-K
relevantesten Chunks mit `ref`-Nummern für Zitate.

USE WHEN: any inhaltliche Frage. Always start retrieval here unless the
user named an explicit page/section.

USE PARALLEL CALLS WHEN: aggregation/list questions ("welche Bauherren",
"alle Termine"). Emit multiple search_chunks calls in one turn with
different queries (synonyms, facets, file_ids) — Gemini will dispatch
them in parallel and return all results before your next turn.

DO NOT USE WHEN: you already know the exact section name and want a
verbatim read of that section in document order — use `read_section`
instead. Or when you need a navigation overview of one file — use
`list_document_outline`.

PARAMETER: `query` is REQUIRED. Always populate it with a noun-phrase
or short descriptive search string derived from the user's question.
Never omit, never empty.

OPTIONAL: `expand_synonyms=true` triggers internal synonym fan-out
(useful for German domain vocabulary like Bauherr↔Grundeigentümer or
Bausumme↔Baukosten). Use sparingly — costs one extra Gemini call per
search.
```

`LIST_DOCUMENT_OUTLINE_TOOL` description:

```
Returns the heading tree (sections + page ranges + chunk counts) of one
file. Tiny payload (~few hundred tokens), navigation-only — no chunk
content.

USE WHEN: you need to know what sections exist in a file before reading
them; specifically before calling `read_section(section=...)` so you
can pick a real section name from the outline rather than guessing.
Also useful when search_chunks didn't surface a relevant file you'd
expect to contain the answer (per the inventory).

DO NOT USE WHEN: search_chunks already returned the chunk you need —
read its content directly from the result. Or when you want chunk
content — that's `read_section` or `search_chunks`, not this.
```

`READ_SECTION_TOOL` description:

```
Returns chunks of one section / page range / heading prefix in document
order — no retrieval ranking. Hard cap 20 chunks per call (~5k tokens).

USE WHEN: search_chunks ranked the answer below the cut-off (typical
for small fact-bearing chunks: figure captions, table headlines, single-
line list items) and you know roughly where in the document it lives.
Pair with list_document_outline first to get the exact section name.

USE WHEN (page-targeted): the user named an explicit page or page
range — pass page_from/page_to and skip section.

DO NOT USE WHEN: you don't know which section/page to read — call
list_document_outline first. Section-string guessing returns empty.

PARAMETERS: `file_id` required. Provide one of `section` (after an
outline call) OR `page_from`+`page_to` OR both. Section name must come
from the outline, not from your prior knowledge.
```

These descriptions explicitly tell the model:
- When to fan out parallel calls (Google's recommended aggregation pattern).
- The boundaries between the three retrieval tools.
- How to recover from common failure modes.

T1 deleted the regex-gated query expansion. We resurrect it as **opt-in via `expand_synonyms` parameter** in `search_chunks`. Now the agent decides; we don't.

**Test**: pytest with mocked supabase. Verify `expand_synonyms=true` triggers the fan-out, `expand_synonyms=false` (or omitted) doesn't. Old `_should_expand` regex tests deleted.

### T5. Structured tool errors

**Files**: `backend/app/tools/search.py`, `backend/app/tools/outline.py`, `backend/app/tools/section.py`

Stop returning ad-hoc strings. Adopt one envelope shape across all retrieval tools, with both machine-readable codes and natural-language guidance:

```python
# Empty-query case in search_chunks
return {
    "results": [],
    "error": {
        "code": "missing_required_argument",
        "argument": "query",
        "guidance": (
            "Du hast `search_chunks` ohne `query`-Parameter aufgerufen. "
            "Übersetze die Frage des Nutzers in einen kurzen Such-String "
            "(z.B. 'Was ist die Bausumme?' → query='Bausumme') und rufe "
            "das Tool erneut auf."
        ),
    },
}
```

```python
# Unknown file_id prefix
return {
    "results": [],
    "error": {
        "code": "unknown_file_id",
        "argument": "file_id",
        "guidance": (
            "Das angegebene file_id-Präfix existiert nicht in der "
            "Inventarliste. Wähle ein 8-stelliges Präfix aus der "
            "Dokumentenliste im System-Prompt."
        ),
    },
}
```

Same shape for `read_section` (e.g., `"code": "no_chunks_in_range"`) and `list_document_outline`.

**Critically**: replace `system_note` with `error.guidance`. The word "TOOL-FEHLER" was being parsed as user-facing — phrasing it as a Du-Form directive avoids that. The wrapping `error` dict is what makes it structured.

**Test**: pytest assertions on each error envelope. The Q5 missing-query test checks `out["error"]["code"] == "missing_required_argument"` and the guidance contains an example transformation.

### T6. Reasoning-Agent sufficiency check (SCA)

**File**: `backend/app/routers/chats.py` — new helper module if cleaner: `backend/app/sufficiency.py`

After the agent loop emits a "no more tool calls" iteration but BEFORE we stream the final answer to the user, run a sufficiency check:

```python
def assess_sufficiency(*, question: str, retrieved_chunks: list[RetrievedChunk]) -> dict:
    """Prompt Gemini to rate whether the retrieved chunks are sufficient
    to answer the user's question. Per Google Research SCA — implemented
    as a prompted autorater. Returns
    {"sufficient": bool, "missing": str|None, "feedback": str|None}.
    """
```

Prompt the autorater with: the user's question + the chunks (excerpts only, ~5k tokens max) + a strict JSON schema requiring `{sufficient: bool, missing: str?, feedback: str?}`.

Wire into the agent loop: when the model emits a final-text iteration, run sufficiency:
- If `sufficient=true`: stream the answer normally.
- If `sufficient=false` and we've used < 8 iterations: append the autorater's `feedback` as a system message ("Sufficiency-Check: deine Antwort deckt die Frage nicht vollständig ab. Fehlt: <missing>. Vorschlag: <feedback>") and continue the loop. The model gets one more chance to retrieve.
- If `sufficient=false` and we're already at the cap: stream the answer with a soft note ("Eine vollständige Antwort konnte mit den verfügbaren Werkzeugen nicht zusammengestellt werden.").

This is exactly the Reasoning Agent pattern Vertex AI RAG Engine documents — implemented in our agent loop without RAG Engine.

Cost: one extra Gemini call per turn (~500ms, ~2k tokens). Worth it for replacing the brittle prompt-only "≥3 entities" rule with a content-aware judgment.

**Test**: pytest with mocked Gemini. Three cases: sufficient → answer streams; insufficient with iterations remaining → loop continues; insufficient at cap → soft-warning streams.

## Out of scope (explicit)

- Replacing the rerank pipeline with a different model. Vertex Ranking is fine; the failure was at routing, not ranking.
- Replacing the hybrid RPC. SQL layer is sound.
- Making the agent loop async-parallel for tool dispatch. Sequential dispatch is fine for our load; Gemini's compat shim has been finicky on parallel before. Re-evaluate after T7.
- Per-corpus heuristic tables. They're regex routing in disguise. The model has to figure out where Projektorganisation lives by reading the outline.
- Dropping `match_chunks_filtered` (legacy plan-14 RPC). Defer to a cleanup plan.
- Removing the regex-detected fallback to projektanalyse v2. We don't currently have one; not adding one.

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| Without the aggregation regex, the model classifies "welche Bauherren" as a point-question and stops too early | Tool descriptions explicitly tell the model to fan out parallel calls for aggregation. Sufficiency check (T6) catches the case where it didn't. |
| Without the workflow tree, the model picks the wrong tool order | Tool descriptions encode order via "USE WHEN / DO NOT USE WHEN" boundaries. Sufficiency check is the safety net. |
| Sufficiency-check autorater hallucinates "insufficient" and forces extra loops on already-good answers | Prompt the autorater with a strict JSON schema and require concrete `missing` text. Bias toward `sufficient=true` in the rubric. Cap at one extra round. |
| Slimmer prompt drops important behavioral rules | The retained 6 rules (language, citations, no-handoff, scope-fallback, smalltalk, error-recovery) cover everything the failures touched. The 1,350→300 token cut removes regex-gated rules and brittle decision trees that weren't firing reliably anyway. |
| Removing `_should_expand` regex regresses the synonym-cluster questions plan 16 T6 fixed | Resurrected as opt-in via `expand_synonyms` parameter. Tool description tells the model when to set it (German domain nouns like Bauherr↔Grundeigentümer). The model gets to decide. |
| Structured error envelopes don't actually fix the misroute (model still asks user) | Phrasing test: the `error.guidance` field is Du-Form directives only, no "Sie haben…" / "Nutzer" / "user" wording. Plus the slim system prompt's NIEMALS-fragen rule is still in place. |
| Sufficiency check is a Gemini call per turn — adds latency + cost | ~500ms, ~2k tokens. Acceptable for the answer-quality gain. Skipped on smalltalk turns (no chunks retrieved → trivially "sufficient"). |
| projektanalyse_v2 escalation gets over-used (model defaults to v2 because it's "easier") | v2 description explicitly says "DO NOT USE WHEN search_chunks oder read_section liefert klare Belege" + "deutlich teurer". Trace audit in T7 catches over-use. |
