# 2. Tailwind Migration — globals.css → Tailwind v4 utilities

⚠️ **Medium** — Mechanical class-by-class conversion across 8 files. Single-pass viable per component, but the whole migration is best done in waves with a dev-server visual diff after each.

## Goal
Replace the ~2,019 lines of hand-rolled CSS in `app/globals.css` with Tailwind utility classes on the JSX, while matching the current rendered output **pixel-perfectly**. Pixel fidelity is the hard constraint — Tailwind purity is the soft one.

## Approach
**Tailwind v4 + `@theme inline` + residual CSS.** This is the cleanest path to pixel parity.

1. The current CSS is built around CSS variables (`--bg`, `--accent`, `--radius`, font stacks). Tailwind v4's `@theme inline` lets us register those exact values as Tailwind tokens (`--color-bg`, `--color-accent`, `--radius`, etc.), so utilities like `bg-bg`, `text-accent`, `rounded-md` resolve to the **same hex/px values** as today. No drift by construction.
2. Custom CSS that Tailwind handles poorly stays in `globals.css`:
   - radial-gradient `::after` on `.login-brand`
   - custom scrollbar (`::-webkit-scrollbar`)
   - keyframes (`fadeIn`, `dialogIn`, `pf-fade`, `pf-shimmer`, `pf-pulse-dot`, `pf-progress`, `toast-in`, `tpl-fade`, `tpl-pop`, `sendIn`, `blink`)
   - sidebar search input's inline SVG `background-image`
   - `::selection`, `::placeholder` global rules
   - `body[data-variant=...]`, `body[data-density=...]` overrides
   - Slider thumb pseudo-elements (`::-webkit-slider-thumb`, `::-moz-range-thumb`)
3. Convert components in dependency order; `git commit` per component so any regression is bisectable.

## Out of scope
- Adding shadcn/ui primitives (deferred — would replace components like `<Dialog>` wholesale, which is a separate refactor).
- Behavior changes — visual parity only.
- Variant theme switcher (`data-variant=cool|mono`) — preserved via residual CSS, no rework.

## Tasks

### T1 — Install Tailwind v4 + PostCSS
- `npm i -D tailwindcss @tailwindcss/postcss postcss`
- Create `postcss.config.mjs` with `@tailwindcss/postcss`.
- Add `@import "tailwindcss";` at the top of `app/globals.css`.
- **Validate:** `npm run dev` boots without errors; an arbitrary `<div className="text-red-500">test</div>` injection shows red. Remove the test.

### T2 — Port `:root` tokens to `@theme inline`
- Define under `@theme inline { … }` in `globals.css`:
  - Colors: `--color-bg`, `--color-bg-elevated`, `--color-bg-sidebar`, `--color-bg-input`, `--color-bg-bubble`, `--color-bg-hover`, `--color-bg-active`, `--color-border`, `--color-border-strong`, `--color-text`, `--color-text-secondary`, `--color-text-tertiary`, `--color-accent`, `--color-accent-hover`, `--color-accent-soft`, `--color-eichenberger-red`
  - Radii: `--radius` (10px), `--radius-sm` (6px), `--radius-lg` (14px) — Tailwind picks these up as `rounded`, `rounded-sm`, `rounded-lg`. Map carefully: Tailwind's default `rounded` = 0.25rem; we want our 10px to win, so register as `--radius-DEFAULT: 10px` in v4 syntax.
  - Fonts: `--font-sans`, `--font-display`, `--font-mono` mapping to the three CSS variables already wired by `next/font` (`--font-inter`, `--font-inter-tight`, `--font-jetbrains-mono`).
- Keep the existing `:root { … }` declarations intact (Tailwind reads `@theme`, residual CSS still reads `:root`). Both can coexist with identical values during migration; we'll dedupe at the end.
- **Validate:** dev server still renders identically (no class changes yet); `bg-bg` test div has same color as current `var(--bg)`.

### T3 — Convert `login.tsx`
- 11 classes: `login-shell`, `login-brand`, `login-brand-top`, `login-brand-eichenberger`, `login-brand-divider`, `login-brand-sleek`, `login-brand-center`, `login-brand-mark`, `login-brand-tagline`, `login-brand-bottom`, `login-form-wrap`, `login-form`, `field`, `btn-primary`, `login-footer`, `accent`, `sub`.
- Keep `.login-brand::after` gradient block in `globals.css` — apply it via `before:`/`after:` on the JSX would require arbitrary-value gradients that are ugly; residual CSS is fine.
- Migrate `.btn-primary`, `.field`, `.login-footer` since these are reused (template/prompt dialogs). Either: (a) convert to Tailwind classes inline at every site, or (b) keep `.btn-primary` as a residual class. **Decision:** keep `.btn-primary` as residual — it's used in 3 places and has hover/active/disabled states; one residual rule is cleaner than three Tailwind clones. Convert `.field`, `.login-footer`, `.sub` inline.
- **Validate:** dev server diff (login screen layout, brand mark gradient still visible, focus states preserved, button hover works). `npm run typecheck` clean.

### T4 — Convert `sidebar.tsx`

- Keep as residual CSS:

  - `.sidebar.collapsed .sidebar-logo, .sidebar-section, .sidebar-footer-text, .sidebar-search { display: none; }` — descendant selectors based on parent state. Either lift the state into JSX (already available via `collapsed` prop, conditionally render) **or** keep residual. **Decision:** convert by conditional render where possible (already done for `sidebar-search`, `sidebar-section`, `sidebar-footer-text`); keep `.sidebar.collapsed .sidebar-section` residual just in case.

### T9 — Final pass
- Walk `globals.css` top to bottom; delete every block that's no longer referenced (search `grep -r "className.*<class-name>"` per residual class).
- Re-run `npm run typecheck` and `npm run dev`. Open every screen: login, chat (empty + with messages + streaming), sidebar (collapsed + expanded + drag), project files modal (with files, analyzing, empty), template modal, confirm dialog, prompt dialog, toasts, drop overlays.
- Document the residual CSS surface in a comment block at the top of `globals.css`.

## Validation strategy
- After each component task: dev server up, eyeball the affected screens against memory of pre-migration state. If a screen has multiple states (hover, active, drag, animating), exercise each.
- `npm run typecheck` after every file edit (catches stray `className` typos and removed exports).
- Single commit per task; if a regression is reported later, `git bisect` lands on the offending file.

## Risk register
- **Tailwind v4 vs Next 16 PostCSS pipeline**: v4 only needs `@tailwindcss/postcss`, no `tailwind.config.ts`. Next 16 picks up `postcss.config.mjs` automatically. Low risk.
- **`@theme inline` token aliasing**: getting `--radius-DEFAULT` vs `--radius` right matters. If wrong, `rounded` resolves to Tailwind's default 0.25rem instead of our 10px. Mitigated by visual diff after T2.
- **Class collisions**: Tailwind has `.dot`, `.label`, `.name`, `.danger` as class names — these aren't Tailwind utilities, so no collision, but worth flagging.
- **Specificity**: residual `.btn-primary` selector vs inline Tailwind utilities — residual CSS in `globals.css` (no `@layer`) wins over Tailwind utilities for same specificity. Either move residuals into `@layer components` or leave as-is. **Decision:** wrap residuals in `@layer components { … }` so Tailwind utilities can override per-instance with no `!important` needed.
