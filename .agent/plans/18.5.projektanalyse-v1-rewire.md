# 18.5 — Projektanalyse v1 rewire

**Complexity:** ✅ Simple
**Prerequisites:** [18.3 Chat retrieval swap](18.3.chat-retrieval-swap.md)
**Companion:** [18.0 Master Overview](18.0.vertex-rag-migration-master.md)

---

## Goal

Confirm and tune Projektanalyse v1 against Vertex RAG retrieval. v1 is the **per-question retrieval-driven** variant: each question runs its own search, gets chunks, sends one Gemini call. After 18.3 swapped `execute_search_chunks` to a Vertex RAG wrapper, v1 inherits the new retrieval automatically — this plan is mostly verification + a few small config adjustments.

This plan is intentionally small. Most likely outcome: a couple of config/parameter trims and a real-template smoke run to confirm answer quality.

### Task 2 — Drop `pre_rerank_k` from v1 config

**What:**

`backend/app/projektanalyse.py` likely passes `projektanalyse_pre_rerank_k` and `projektanalyse_top_k` to `execute_search_chunks` for two-stage retrieval. With Vertex RAG, only `top_k` matters — RAG Engine ranks internally. Drop the `pre_rerank_k` parameter.

```python
# Before
chunks = await execute_search_chunks(
    args={"query": q, "top_k": settings.projektanalyse_top_k,
          "pre_rerank_k": settings.projektanalyse_pre_rerank_k},
    project_id=project_id, user_id=user_id, ref_offset=0,
)

# After
chunks = await execute_search_chunks(
    args={"query": q, "top_k": settings.projektanalyse_top_k},
    project_id=project_id, user_id=user_id, ref_offset=0,
)
```

`backend/app/config.py`:
- Remove `projektanalyse_pre_rerank_k`
- Keep `projektanalyse_top_k` (default 15 — slightly higher than chat's default since v1 has a single shot per question)

**Files:**
- `backend/app/projektanalyse.py` — drop the parameter
- `backend/app/config.py` — drop the field

**Validation:**
- `grep -r "pre_rerank_k\|pre_rerank" backend/` → no hits
- `pytest backend/tests/` — passes

---

### Task 3 — Smoke v1 against a real template

**What:**

Manual: log into the dev frontend with the test account, open a chat in a project with at least 3 small PDFs, send the message `Projektanalyse erstellen`. Observe:

1. SSE progress events stream: `_Projektanalyse läuft… 1/N — zuletzt: <Frage>_`
2. Each question completes within reasonable time
3. Final markdown report is persisted as the assistant message
4. Citations in the report point to the correct files / pages

If answer quality is materially worse than before the migration, the most likely culprit is either:
- LLM Parser custom prompt missing `[Seite N]` markers consistently → tighten prompt
- top_k too low → bump to 20
- Sufficiency rater is not running on v1 (it isn't by design — v1 is single-shot per question)

For the migration's vanilla phase, **don't add sufficiency to v1**. If quality regresses meaningfully, file a follow-up issue and decide whether v1 needs a sufficiency check.

**Validation:**
- Smoke run completes without errors
- Report quality acceptable on first 5 questions of the template
- Persisted assistant message visible in `chat_messages` with correct citations

## Out of scope

- Adding sufficiency / verifier hooks to v1 (deferred — these are chat-loop-only by design)
- Parallel-batch tuning beyond the existing `_run_batch` helper
- Replacing v1 with v2's single-shot pattern (different semantics — v1 stays per-question)
