# 17. Scalable Agentic Deep-Dive — Multi-Tool Retrieval for Aggregation Q&A

## Complexity: 🔴 Complex

Restructures the chat retrieval surface from one tool (`search_chunks`) to three (`search_chunks`, `list_document_outline`, `read_section`), bumps candidate-pool sizes and the agent loop cap, and rewrites the system prompt around a coverage-aware "first retrieve, then dig" protocol. Plus one new RPC (`document_outline`) and a small SQL helper for ordered-by-page chunk reads. Scope is bounded by the chunk store we already have — no schema changes beyond a read-side helper. Backwards-compatible: old tool flow disappears in favor of the new tool surface; v2 (Volltext) stays untouched as the manual escape valve.

> The cross-encoder rerank optimizes for **single-intent semantic relevance**, not **fact-bearing density**. For aggregation questions ("list all X"), the right chunk is often a small, lexically thin fact carrier (table headline, figure caption, single-line list) that loses to template / scaffolding chunks with higher question-keyword density.

This is not a parameter-tuning problem. It's an architectural mismatch between "retrieve top-K by relevance" and "list all X across files."

## Research — what Google actually recommends

External research (see References) into Vertex AI RAG Engine, Gemini function-calling, Vertex AI Ranking API guidance, and Gemini long-context docs converges on three patterns for multi-document aggregation Q&A on tightly-related corpora:

1. **Multiple specialized retrieval tools, model-orchestrated.** Don't ship one tool that "does retrieval." Ship a global semantic search, a per-document inventory tool, and a per-section exhaustive read. Let Gemini decide which to call when. (Vertex AI RAG Engine cross-corpus pattern; Gemini function-calling best practices.)
2. **Sufficiency-aware iteration.** The agent retrieves, reads its own results, decides whether coverage is complete, and loops. The agent's iteration cap should accommodate this — single-digit iteration cap kills the pattern.
3. **Right-sized rerank.** For aggregation, retrieve more candidates (50–100) and keep more (15–20), not fewer. Reranking is a noise-filter, not a compressor. (Vertex AI Ranking API guidance.)

Long-context "stuff the whole corpus" (what Volltext / v2 does) is also a Google-recommended pattern but **only for tightly-related corpora that fit in a single Gemini call**. It does not scale: token cost is linear in corpus size. We keep it as the manual escape valve, not the default. The path forward for v1 (chunked retrieval) is **targeted exhaustive reads** — bounded by file size, not corpus size.

## Goal

After this plan, the chat path closes ≥80% of the v2-vs-chat gap on the same six UAT questions while staying scalable: per-turn token cost bounded at ~15–20k regardless of corpus size, vs. v2's linear-in-corpus cost. The agent reaches the right answer through deeper investigation (more iterations, more focused tool calls), not through wider context.

Specifically:

1. New `list_document_outline(file_id)` tool: returns the heading tree + page ranges for one file. Tiny payload (~few hundred tokens). Lets the agent see what's *in* a file before deciding where to dig.
2. New `read_section(file_id, section?, page_from?, page_to?, max_chunks=20)` tool: returns chunks of one section or page range, in document order, no retrieval ranking. Capped so one call stays cheap (~5k tokens) and the agent can target precisely.
3. `search_chunks` stays — the global hybrid+rerank+expansion entry point. Slimmed schema (no longer the only tool); bumped pool sizes (`top_k` default 15, `pre_rerank_k` 80) to match Vertex AI's aggregation guidance.
4. `MAX_TOOL_ITERATIONS` 4 → 8 — the deep-dive protocol commonly takes 5–6 calls, and we want headroom over the cap.
5. Coverage-aware system prompt: for "welche/wer/list all" patterns, the agent's protocol is *retrieve → identify uncovered files → outline → dig*, with explicit instructions never to delegate query generation to the user.
6. New SQL function `document_outline(file_id, user_id)` returns deduplicated `heading_path[]` entries with min/max page span. New SQL function `chunks_in_range(file_id, page_from, page_to)` returns ordered chunks for `read_section`.

## Architectural decisions

### Why three tools, not one

Single-tool agents max out on Q3-style questions (small fact in single chunk that retrieval-rank doesn't surface). The cleanest fix is to give the agent a way to *bypass retrieval ranking* and read content in document order. That's what `read_section` is. To use it, the agent first needs to know what sections exist — that's `list_document_outline`. Three tools, each with a narrow, well-named purpose. Gemini's function-calling docs explicitly recommend this shape over generic mega-tools.

### Why we are NOT adding `read_full_corpus`

It's tempting (one tool, model decides when to use it) but **it's exactly v2's cost shape**. On a 5-file corpus it's already 50–100k tokens. On a 50-file archive it's untenable. The whole point of this plan is to make v1 dig deeper *without* paying v2's cost. v2 stays as the explicit manual route.

### Why `read_section` is capped at 20 chunks

A single `read_section(file_id, section="Projektorganisation")` returning ~10–20 chunks (~5k tokens) is one cheap call. If the agent needs more it can chain (e.g., next section). Caps prevent the model from accidentally loading a whole file via a wide page range — that defeats the scalability point. If a future use case really needs whole-file reads, add a separate `read_document` tool with explicit semantics rather than letting `read_section` grow into one.

### Bigger candidate pools

Aligns chat with Vertex AI Ranking API guidance (50–100 candidates, keep 15–20). Plan 16 set chat to 30→8 (over-aggressive compression, helps single-fact questions and hurts aggregation). Plan 17 sets chat to 80→15 (matches projektanalyse batch path; leaves the ranker as a noise filter, not a top-K compressor).

### Coverage-aware prompt

The current prompt (post-plan-16 fix) says "do one synonym retry if needed." The deep-dive prompt says: for "welche/wer" / list-all questions, **after** the initial `search_chunks`, look at which files appear in the result, identify which inventory files are *missing*, and either call `list_document_outline(missing_file_id)` then `read_section`, or directly `read_section` if you know the section name. The model's stop condition is "I have at least one chunk from every plausibly-relevant file." This is the Sufficiency-Awareness pattern from Vertex AI RAG Engine docs, manually instantiated.

### Tool-result framing

Each new tool returns the same `{results: [...], _chunks: [...]}` envelope as `search_chunks`, with a `ref` index that continues from the same `ref_offset` accumulator the agent loop already maintains. Citations stay consistent across all tools — `[3]` always means the same chunk in the final answer regardless of which tool retrieved it.

### `document_outline` RPC contract

```sql
create or replace function public.document_outline(
  p_file_id uuid,
  p_user_id uuid
) returns table (
  heading_path text[],
  page_start int,
  page_end int,
  chunk_count int
) language sql stable as $$
  with expanded as (
    select unnest(c.heading_path) as heading,
           c.page_start,
           c.page_end,
           c.id
    from public.document_chunks c
    join public.project_files pf on pf.id = c.file_id
    where c.file_id = p_file_id
      and pf.user_id = p_user_id
      and c.heading_path is not null
  )
  select array[heading]::text[],
         min(page_start) as page_start,
         max(page_end) as page_end,
         count(distinct id)::int as chunk_count
  from expanded
  group by heading
  order by min(page_start), heading;
$$;
```

Returns one row per distinct heading entry with its page span and chunk count. Agent gets a compact, scannable structure ("`1.3 FRAGEN` — pages 3–4, 2 chunks; `Projektorganisation` — pages 21–22, 3 chunks; ..."). Joining `project_files` enforces user ownership at the SQL layer.

### `chunks_in_range` RPC contract

```sql
create or replace function public.chunks_in_range(
  p_file_id uuid,
  p_user_id uuid,
  p_page_from int default null,
  p_page_to int default null,
  p_heading_prefix text default null,
  p_limit int default 20
) returns table (
  id uuid,
  file_id uuid,
  project_id uuid,
  content text,
  page_start int,
  page_end int,
  figure_label text,
  block_type text,
  filename text
) language sql stable as $$
  select c.id,
         c.file_id,
         c.project_id,
         c.content,
         c.page_start,
         c.page_end,
         c.figure_label,
         c.block_type,
         pf.filename
  from public.document_chunks c
  join public.project_files pf on pf.id = c.file_id
  where c.file_id = p_file_id
    and pf.user_id = p_user_id
    and (p_page_from is null or c.page_end >= p_page_from)
    and (p_page_to is null or c.page_start <= p_page_to)
    and (
      p_heading_prefix is null
      or (
        c.heading_path is not null
        and exists (
          select 1 from unnest(c.heading_path) h
          where h ilike p_heading_prefix || '%'
        )
      )
    )
  order by c.page_start, c.chunk_index
  limit greatest(1, least(coalesce(p_limit, 20), 30));
$$;
```

Filters by file_id (with ownership check), optional page range, optional heading prefix. Returns chunks in document order (by `page_start`, then `chunk_index`), capped at 30 hard limit. Agent decides which combination of filters to use — section name, page range, or both.

### Coverage-aware prompt — added section

```
PROTOKOLL FÜR AUFZÄHLUNGS- / WELCHE-FRAGEN:
1. Rufe `search_chunks` mit der Frage auf.
2. Prüfe in den Treffern, welche `file_id`s zurückkamen. Vergleiche mit der
   Dokumentenliste.
3. Wenn die Frage plausibel mehrere Dateien betrifft (Liste von
   Stakeholdern, Übersicht über alle Etappen, Termine über mehrere
   Phasen, ...) und ein relevanter `file_id` NICHT in den Treffern
   erscheint, rufe `list_document_outline(file_id)` für den fehlenden auf.
4. Identifiziere im Outline die wahrscheinlichste Section. Rufe
   `read_section(file_id, section=...)` ODER mit `page_from`/`page_to` auf.
5. Wiederhole 3–4 für maximal 2 weitere Dateien, bevor du antwortest.
6. Synthetisiere die Antwort über alle gesammelten Treffer (search +
   read_section). Zitiere mit `ref`-Nummern wie immer.

Wenn `search_chunks` einen kleinen Fakt (Tabellenkopf, Bildunterschrift,
Einzelzeile) NICHT findet und du anhand der Frage weisst, wo er stehen
sollte, überspringe Schritt 1 und gehe direkt zu Schritt 4 mit
read_section auf der vermuteten Stelle.
```

### Iteration cap bump

Plan 14 set `MAX_TOOL_ITERATIONS = 4`. The deep-dive protocol fits in 4 in the easy case (1 search + 2 outlines + 1 read = 4) but tilts over for harder cases. Bumping to 8 leaves headroom and matches Google's "iterate until sufficiency" pattern. Latency cost: each tool iteration is ~200–800ms; worst-case 8-iteration turn ~6–8s wall-clock, which the user already accepted ("i don't care how many turns we need").

### Citation continuity

`ref_offset` already accumulates across multiple `search_chunks` calls in plan 14's loop. The new tools use the same accumulator. So a turn that does `search_chunks → list_document_outline → read_section` ends up with refs `[1..15]` from search and `[16..28]` from read_section, contiguous. The frontend's renumbering / dedup at render-time stays unchanged.

### Tracing

Each new tool gets `@traceable(run_type="tool", name="list_document_outline" / "read_section")` so LangSmith spans show the protocol clearly per turn. The chat trace becomes a clean ladder: search → outline → outline → read_section → final answer.

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| Agent doesn't use `list_document_outline` / `read_section` even when needed (defaults to repeated `search_chunks`) | Strong protocol section in system prompt + tool descriptions explicitly cross-reference each other ("if search_chunks is incomplete, try list_document_outline"). Verify via LangSmith — if Q2/Q3/Q4/Q5 traces don't show `read_section`, sharpen prompt or add T6. |
| Agent calls `read_section` with vague section names ("Projekt") and gets wrong results | RPC's `p_heading_prefix` uses `ilike` prefix match — wrong prefix returns 0 rows, agent re-plans. Better-than-fail-loudly: agent then calls `list_document_outline` and tries again. Iteration cap absorbs the recovery. |
| Token cost balloons on a 50-file corpus | `read_section` cap at 20 chunks (~5k tokens) hard-bounds per-call cost. Agent typically reads 1–2 sections per turn. Worst-case turn: 1 search + 3 outlines + 2 read_sections = ~25k input. Linear in *number of read_sections*, not corpus size — exactly what we wanted. |
| Outline returns 100+ rows on a giant document | RPC is unbounded today. If a future doc surfaces this, add a `limit 100` server-side and a "use page_from/page_to to narrow" hint in the tool description. Not relevant for the test corpus. |
| `ilike` prefix on heading_path is ASCII-only and fails on `Übersicht` ↔ `ubersicht` | Postgres `ilike` is locale-aware on UTF-8; this works for our German content. If we hit a real case-folding edge case, switch to `unaccent()` extension. Not currently needed. |
| `read_section` fetches the same chunk that `search_chunks` already returned in the same turn | Loop's chunk dedup at render time (frontend) handles this — same chunk_id collapses to one citation. Backend allows the duplicate (different `ref` indices) which is fine for the model's reasoning. |
| `MAX_TOOL_ITERATIONS=8` lets the agent loop forever on an unanswerable question | Cap stays a hard cap; if hit, the existing "loop cap reached" warning surfaces to the user. Average turns on UAT should stay ≤6; an 8-cap-hit turn is the new failure signal. |
| Prompt rewrite bloats system message + breaks cache hit rate | The protocol section is ~250 tokens. Total prompt ~3.5k tokens. Cache-stable across the conversation (system + inventory don't change turn-to-turn). One-time payoff acceptable. |
| New tools add latency to every chat turn even on simple questions | Tools are *available*, not *called*. Simple "Hallo wer bist du?" questions still emit 0 tool calls. Latency impact is per-call, not per-turn-presence. |

---

## Out of scope

- `read_full_corpus` / "load everything" tool. Explicitly rejected — that's v2's cost shape and doesn't scale. Use Volltext / v2 manually if a question genuinely needs full-corpus synthesis.
- Replacing `match_chunks_hybrid` with a multi-vector / late-interaction retriever (ColBERT-style). Bigger architectural shift; revisit only if plan 17 still leaves gaps after T5+T6.
- HyDE (hypothetical document embeddings). Plausible follow-up if rerank still buries small fact chunks at the new `pre_rerank_k=80` setting. Defer until UAT data justifies it.
- Domain synonym dictionary at ingest (tag chunks mentioning "Grundeigentümer" with `alias=Bauherren`). Low ROI vs the agentic-loop fix; revisit if German FTS stem collisions persist on a different corpus.
- Replacing PostgreSQL FTS German stemmer with a lemmatizer (GermaLemma). Domain-specific, infrastructure-heavy, test-corpus doesn't need it.
- Server-side question-type detection ("is this an aggregation question?") to auto-trigger the protocol. Currently relying on the model to recognize the pattern via prompt instructions. If model adherence is unreliable in UAT, add detection later.
- Caching `list_document_outline` results per file. Each call is ~20ms (small RPC); not hot enough yet.
- Removing the legacy `match_chunks_filtered` RPC and `_by_vector` helper. Both unused after plan 16; drop in a separate cleanup plan once plan 17 is verified.
- Running the multi-tool agent loop async with parallel tool calls. Gemini's OpenAI-compat shim has historically been finicky on parallel tool calls; sequential is safer and the cost is ~200–800ms × tools, not enough to motivate parallelization.

## Future enhancements (parking lot)

These came up during research and design; recording them for later prioritization rather than scope-creeping plan 17:

1. **Sufficiency-check tool**: explicit `verify_coverage(claim, found_entities)` tool the agent calls before answering an aggregation question. The model self-fact-checks, then re-retrieves on gaps. Heavier than the current "agent decides via prompt" pattern; revisit if prompt-driven coverage proves unreliable.
2. **Document-level summary cache**: pre-generate a per-file "key facts" summary at ingest time (one Gemini call per file). Tool `read_document_summary(file_id)` returns it. Cheap navigation aid for the agent. Low ingest-time cost, may dramatically reduce read_section usage.
3. **Cross-file aggregation result cache**: for repeated questions like "Welche Bauherren?" across chat turns in the same project, cache the aggregated answer keyed by `(project_id, question_class)`. Invalidate on file upload.
4. **Question-class auto-routing**: detect aggregation patterns server-side (e.g., regex on user query) and silently inject the protocol's first 1–2 calls before invoking the model. Removes the "model has to remember to use the protocol" failure mode at the cost of a less general agent.
5. **Long-context tool for cross-doc synthesis**: an explicit `synthesize_full_corpus(question)` tool that the model can choose to call when targeted reads aren't enough. Internally this is just v2 (Volltext) wrapped as a tool. Bridges the chat path and v2 — but expensive, so the agent would need to know when it's worth it. (User explicitly rejected an unconditional `read_full_corpus`; this conditional version is acceptable as a parking-lot idea.)
6. **Visual reranker for figure-heavy chunks**: at present the rerank is text-only. For Q3-style "small fact in a figure caption", a multimodal reranker (passing the figure image alongside the caption text) could surface the right chunk without `read_section`. Vertex AI doesn't ship one today; revisit when they do.
7. **Per-corpus eval harness**: codify the 6 UAT questions as an automated regression suite (LangSmith eval dataset). Run on every retrieval-relevant change. Currently manual; should become automatic before this codebase has more than one operator.
