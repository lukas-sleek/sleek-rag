# 6. Responses API Chat + SSE Streaming

⚠️ **Medium** — Streaming + state hydration are the tricky parts.

## Context

After sub-plan 5, the user is authenticated and projects hydrate from the backend. But chat is still mocked: `app-shell.tsx` `sendMessage` (lines 324, 376–387) splits a hardcoded string with random delays. Threads (`Record<string, Message[]>`) are in-memory only. This sub-plan replaces the mock loop with a real SSE stream from FastAPI → OpenAI Responses API, and persists chat history server-side via the Responses API's managed conversation state.

## Goal

Sending a message in the composer streams real assistant tokens into the bubble. Refreshing the page or switching chats restores message history. Chat list and messages are scoped per-user via existing RLS.

## Prerequisites

- `OPENAI_API_KEY` in `backend/.env` with Responses API access.
- Sub-plan 5 fully landed (auth + projects).

## Design Notes

- **Persistence model:** Each `chats` row holds an `openai_thread_id` (sub-plan 5 schema already includes it). The first time we send a message in a chat, we create an OpenAI conversation (`openai.conversations.create()`) and store its id. Subsequent messages pass `conversation=<id>` to `responses.create()` so OpenAI manages history; we don't store individual messages in our DB.
- **Reading history:** `GET /api/chats/{id}/messages` calls `openai.conversations.items.list(conversation_id=...)` and returns `{role, content}` pairs that match the existing frontend `Message` shape.
- **Streaming:** FastAPI returns `StreamingResponse(media_type="text/event-stream")`. Each OpenAI delta event becomes one SSE `data: {"delta": "..."}` line. Final `data: [DONE]` closes the stream.
- **Frontend SSE:** No EventSource (we need POST + auth headers). Use `fetch(...).body.getReader()` + manual line-buffer parsing.

from app.config import settings

_client: OpenAI | None = None

def openai_client() -> OpenAI:

### 6.3 Chats router

Create `backend/app/routers/chats.py` with:

- `GET /api/chats?project_id=<uuid>` → list chats for project (RLS-scoped).
- `POST /api/chats` body `{project_id, title}` → create row, no thread yet.
- `DELETE /api/chats/{id}` → remove row (cascade handled by FK).
- `GET /api/chats/{id}/messages` → if `openai_thread_id` is null, return `[]`. Else fetch from `openai.conversations.items.list()` and map to `[{role, content}]`.
- `POST /api/chats/{id}/messages` body `{text}` → SSE stream.

Streaming endpoint (sketch):
```python
@router.post("/{chat_id}/messages")
async def send_message(chat_id: str, body: MessageIn, user_id: str = Depends(current_user_id)):
    chat = supabase().table("chats").select("*").eq("id", chat_id).eq("user_id", user_id).single().execute().data
    if not chat:
        raise HTTPException(404)

    if not chat["openai_thread_id"]:
        conv = openai_client().conversations.create()
        supabase().table("chats").update({"openai_thread_id": conv.id}).eq("id", chat_id).execute()
        chat["openai_thread_id"] = conv.id

    async def event_stream():
        with openai_client().responses.stream(
            model="gpt-4o-mini",
            input=[{"role": "user", "content": body.text}],
            conversation=chat["openai_thread_id"],
        ) as stream:
            for event in stream:
                if event.type == "response.output_text.delta":
                    yield f"data: {json.dumps({'delta': event.delta})}\n\n"
            yield "data: [DONE]\n\n"

    return StreamingResponse(event_stream(), media_type="text/event-stream")
```

Wire into `main.py`.

**Validation:** With a valid token,
```
curl -N -H "Authorization: Bearer $TOKEN" -H "Content-Type: application/json" \
  -d '{"text":"hello"}' http://localhost:8000/api/chats/<chat_id>/messages
```
streams `data: {"delta":"..."}` lines, ending with `data: [DONE]`.

### 6.4 Frontend: replace mock sendMessage

In `app-shell.tsx`:

1. Drop the 35–85ms split-word loop (lines 376–387).
2. Implement a streaming reader:

```ts
async function sendMessage(text: string) {
  const chatId = activeChatId;
  // optimistic user bubble
  setThreads(t => ({ ...t, [chatId]: [...(t[chatId] ?? []), { role: "user", content: text }] }));
  // empty assistant bubble
  setThreads(t => ({ ...t, [chatId]: [...t[chatId], { role: "assistant", content: "" }] }));

  const res = await api(`/api/chats/${chatId}/messages`, {
    method: "POST",
    headers: { "Content-Type": "application/json" },
    body: JSON.stringify({ text }),
  });

  const reader = res.body!.getReader();
  const decoder = new TextDecoder();
  let buffer = "";
  while (true) {
    const { done, value } = await reader.read();
    if (done) break;
    buffer += decoder.decode(value, { stream: true });
    const lines = buffer.split("\n");
    buffer = lines.pop()!;
    for (const line of lines) {
      if (!line.startsWith("data: ")) continue;
      const data = line.slice(6);
      if (data === "[DONE]") return;
      const { delta } = JSON.parse(data);
      setThreads(t => {
        const arr = [...t[chatId]];
        arr[arr.length - 1] = { ...arr[arr.length - 1], content: arr[arr.length - 1].content + delta };
        return { ...t, [chatId]: arr };
      });
    }
  }
}
```

Existing scroll behaviors (lines 90–231) remain unchanged — they react to message content growth.

## Validation

1. Send a message → tokens stream live; the assistant bubble grows letter-by-letter; 40%-scroll engages on the new message; stick-to-bottom kicks in mid-stream.
2. Refresh the page → chat history reloads via `GET /api/chats/{id}/messages`.
3. Stop button mid-stream → stream aborts cleanly; partial text remains.
4. Switch to a different chat and back → state hydrates correctly.
5. Two chats in same project keep separate histories (different `openai_thread_id`).
6. Backend logs show one `responses.stream` call per send; no duplicate API hits.
