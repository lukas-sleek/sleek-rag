# 18.0 — Vertex RAG Migration: Master Overview

**Complexity:** 🔴 Complex (orchestrator only — execution is split across 18.1–18.7)
**Prerequisites:** none (this is the umbrella spec)
**Status:** spec — no execution

---

## Goal

Replace sleek-rag's hand-built retrieval/embedding/ingest pipeline AND the custom chat-loop orchestration with **Vertex AI RAG Engine + a Pattern-A grounding-tool chat session** — Gemini auto-retrieves, ranks, and composes answers. Domain rules (SIA scoping, no self-sum, role policy, no-v2-auto-escalation) move from code-enforced gates into a single `system_instruction` string. The migration mirrors the user's vanilla JS test verbatim.

The migration is "vanilla first": ship the simplest viable swap, observe behavior on real data, only re-add complexity where evidence demands it.

## Net architecture

```
Upload (PDF / DOCX / XLSX)
  └─ DOCX/XLSX → LibreOffice → PDF  (existing path, unchanged)
  └─ PDF → backend uploads to GCS at gs://sleek-rag-files/{user_id}/{project_id}/{file_id}/original.pdf
       ├─ persists project_files row (status=parsing, gcs_blob_path)
       └─ rag.import_files(corpus_for(project_id), [gs://...]) → LRO

LRO poller worker
  └─ polls outstanding imports → flips project_files.status → 'ready' or 'failed'
  └─ Realtime channel pushes status updates to UI (unchanged)

Chat (Pattern A — Vertex RAG grounding tool in a Gemini chat session)
  └─ Load last 20 messages from Supabase chat_messages
  └─ chat = client.aio.chats.create(
        model="gemini-2.5-flash",
        config=GenerateContentConfig(
            system_instruction=SIA_SYSTEM_INSTRUCTION,  # all 26 rules + no-v2 + honesty clauses
            tools=[grounding_tool, view_document_page, run_projektanalyse, run_projektanalyse_v2],
            temperature=1.0, top_p=0.95, max_output_tokens=65535,
            thinking_config=ThinkingConfig(thinking_level="HIGH"),
            safety_settings=[OFF × 4],
        ),
        history=history,
     )
  └─ async for chunk in chat.send_message_stream(text):
       ├─ stream chunk.text as SSE delta
       └─ on function_call: dispatch (view_doc inline / projektanalyse hand-off)
  └─ After stream: parse grounding_metadata.grounding_chunks → Citation list (regex enrichment)
  └─ Emit SSE meta frame with citations, then done

view_document_page tool (function tool alongside grounding tool)
  └─ PyMuPDF renders a specific page or region of a project PDF
  └─ Returns image Part to Gemini for visual inspection
  └─ Used when textual chunk descriptions are insufficient (drawings, diagrams)

Projektanalyse v1 (function tool — model can call, or frontend triggers via text match)
  └─ Per-question retrieval via the SAME grounding tool (or rag.retrieval_query if needed)
  └─ Hand-off pattern preserved

Projektanalyse v2 (function tool — model-elected ONLY when user asks; or frontend trigger)
  └─ Files API upload of all project PDFs
  └─ Single streaming Gemini 2.5 Pro call: contents = [*files, big_prompt_with_all_questions]
  └─ SSE delta frames stream the answer live
  └─ No-v2-escalation rule enforced via system_instruction prompt
```

## Domain rules that MUST survive

These are invariants the migration cannot violate. Any plan that touches them must explicitly preserve them.

1. **No v2 auto-escalation.** `run_projektanalyse_v2` is invoked ONLY when the user explicitly asks for it (frontend trigger string OR model election based on user request). Code-level enforcement is gone (force-tool deleted in 18.3); enforcement now lives entirely in `system_instruction` with an explicit clause: "Du darfst run_projektanalyse_v2 NUR aufrufen, wenn der Nutzer EXPLIZIT um die v2-Variante bittet." If the model violates this in practice, tighten the prompt; if persistent, add a code-level pre-call guard as a fallback.

2. **SIA `ANSWER_INSTRUCTIONS` (26 rules)** survive — moved from custom system-message append to the `system_instruction` field of `GenerateContentConfig`. Rules cover: SIA phase scoping (21/31 vs 32/41), totals (NEVER self-sum, only headline values count), role policy (tender pre-award context, surface client-side Projektleiter), Schweizer Bahn terminology, etc.

3. **Honesty/uncertainty clause** added to `system_instruction`: model must say "nicht angegeben" when sources don't contain the answer; must not fabricate values. Replaces the deleted answer verifier.

4. **Aggregation/total clause** added to `system_instruction`: for "wie viele X?"-type questions, only count what sources explicitly enumerate; say "mindestens N, weitere möglich" when uncertain. Replaces part of the deleted sufficiency rater.

5. **`view_document_page` is escalation-only**: model is instructed in `system_instruction` to call it ONLY when textual chunk description is insufficient — drawings, diagrams, complex tables, unlabeled details.

6. **Stateless completions** (per CLAUDE.md "Module 2+ uses stateless completions"). Load last 20 messages from Supabase per turn, pass as `history=[...]` to `client.aio.chats.create(...)`.

## Success criteria for the whole migration

The migration is complete when:

1. ✅ The simplified chat-loop test suite passes: `pytest backend/tests/test_vertex_rag_grounding.py test_citations.py test_chat_pattern_a_stream.py test_chat_system_prompt.py`
2. ✅ A user can upload a PDF / DOCX / XLSX and see status transitions `parsing → ready` (no `embedding` intermediate state, since RAG Engine handles it as one unit)
3. ✅ A user can ask a question in chat, get a streamed answer with citation chips that include filename + page range (regex-derived from grounding_metadata) + figure label where applicable
4. ✅ A user can trigger `Projektanalyse erstellen` (v1) and see per-question streaming progress + a markdown report
5. ✅ A user can trigger `Projektanalyse v2 erstellen` and see a single streaming response containing all answers (no progress overlay)
6. ✅ A user can ask about a technical drawing → model calls `view_document_page` → answer references visual content
7. ✅ Project deletion cascades to corpus delete + GCS prefix delete
8. ✅ All retrieval/embedding/ingest/orchestration dead code removed; `grep -r "ranking_client\|retrieval\.py\|sufficiency\|answer_verifier\|ingest_headings\|figure_caption\|outlined_file_ids\|force_tool_next_iter" backend/app/` returns no hits
9. ✅ Supabase tables `document_chunks`, `chunk_images`, `ingest_jobs` dropped
10. ✅ End-to-end manual smoke on the test account (`test@test.com` / `12345678`) green
11. ✅ No-v2-escalation holds across 5-10 adversarial chat prompts (model never calls `run_projektanalyse_v2` without user election)
12. ✅ Re-run of the 18.0.1 benchmark after 18.3 lands shows post-migration scores ≥ Variant A's pre-migration scores

## Open risks (going in eyes-open)

- **Domain answer correctness now lives in `system_instruction`, not in code.** If aggregation/total/inversion errors happen post-migration, the fix is prompt tightening, NOT re-adding sufficiency/verifier code. Test by re-running the 18.0.1 benchmark after 18.3 lands. Worst case: a single post-hoc verification call (one extra Gemini call after the answer to score it) can be added back.
- **No-v2-escalation is now prompt-only.** Code-level guards in `chats.py:371-375` are gone. If the model ever calls `run_projektanalyse_v2` without user election, treat as a prompt-tightening issue first. Document any incident; if persistent, add a pre-call code-level guard as fallback.
- **Citation page numbers depend on LLM Parser prompt fidelity.** If the parser's custom prompt occasionally drops a `[Seite N]` marker, regex enrichment fails for that chunk → citation chip shows no page. Mitigation: monitor first 50 chats, tighten parsing prompt if needed.
- **`grounding_metadata` may be empty when Gemini answers from training instead of grounding.** Mitigation: `system_instruction` clause forcing the model to retrieve before answering; if persists, log for prompt tuning.
- **`view_document_page` cost on large drawings.** Trivial per-call (~$0.0003), but a model that abuses the tool can rack up tokens. Mitigation: tool description + system_instruction discourage "always view"; only when text description is insufficient.
- **Files API 48-hour TTL** for v2. Re-upload on each invocation is acceptable (cheap, fast). Long-lived cache deferred until needed.
- **Per-project corpus quota.** Google's published quotas don't cap corpora per project explicitly. Practical reports indicate thousands work fine. Watch this if user count grows past several thousand.
- **Pattern A streaming requires `meta` SSE frame at end** (after `grounding_metadata` is available) instead of at start as today. Frontend tweak in 18.3 makes ordering not load-bearing.

## Out of scope for the vanilla migration

- Production deployment topology (Cloud Run / GCE / on-prem)
- Workload Identity Federation (defer until prod deploy)
- Multi-region replication / data residency beyond `europe-west3`
- Memory Bank / persistent agent sessions (we're staying stateless)
- Cross-project corpus sharing / shared knowledge bases
- Custom embedding model (we use the managed `text-embedding-005`)
- Re-adding figure thumbnails / image extraction
- v2 batching for projects exceeding 1000 PDF pages — handle when it occurs
