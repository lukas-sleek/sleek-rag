# 18.0.1 — Benchmark: Vanilla Vertex RAG agent vs current sleek-rag

**Complexity:** ⚠️ Medium
**Prerequisites:** [18.0 Master Overview](18.0.vertex-rag-migration-master.md). The user has already created a Vertex RAG corpus (`projects/sleek-rag/locations/europe-west3/ragCorpora/6917529027641081856`) and ingested the Südi-Areal docs into it via Console "plug and play"
**Companion:** runs **before 18.1** as a baseline measurement (and **after 18.3** as a regression check)

---

## Goal

Measure how the **post-migration target** (Pattern A: Vertex RAG grounding tool inside a Gemini chat session, exact config from 18.3) performs against the **pre-migration current state** (custom chat loop + projektanalyse v2) on the user's 11-question Südi-Areal test set.

**Variant A in this benchmark IS what 18.3 ships.** Same model (`gemini-2.5-flash`), same generation config (`temperature=1.0`, `top_p=0.95`, `max_output_tokens=65535`, `thinking_level=HIGH`, all safety OFF), same tool surface (Vertex RAG grounding tool, no `rag_retrieval_config`), same multi-turn pattern (`client.aio.chats.create(...).send_message_stream(...)`). So the benchmark is **not** a go/no-go gate — it's:

1. **Baseline pre-migration**: tells you what the future state looks like before you ship it
2. **Regression check post-migration**: re-run after 18.3 lands; post-migration A ≈ pre-migration A confirms the implementation faithful to the spec
3. **Quantification**: gives you a concrete record (judge scores + per-question wins) instead of hand-waving

The outcome shapes nothing about whether to migrate (that's already decided), but it does:
- Surface which **question types** Pattern A weakens vs the current loop → these are the prompt clauses to tighten in `system_instruction`
- Provide a record for stakeholders / future you about what the simplification cost / gained
- Catch parser-prompt regressions (if `[Seite N]` markers go missing, Variant A's citation quality drops visibly)

---

## What we're comparing

| Variant | Model | Retrieval | Loop logic | Notes |
|---|---|---|---|---|
| **A. Future state (= 18.3 target)** | `gemini-2.5-flash` | `Tool(retrieval=Retrieval(vertex_rag_store=VertexRagStore(...)))` — Gemini auto-retrieves, citations via `grounding_metadata`. No `rag_retrieval_config` (defaults) | Multi-turn chat session via `client.aio.chats.create(...).send_message_stream(...)`. Tools list also includes view_document_page + projektanalyse function tools | **Exact** config 18.3 ships: temp=1.0, top_p=0.95, max_tokens=65535, thinking=HIGH, safety OFF |
| **B. Current — chat (pre-migration)** | `gemini-2.5-flash` | Hybrid pgvector + FTS + RRF + rerank via `match_chunks_hybrid` | Full custom loop: sufficiency, verifier, force-tool, outlined_file_ids, error-streak | Uses existing app, existing data, existing prompts |
| **C. Current — v2 (pre-migration)** | `gemini-2.5-pro` | `_load_full_corpus()` Postgres concat | Per-question parallel batch | Single-shot per question; corpus prepended |

Three variants because chat and v2 are different beasts — chat is per-question retrieval, v2 is full-context. Both deserve their own apples-to-apples comparison vs the future state.

### Task 2 — Vanilla agent client (Python port of the JS test)

**What:**

`scripts/benchmark/clients/vanilla.py` — Python equivalent of the user's JS code. Uses `google-genai` SDK with `vertexai=True`, ADC via existing service account JSON.

```python
"""Variant A: Vanilla Gemini + Vertex RAG retrieval tool.
Direct port of the user's JS test — same model, same config, same corpus."""

import os
from google import genai
from google.genai import types

PROJECT_ID = os.environ["GCP_PROJECT_ID"]
LOCATION = os.environ.get("GCP_LOCATION", "europe-west3")
RAG_CORPUS = os.environ["BENCHMARK_RAG_CORPUS"]  # full resource name
MODEL = os.environ.get("BENCHMARK_VANILLA_MODEL", "gemini-2.5-flash")  # MATCHES 18.3 SHIP TARGET


def make_client() -> genai.Client:
    return genai.Client(vertexai=True, project=PROJECT_ID, location=LOCATION)


def make_chat(client: genai.Client):
    tools = [
        types.Tool(
            retrieval=types.Retrieval(
                vertex_rag_store=types.VertexRagStore(
                    rag_resources=[types.VertexRagStoreRagResource(rag_corpus=RAG_CORPUS)]
                )
            )
        )
    ]
    config = types.GenerateContentConfig(
        max_output_tokens=65535,
        temperature=1.0,
        top_p=0.95,
        tools=tools,
        thinking_config=types.ThinkingConfig(thinking_level="HIGH"),
        safety_settings=[
            types.SafetySetting(category=c, threshold="OFF") for c in [
                "HARM_CATEGORY_HATE_SPEECH",
                "HARM_CATEGORY_DANGEROUS_CONTENT",
                "HARM_CATEGORY_SEXUALLY_EXPLICIT",
                "HARM_CATEGORY_HARASSMENT",
            ]
        ],
    )
    return client.chats.create(model=MODEL, config=config)


def ask(chat, question: str) -> dict:
    """Ask one question, return {answer, grounding_uris, latency_s, thinking}."""
    import time
    t0 = time.monotonic()
    response = chat.send_message(question)
    latency = time.monotonic() - t0

    grounding_uris = []
    try:
        meta = response.candidates[0].grounding_metadata
        for chunk in (meta.grounding_chunks or []):
            if chunk.retrieved_context:
                grounding_uris.append(chunk.retrieved_context.uri)
    except (AttributeError, IndexError):
        pass

    return {
        "answer": response.text,
        "grounding_uris": grounding_uris,
        "latency_s": round(latency, 2),
    }


def run_question_set(questions: list[dict]) -> list[dict]:
    client = make_client()
    chat = make_chat(client)
    results = []
    for q in questions:
        r = ask(chat, q["question"])
        results.append({"id": q["id"], "question": q["question"], **r})
    return results
```

**Validation:**
- `python -c "from scripts.benchmark.clients.vanilla import run_question_set; ..."` runs against a 1-question subset, returns a result with `answer`, `grounding_uris`, `latency_s`
- Multi-turn context preserved across calls in the same session (test: "kannst du mir dazu mehr details geben?" follow-up should reference prior context)

### Task 3 — Current app client (chat variant)

**What:**

`scripts/benchmark/clients/current_chat.py` — drives the current app's chat API end-to-end. Logs in with the test account, creates a chat in a chosen project, sends each question, parses SSE stream, accumulates answer + citations.

```python
"""Variant B: Current sleek-rag chat (custom loop, pgvector retrieval, sufficiency, verifier)."""

import os
import json
import time
import httpx

API_BASE = os.environ.get("BENCHMARK_API_BASE", "http://localhost:8000")
SUPABASE_URL = os.environ["SUPABASE_URL"]
SUPABASE_ANON_KEY = os.environ["SUPABASE_ANON_KEY"]
TEST_EMAIL = os.environ.get("BENCHMARK_TEST_EMAIL", "test@test.com")
TEST_PASSWORD = os.environ.get("BENCHMARK_TEST_PASSWORD", "12345678")
PROJECT_ID = os.environ["BENCHMARK_PROJECT_ID"]  # ID of the project that has the Südi-Areal files


async def login() -> str:
    async with httpx.AsyncClient() as client:
        r = await client.post(
            f"{SUPABASE_URL}/auth/v1/token?grant_type=password",
            json={"email": TEST_EMAIL, "password": TEST_PASSWORD},
            headers={"apikey": SUPABASE_ANON_KEY},
        )
        r.raise_for_status()
        return r.json()["access_token"]


async def create_chat(token: str) -> str:
    async with httpx.AsyncClient() as client:
        r = await client.post(
            f"{API_BASE}/api/chats",
            json={"project_id": PROJECT_ID, "title": f"benchmark-{int(time.time())}"},
            headers={"Authorization": f"Bearer {token}"},
        )
        r.raise_for_status()
        return r.json()["id"]


async def ask(token: str, chat_id: str, question: str, timeout_s: int = 180) -> dict:
    answer_parts = []
    citations = []
    t0 = time.monotonic()

    async with httpx.AsyncClient(timeout=timeout_s) as client:
        async with client.stream(
            "POST",
            f"{API_BASE}/api/chats/{chat_id}/messages",
            json={"text": question},
            headers={"Authorization": f"Bearer {token}"},
        ) as resp:
            async for line in resp.aiter_lines():
                if not line.startswith("data: "):
                    continue
                payload = json.loads(line[len("data: "):])
                t = payload.get("type")
                if t == "delta" and payload.get("content"):
                    answer_parts.append(payload["content"])
                elif t == "meta" and payload.get("citations"):
                    citations = payload["citations"]
                elif t == "done":
                    break

    return {
        "answer": "".join(answer_parts),
        "citations": citations,
        "latency_s": round(time.monotonic() - t0, 2),
    }


async def run_question_set(questions: list[dict]) -> list[dict]:
    token = await login()
    chat_id = await create_chat(token)
    results = []
    for q in questions:
        r = await ask(token, chat_id, q["question"])
        results.append({"id": q["id"], "question": q["question"], **r})
    return results
```

**Validation:**
- Run against local backend with `test@test.com` account, project that has the Südi-Areal docs ingested
- One-question dry run returns `answer` + `citations` (with `chunk_id`, `file_id`, `page_start`, ...)

### Task 5 — LLM-judge evaluator (optional but recommended)

**What:**

`scripts/benchmark/judge.py` — for each (question, answerA, answerB) tuple, ask Gemini 2.5 Pro to score on 4 dimensions:

- **Accuracy** (0-3): are the facts correct against the source documents?
- **Completeness** (0-3): how much of the answer-relevant content is captured?
- **Citation quality** (0-3): are citations specific (page-level), present, correct?
- **Conciseness** (0-3): is the answer appropriately scoped, no waffle?

Produce a structured Pydantic output:

```python
class JudgeVerdict(BaseModel):
    accuracy_a: int
    accuracy_b: int
    completeness_a: int
    completeness_b: int
    citation_a: int
    citation_b: int
    conciseness_a: int
    conciseness_b: int
    winner: Literal["A", "B", "tie"]
    rationale: str  # 2-3 sentences
```

Pass the source documents to the judge as Files-API parts (same trick as v2) so it can verify factuality against ground truth. Cache the upload across all 11 judgments.

The judge is optional — if running it adds friction or cost concern, skip and rely on manual diff review of the markdown report (Task 6). But automation is much more scalable for re-runs after prompt tweaks.

**Validation:**
- Judge produces a JSON output per question
- Run on 1 sample → reasonable rationale, scores in [0,3]

## How to read the result

The migration target is locked (Pattern A). The benchmark is a **measurement**, not a decision. What it tells you:

- **A scores within 10% of B/C overall** → migration ships at quality parity. Good outcome. Proceed.
- **A scores 10-25% below B/C on specific dimensions (e.g., citation specificity, completeness on aggregation questions)** → expected; these are the dimensions custom loop logic was helping with. Tighten `system_instruction` clauses for those dimensions before/during 18.3 execution. Re-run after 18.3 to confirm the post-migration A scores higher than the pre-migration A.
- **A scores 25%+ below B/C on most dimensions** → either the parsing prompt isn't producing structured chunks (no `[Seite N]` markers), or the corpus needs different chunking, or `gemini-2.5-flash` isn't strong enough → consider escalating chat model to `gemini-2.5-pro` for chat (slower, more expensive, much better on instruction-following).

After 18.3 ships, re-run the benchmark. **Post-migration A scores should match or beat pre-migration A scores** — same setup, just inside the production app. Any gap reveals an implementation bug in 18.3.

## Out of scope

- Tuning the current loop to "match" Variant A — current loop is being deleted
- Running on multiple projects (one project / one corpus / one question set is enough directional signal)
- Statistical significance testing (n=11 is small; treat results as directional, not statistical)
- Including system_instruction in Variant A for the pre-migration run — at this point the benchmark measures naked Pattern A (no domain prompt) vs current. Post-migration re-run uses the actual 18.3 system_instruction for Variant A.
