# 18.4 — `view_document_page` multimodal tool

**Complexity:** ⚠️ Medium
**Prerequisites:** [18.3 Chat retrieval swap](18.3.chat-retrieval-swap.md)
**Companion:** [18.0 Master Overview](18.0.vertex-rag-migration-master.md)

---

## Goal

Add a new tool the chat agent can call when text retrieval is insufficient — for technical drawings, plans, schematics, complex tables, unlabeled visual details. The tool renders a specific PDF page (or a region of one) as an image and returns it as a multimodal `Part` to Gemini, which then "sees" the page directly and can answer about visual content.

Pattern is **lazy / on-demand**: the corpus stays text-only; visual inspection is an escalation tool the model decides to call when the LLM Parser's textual figure description (`[Visuelle Inspektion empfohlen für: ...]`) signals it. Mirrors the `search → read_section` escalation pattern, just multimodal.

## Tasks

### Task 1 — `backend/app/tools/view_page.py` executor

**What:**

```python
"""View a PDF page (or region) as a rendered image. Multimodal escalation tool."""

from typing import Optional
import io
import fitz  # pymupdf
from google.cloud import storage
from google.genai import types
from backend.app.db import supabase
from backend.app.config import settings
from backend.app.tools.utils import resolve_file_id_prefixes

MAX_EDGE_PX = 3072  # Gemini 2.5 hard cap; larger gets resized server-side, losing detail


async def view_document_page_executor(
    *, args: dict, project_id: str, user_id: str, ref_offset: int = 0,
) -> dict:
    file_id = await resolve_file_id_prefixes(args["file_id"], project_id)
    page_number = int(args["page_number"])
    region = args.get("region")          # tuple[float, float, float, float] in 0..1 normalized, or None
    requested_dpi = int(args.get("dpi", 200))
    requested_dpi = max(72, min(requested_dpi, 400))

    # Fetch PDF from GCS
    f = await supabase.table("project_files") \
        .select("filename, gcs_blob_path") \
        .eq("id", file_id).single().execute()
    if not f.data:
        return {"error": "file_not_found"}
    gcs_uri = f.data["gcs_blob_path"]

    bucket_name, blob_path = _parse_gs_uri(gcs_uri)
    blob = storage.Client().bucket(bucket_name).blob(blob_path)
    pdf_bytes = blob.download_as_bytes()

    # Render
    doc = fitz.open(stream=pdf_bytes, filetype="pdf")
    if not (1 <= page_number <= doc.page_count):
        return {"error": f"page_out_of_range (doc has {doc.page_count} pages)"}
    page = doc.load_page(page_number - 1)

    clip = None
    if region:
        x1, y1, x2, y2 = region
        rect = page.rect
        clip = fitz.Rect(
            rect.x0 + x1 * rect.width,  rect.y0 + y1 * rect.height,
            rect.x0 + x2 * rect.width,  rect.y0 + y2 * rect.height,
        )

    # Compute effective DPI — clamp so the longest edge ≤ MAX_EDGE_PX
    target_rect = clip or page.rect
    longest_inches = max(target_rect.width, target_rect.height) / 72.0
    max_dpi_for_edge = int(MAX_EDGE_PX / longest_inches)
    effective_dpi = min(requested_dpi, max_dpi_for_edge)

    pix = page.get_pixmap(dpi=effective_dpi, clip=clip)
    png_bytes = pix.tobytes("png")
    doc.close()

    # The chat-loop dispatch (Task 2) injects this as a Part into the next contents block
    return {
        "results": [{
            "filename": f.data["filename"],
            "page_number": page_number,
            "region": region,
            "rendered_dpi": effective_dpi,
            "image_size_px": [pix.width, pix.height],
        }],
        "_image_part": types.Part.from_bytes(data=png_bytes, mime_type="image/png"),
    }


def _parse_gs_uri(uri: str) -> tuple[str, str]:
    assert uri.startswith("gs://")
    bucket, _, path = uri[5:].partition("/")
    return bucket, path
```

```python
VIEW_DOCUMENT_PAGE_DECL = {
    "name": "view_document_page",

        "properties": {
            "file_id": {
                "type": "STRING",

                "type": "ARRAY", "items": {"type": "NUMBER"},
                "minItems": 4, "maxItems": 4,
                "description": "Optional crop (x1, y1, x2, y2) in 0..1 normalized page coords",
            },
            "dpi": {
                "type": "INTEGER",
                "description": "Render DPI (72-400). Default 200. Auto-clamped to keep image ≤3072px max edge.",
            },

}

# Add to TOOLS list:
TOOLS = [..., types.Tool(function_declarations=[..., VIEW_DOCUMENT_PAGE_DECL])]
```

In the dispatch logic, when handling a `view_document_page` call:

1. Run the executor as usual
2. If `_image_part` is present in the result, after the function-response message, inject a separate user-role message containing the image Part (Gemini convention for multimodal tool returns)
3. The textual `results` block goes into the function response normally


## Out of scope

- Caching rendered PNGs across calls (defer until cost/latency profile justifies)
- Annotating the rendered image (highlighting bbox of text from the chunk) — overkill for vanilla
- Multi-page renders in one call (model can chain calls if needed)
- Vector graphics extraction (e.g., parsing CAD-like content) — pure raster render is enough
