# 9. Projektanalyse als LLM-Tool

⚠️ **Medium** — Three moving pieces (function tool registration, parallel batch handler, new SSE event type) and the tool-call interception flow has subtle pitfalls. Single-pass feasible but expect one or two iterations on the streaming protocol.

## Context

The Projektanalyse template lives in `components/template-modal.tsx` (localStorage key `eag-llm.projektanalyse-template`, default = 11 German questions about a railway engineering project). Today the only trigger is a static suggestion card (`SUGGESTIONS` in `components/chat.tsx`) whose click handler just sends the literal string `"Projektanalyse erstellen"` as a normal chat message — no batching, no per-question retrieval, no structured report.

We want the LLM to recognize natural-language intent ("erstelle mir eine Projektanalyse", "Projektanalyse erstellen", "mach mal ne Analyse", typos like "projektanalys") and trigger a deterministic batch flow that runs every template question in parallel against the project's vector store and assembles a structured markdown report.

Research on `file_search` decomposition and the Batch API (chat 2026-04-27) settled the design:
- **No button.** Tool-only path. Intent detection is the LLM's job.
- **Loop over questions, fire in parallel** via `asyncio.gather`. `file_search` *does* internally decompose multi-question prompts, but the decomposition is opaque and merges questions in the final answer — unacceptable for a structured report.
- **One Responses call per question, no shared `conversation`** so questions don't pollute each other's context.
- **SSE option 1** from chat: progress events while running, single final markdown drop at the end. No per-question token streaming.

## Goal

User types "erstell mir eine Projektanalyse" (or any reasonable variant) into the chat composer → backend's chat turn invokes `run_projektanalyse` tool → 11 (or N, from saved template) parallel `responses.create` calls fan out against the project's vector store with `file_search` → backend assembles a numbered markdown report and streams it back as a single assistant message. Refresh reloads the report from the thread.

## Prerequisites

- Module 1 plans 4–8 landed (chat backend, file_search wiring, LangSmith).
- Saved template format unchanged (one question per line, optional `1.` prefix stripped server-side).

### 9.5 Backend: intercept the tool call and run the parallel batch

The Responses API surfaces tool calls in the SSE stream as `response.output_item.added` events with `item.type == "function_call"`. The current generator only handles `response.output_text.delta`. Extend it:

```python
import asyncio

for event in stream:
    if event.type == "response.output_text.delta":
        yield f"data: {json.dumps({'delta': event.delta})}\n\n"
    elif (
        event.type == "response.output_item.done"
        and getattr(event.item, "type", None) == "function_call"
        and event.item.name == "run_projektanalyse"
    ):
        stream.close()
        async for sse in _run_projektanalyse(
            chat=chat,
            template=template,
            tool_call_id=event.item.call_id,
            response_id=event.response_id,  # for tool-output submission
        ):
            yield sse
        return
```

Because the route handler is `async def` but the stream is sync, switch the generator to `async def` and use `await asyncio.to_thread(...)` to drive each iteration of the OpenAI sync stream — or import `AsyncOpenAI` and rewrite. **Simpler: keep sync, run the batch in `asyncio.to_thread` chunks via `anyio`.** The cleanest path is to rewrite `_send_message_stream` as `async` and use `AsyncOpenAI` for the batch only (regular chat stays on the existing sync client). Decide during implementation; both work.

### 9.6 Backend: the `_run_projektanalyse` coroutine

New module `backend/app/projektanalyse.py`:

```python
import asyncio
import json
from langsmith import traceable
from app.openai_client import openai_client  # async variant

@traceable(run_type="chain", name="projektanalyse.run")
async def run_batch(*, vector_store_id: str | None, questions: list[str]) -> list[str]:
    if not questions:
        return []
    if not vector_store_id:
        return ["_(Keine Projektdateien vorhanden — Vorlage konnte nicht beantwortet werden.)_"]

    async def answer_one(idx: int, q: str) -> tuple[int, str]:
        # Independent call — no `conversation` param so questions stay isolated.
        resp = await asyncio.to_thread(
            openai_client().responses.create,
            model="gpt-4o-mini",
            input=[{"role": "user", "content": q}],
            tools=[{"type": "file_search", "vector_store_ids": [vector_store_id]}],
            include=["file_search_call.results"],
        )
        return idx, resp.output_text or "_(keine Antwort)_"

    results = await asyncio.gather(*(answer_one(i, q) for i, q in enumerate(questions)))
    results.sort(key=lambda t: t[0])
    return [r for _, r in results]


def assemble_report(questions: list[str], answers: list[str]) -> str:
    parts = ["# Projektanalyse\n"]
    for i, (q, a) in enumerate(zip(questions, answers), 1):
        parts.append(f"## {i}. {q}\n\n{a}\n")
    return "\n".join(parts)
```

The async-generator wrapper in `chats.py` calls `run_batch`, but instead of awaiting all at once, uses `asyncio.as_completed` so it can emit progress events as each question finishes:

```python
async def _run_projektanalyse(*, chat, template, tool_call_id, response_id):
    questions = template or []
    total = len(questions)
    yield f"data: {json.dumps({'progress': {'done': 0, 'total': total}})}\n\n"

    project = supabase().table("projects").select(
        "openai_vector_store_id"
    ).eq("id", chat["project_id"]).single().execute().data
    vsid = (project or {}).get("openai_vector_store_id")

    answers: list[str | None] = [None] * total
    coros = [answer_one(i, q, vsid) for i, q in enumerate(questions)]
    done = 0
    for fut in asyncio.as_completed(coros):
        idx, ans = await fut
        answers[idx] = ans
        done += 1
        yield f"data: {json.dumps({'progress': {'done': done, 'total': total, 'question': questions[idx]}})}\n\n"

    report = assemble_report(questions, [a or '' for a in answers])

    # Persist as an assistant message in the conversation thread so refresh
    # shows the report. Skip the LLM-wrap step entirely — we don't submit the
    # tool output back to the model. The orphan function_call item in the
    # thread is harmless; alternatively delete it via conversations.items.delete.
    await asyncio.to_thread(
        openai_client().conversations.items.create,
        conversation_id=chat["openai_thread_id"],
        items=[{"type": "message", "role": "assistant", "content": [
            {"type": "output_text", "text": report}
        ]}],
    )

    yield f"data: {json.dumps({'delta': report})}\n\n"
    yield "data: [DONE]\n\n"
```

(If `conversations.items.create` doesn't accept that schema in the current SDK version, fall back to feeding the report back as a `function_call_output` and letting the model echo it under the strict `instructions` from 9.4. That costs one extra short LLM call but is bulletproof.)

**Validation:**
1. Project with files. Send "erstell mir eine Projektanalyse" → SSE stream shows 11 progress events followed by one delta containing the assembled markdown.
2. Project with no files. Same prompt → single delta with the no-files placeholder.
3. Refresh the chat → assistant message with the full report is present in `GET /messages`.
4. LangSmith → parent run `chats.send_message` has child `projektanalyse.run` with 11 parallel `responses.create` children.

### 9.7 Frontend: handle progress events

In `sendMessage` (around line 720), extend the SSE parser:

```ts
const payload = JSON.parse(data) as { delta?: string; progress?: { done: number; total: number; question?: string } };
if (payload.progress) {
  const { done, total, question } = payload.progress;
  setThreads((prev) => {
    const arr = [...(prev[chatId] || [])];
    arr[arr.length - 1] = {
      role: "assistant",
      content: `_Projektanalyse läuft… ${done}/${total}${question ? ` — zuletzt: ${question.slice(0, 80)}` : ""}_`,
    };
    return { ...prev, [chatId]: arr };
  });
} else if (payload.delta) {
  // existing append logic — but for projektanalyse, the single delta IS the
  // full report, so replace rather than append. Detect by checking if the
  // current bubble is the progress placeholder.
  setThreads((prev) => {
    const arr = [...(prev[chatId] || [])];
    const last = arr[arr.length - 1];
    const isProgressPlaceholder = last?.content?.startsWith("_Projektanalyse läuft");
    arr[arr.length - 1] = {
      role: "assistant",
      content: isProgressPlaceholder ? payload.delta! : (last?.content ?? "") + payload.delta!,
    };
    return { ...prev, [chatId]: arr };
  });
}
```

**Validation:** Manually trigger Projektanalyse, watch the assistant bubble cycle through "Projektanalyse läuft… 1/11", "2/11", … then snap to the full report.

## Risks / open questions

- **Rate limits**: 11 concurrent `gpt-4o-mini` calls is well within tier-1 limits. If template grows past ~30 questions, add a `Semaphore(10)` cap.
- **Tool false positives**: model might call `run_projektanalyse` when the user only asks "was steht in der Vorlage?". Mitigate via the explicit "RUFE AUF, WENN…" examples in the description; tighten if it misfires in testing.
- **`conversations.items.create` schema**: not used elsewhere in the codebase yet. Verify in the SDK before committing to the short-circuit path; fallback (function_call_output + LLM echo) is documented inline in 9.6.
- **Aborting mid-batch**: the existing `AbortController` in `sendMessage` will close the SSE connection but won't cancel the in-flight `gather`. Add a request-scoped `asyncio.CancelledError` propagation if users complain about runaway costs after pressing stop.
