# Module 1: App Shell + Observability — Kickoff Plan

## Context

`PROGRESS.md` shows Module 1's design port is done; what remains is the actual functionality:

- [ ] Auth (Supabase)
- [ ] OpenAI Responses API integration
- [ ] LangSmith tracing

The frontend at `components/` is fully built but **entirely mocked** — `app-shell.tsx` fakes assistant streaming with a 35–85ms split-word loop (lines 376–387), `login.tsx` calls `setUser({email})` after a 650ms timeout, and `projects` hydrates from `fixtures.ts`. No backend exists. This kickoff plan defines the M1 architecture and breaks the work into single-pass sub-plans following the project's `.agent/plans/{seq}.{name}.md` convention.

**Decisions locked in (from clarifying Q&A):**

1. **M1 RAG scope = Chat + OpenAI vector stores.** ProjectFilesModal will upload to per-project OpenAI vector stores; `file_search` tool is enabled on Responses API. Code is intentionally throwaway — M2 replaces the OpenAI vector store with pgvector.
2. **Data model = mirror UI.** Supabase `projects` + `chats` tables; RLS by `user_id`.
3. **Auth = wire existing `login.tsx` to `supabase-js` directly.** Keep the design-ported form; replace the timeout with `supabase.auth.signInWithPassword` / `signUp`.

## Architecture

```
┌─────────────────────────┐         ┌──────────────────────────┐
│  Next.js 16 (port 3000) │         │  FastAPI (port 8000)     │
│  app/ + components/     │ ──CORS─▶│  backend/ (venv)         │
│  @supabase/ssr session  │ ◀──SSE──│  validates Supabase JWT  │
└─────────────────────────┘         │  calls OpenAI Responses  │
            │                       │  LangSmith @traceable    │
            │ (auth only)           └────────┬─────────────────┘
            ▼                                │
┌─────────────────────────────────────────────┐
│  Supabase: Postgres + Auth + Storage        │
│  projects, chats, project_files (RLS)       │
└─────────────────────────────────────────────┘
            │
            ▼
┌─────────────────────────┐
│  OpenAI: Responses API  │
│  + Vector Stores (M1)   │
└─────────────────────────┘
```

**Why a Python backend** (vs. Next.js API routes): CLAUDE.md mandates Python + FastAPI, and M2+ adds heavier deps (chunkers, embedders, rerankers) that don't belong in the Next.js process.

**Auth model:** Frontend holds the session in cookies via `@supabase/ssr`. Every backend call attaches the access token; FastAPI validates the JWT against `SUPABASE_JWT_SECRET` and extracts `user_id`. Backend uses the **service-role key** for DB writes but always filters by the validated `user_id` (RLS is the second line of defense, not the only one).

**Streaming:** OpenAI Responses API stream → FastAPI `StreamingResponse` re-emitting SSE → browser `fetch` body reader → token-level `setThreads` updates in `app-shell.tsx`. The existing 40%-scroll, stick-to-bottom, and post-send padding logic (lines 90–231) keeps working unchanged because it's keyed on message content growth.

## Database Schema (target — finalized in plan 4)

```sql
create table projects (
  id uuid primary key default gen_random_uuid(),
  user_id uuid not null references auth.users(id) on delete cascade,
  name text not null,
  openai_vector_store_id text,           -- null until first file uploaded
  created_at timestamptz default now()
);

create table chats (
  id uuid primary key default gen_random_uuid(),
  project_id uuid not null references projects(id) on delete cascade,
  user_id uuid not null references auth.users(id) on delete cascade,
  title text not null,
  openai_thread_id text,                 -- Responses API conversation id
  created_at timestamptz default now()
);

create table project_files (
  id uuid primary key default gen_random_uuid(),
  project_id uuid not null references projects(id) on delete cascade,
  user_id uuid not null references auth.users(id) on delete cascade,
  filename text not null,
  size_bytes bigint,
  openai_file_id text,
  status text default 'pending',         -- pending|indexed|failed
  created_at timestamptz default now()
);

-- RLS: every table gets `using (auth.uid() = user_id)` for select/insert/update/delete
```

Frontend `Project` / `Chat` shapes from `components/fixtures.ts` map cleanly — only difference is real ids and the addition of `openai_vector_store_id` / `openai_thread_id` (kept server-side, never sent to client).

## Verification

End-to-end smoke (run after plan 7 lands):

1. `cd backend && source venv/bin/activate && uvicorn app.main:app --reload --port 8000`
2. `npm run dev` (port 3000)
3. Open `http://localhost:3000` → sign up via login form → land in app shell.
4. Confirm `projects` table in Supabase has a default project for the new user (seeded by signup hook or by app-shell on first load — decided in plan 4).
5. Send a chat message → see SSE tokens stream into the assistant bubble in real time, 40%-scroll behavior preserved.
6. Upload a PDF via ProjectFilesModal → file row appears with `status=indexed` after vector store finishes processing.
7. Send a chat message that references the uploaded doc → response cites file content (Responses API + file_search).
8. Open LangSmith UI → see traces for the chat completion and file ingestion runs.
9. `npm run typecheck` clean. `cd backend && python -m pytest` clean (per-sub-plan tests).
10. RLS check: sign up a second user, confirm they cannot see the first user's projects/chats/files via direct Supabase REST query with their JWT.
