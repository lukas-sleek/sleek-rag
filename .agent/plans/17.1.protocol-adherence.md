# 17.1 Protocol Adherence — Force Outline-First, Domain Heuristics, Aggregation Detection

## Complexity: ⚠️ Medium

Plan 17 shipped three retrieval tools (`search_chunks`, `list_document_outline`,
`read_section`), generalized the agent loop, and added a coverage-aware
prompt protocol. Live UAT against the 11-question test set scored
**2 PASS / 3 PARTIAL / 6 FAIL** — *worse* than plan 16 (1 PASS / 1 PARTIAL /
4 FAIL of 6).

LangSmith trace inspection shows two distinct failure modes that are NOT
"the protocol doesn't fire":

1. **Protocol skipped on Q2/Q5/Q7/Q11** (Bauherren / Bausumme / Rahmen-
   bedingungen / "spätere Phase") — single `search_chunks` only, then
   final answer. The agent didn't recognize these as aggregation questions
   despite the prompt's protocol section.
2. **Protocol fires with wrong section guess on Q3/Q4/Q6/Q10** — agent
   skips `list_document_outline` and goes straight to `read_section(section="6"
   |"2.4"|"8.3")`. Sections all wrong, recovery loop only succeeds on Q8.

Plan 17 anticipated mode 2 ("if T5 reveals the agent isn't finding the
right section name … strengthen the system prompt with domain heuristics"
— T6, deferred). Mode 1 is new and bigger.

This plan ships the deferred T6 plus a structural prompt rewrite plus a
small server-side reinforcement to address mode 1.

## Goal

After this plan: ≥ 80% gap closure on the same 11-question UAT, with

- **Every aggregation/welche-question fires `list_document_outline` at least

## Tasks

### T1. Prompt restructure — workflow tree at top, heuristics table, hard gates

**File**: `backend/app/routers/chats.py` — `CHAT_SYSTEM_PROMPT`.

Reshape the prompt from a flat 10-rule list into a 3-section structure:

```
1. WORKFLOW (decision tree — what to do FIRST)
2. REGELN (mechanical rules — citations, German, smalltalk)
3. HEURISTIKEN (domain priors for this corpus type)
```

The workflow tree replaces today's "rules 1, 5, 6, 7" with one
unambiguous decision flow that the model must follow before answering:

```
WORKFLOW (verbindlich, in dieser Reihenfolge):

A) Frage-Klassifikation. Vor jedem Tool-Aufruf entscheidest du:
   - "Smalltalk/Meta" (Hallo, wer bist du, …) → kein Tool, kurz antworten.
   - "Scope-out-of-bounds" (Bauprojekt SIA 32+/Ausführung SIA 51+ während
     Beschaffung nur 21/31 umfasst) → Scope-Fallback (siehe REGELN).
   - "Aggregations-/Sammelfrage" (welche, wer alles, alle Termine, gesamte
     Liste, „liste auf", "übersicht", "wo überall steht …") → Workflow B.
   - "Punkt-Frage" (eine konkrete Zahl, eine konkrete Stelle, eine konkrete
     Definition) → Workflow C.

B) AGGREGATIONS-WORKFLOW. PFLICHT für ALLE „welche/wer/wo überall/
   liste"-Fragen, auch wenn du denkst die Antwort liegt nahe:
   1. Rufe `search_chunks` mit der Frage als Query.
   2. Sammle die `file_id`s der Treffer.
   3. PRÜFE die Inventarliste unten: für JEDE Datei, die plausibel zur
      Frage beiträgt und NICHT in den search_chunks-Treffern erscheint,
      rufe `list_document_outline(file_id)` auf.
   4. Aus jedem Outline (eigene Treffer + neu geöffnete) wähle die
      wahrscheinlichste(n) Section(s) — siehe HEURISTIKEN.
   5. Rufe `read_section(file_id, section=...)` auf.
   6. Antworte erst, wenn du mindestens 3 distinkte Entitäten genannt
      hast ODER explizit verifiziert hast, dass es weniger gibt.
      (Beispiel: „Welche Bauherren?" — wenn dein Treffer-Set nur einen
      einzigen Bauherrn nennt, MUSST du outline+read_section auf den
      anderen Inventardateien ausführen, bevor du antwortest.)

C) PUNKT-WORKFLOW.
   1. Rufe `search_chunks` mit der Frage. Wenn der Treffer den Fakt
      eindeutig belegt → antworte mit Zitat.
   2. Wenn die Treffer den Fakt NICHT eindeutig belegen, rufe
      `list_document_outline(file_id)` für die wahrscheinlichste Datei und
      anschliessend `read_section(...)` auf.
   3. NIEMALS `read_section` mit einem geratenen `section`-Wert ohne
      vorigen `list_document_outline`-Aufruf auf derselben Datei in
      diesem Turn. (Ausnahme: der Nutzer hat eine konkrete Seitenzahl
      genannt — dann mit `page_from`/`page_to`.)
```

Add the heuristics block:

```
HEURISTIKEN (Wo typische Fakten in Schweizer Bahn-/Ingenieurprojekt-
Ausschreibungen liegen — verwende diese als Hinweis, NIE als Ersatz für
list_document_outline):

| Frage betrifft | Wahrscheinliche Section(s) im Outline |
|---|---|
| Auftraggeber, Bauherr, Grundeigentümer, Stakeholder | Projektorganisation, Bauherrschaft, Beteiligte, Auftraggeber |
| Projektleiter, Schlüsselpersonen, Personenrollen | Projektorganisation, Organigramm, Abbildung mit Rollen |
| Bausumme, Baukosten, Investitionskosten, Total | Grobkostenschätzung, Investitionskosten, Kostentabelle, Etappenkosten |
| Drittprojekte, Schnittstellen, tangierende Projekte | Schnittstellenprojekte, Tangierende Projekte, Drittprojekte |
| Termine, Meilensteine, Zeitplan | Terminplan, Meilensteine, Ausschreibungstermine, Gantt, Realisierungsetappierung |
| Vermessung, Geologie, Soziologie, Spezialfachmandate | Mandate, Spezialisten, Separate Fachmandate, Schnittstellen Fachplaner |
| „Spätere Phase", „abschliessend zu definieren" | Etappierung, Realisierung, Phase 31, Bildunterschriften |
| Honorar, Aufwand, Stunden | Honorar, Honorarermittlung, Stundenaufwand, Phasenstunden |

Wenn der Outline keine offensichtliche Übereinstimmung zeigt: nimm die
Section, deren Heading die meisten der Frage-Stichworte ODER ein Synonym
davon enthält. Wenn nichts passt, mach einen breiten `read_section` mit
`page_from`/`page_to` über die plausible Region (z.B. erste 5 Seiten für
Auftraggeber).
```

Move the citation rule, German rule, smalltalk rule into a much shorter
REGELN section so they're not lost in the noise.

**Test**: smoke-import; assert the final prompt contains both the WORKFLOW
B step ("3 distinkte Entitäten") and the HEURISTIKEN table headers.

### T2. Server-side aggregation hint (mode-1 reinforcement)

**File**: `backend/app/routers/chats.py` — `_build_system_message` becomes
`_build_system_messages` and returns a list, plus a new module-level
helper `detect_aggregation(text)`.

Detect on the fly whether the user's current message is an aggregation
question. Pattern (regex, German + Swiss-German variants):

```python
_AGG_RE = re.compile(
    r"\b("
    r"welche[mnrs]?|welches|wer|wem|wen|wo|wo überall|"
    r"alle (?:bauherren|termine|drittprojekte|stakeholder|"
    r"projekte|meilensteine|phasen|etappen|mandate|partner)|"
    r"liste auf|übersicht|aufzählung|"
    r"gibt es (?:zwingende|alle)|"
    r"steht (?:irgendwo|in den plänen|in einem)"
    r")\b",
    re.IGNORECASE,
)
```

When matched, prepend a per-turn injection RIGHT BEFORE the user message
(separate from the static system prompt so it survives prompt-cache reuse
across turns):

```python
{"role": "system", "content":
   "AUFGABE: Diese Frage ist eine AGGREGATIONS-FRAGE. Folge zwingend "
   "Workflow B aus dem System-Prompt: search_chunks → list_document_outline "
   "auf allen plausibel relevanten Dateien aus der Inventarliste, die nicht "
   "in den Treffern erschienen → read_section. Antworte erst, wenn du "
   "mindestens 3 distinkte Entitäten genannt hast oder explizit verifiziert "
   "hast, dass es weniger gibt."}
```

This is belt-and-suspenders: even if the model glosses over the
WORKFLOW section in the static prompt, the per-turn injection sits
right next to the user's question and is much harder to ignore.

**Test**: pytest assertion that `_AGG_RE` matches Q2/Q5/Q7/Q11 from the
UAT and does NOT match Q1 ("In welcher Phase…", which is a single-fact
question) or Q9 ("Welche Elemente sind im Ausführungsprojekt…", which is
scope-fallback). Frontmatter test: `_send_message_stream` includes the
extra system message in `messages` only when the trigger fires.

### T3. Tool description tightening — `read_section` requires outline-first

**File**: `backend/app/tools/section.py`

Update the `READ_SECTION_TOOL` description to make outline-first explicit:

```
Liefert die Chunks eines Abschnitts oder Seitenbereichs einer Datei in
Dokumentreihenfolge — ohne Retrieval-Ranking.

WICHTIG: Rufe ZUERST `list_document_outline(file_id)` auf derselben Datei
auf, wenn du `section=...` setzt. Section-Werte zu raten ohne den Outline
zu kennen führt fast immer zu leeren Treffern. Ausnahme: der Nutzer hat
eine konkrete Seite genannt — dann reicht `page_from`/`page_to` direkt.

Begrenzt auf max. 20 Chunks pro Aufruf.
```

This is just a description change — no executor logic changes. The model
is the gatekeeper here; we're telling it explicitly what plan 17's
description only implied.

(We deliberately do NOT enforce outline-first server-side. If the model
already has a good prior — e.g., the user said "auf Seite 5" or
mentioned a section name verbatim — forcing an outline call wastes a tool
turn. The prompt instruction is sufficient.)

**Test**: smoke-import; assert the new sentence is in the tool
description.

## Out of scope (explicit)

- Server-side hard enforcement of outline-before-read_section. Adds
  latency on cases where the agent already has a strong prior (user
  named a page). Reconsider only if T1+T3 prompt-only enforcement still
  shows guess-then-fail traces in T4.
- Per-project domain heuristics (different vocabulary for, e.g.,
  building-permit vs. railway-tender corpora). The HEURISTIKEN in T1 are
  tuned for this corpus type. If we onboard a different corpus type,
  templatize then.
- Question-class auto-routing (server-side dispatch into v1/v2 batch
  paths based on detected aggregation pattern). Bigger architectural
  change; revisit only after T4 proves the prompt-only path saturates.
- HyDE / multi-vector retriever / BGE-M3. All deferred from plan 17;
  re-evaluate if T4 still misses on Q5 (Bausumme hallucination — that
  failure is orthogonal to the protocol, it's a chunk-content issue).

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| `_AGG_RE` regex over-fires (matches non-aggregation questions like "Wer ist der Auftraggeber?") and forces outline calls on simple lookups | Acceptable: outline calls are cheap (~20ms RPC, ~few hundred tokens). Worst case: one wasted outline. The trade is heavily in favor of false-positive trigger over false-negative. |
| `_AGG_RE` under-fires on Q5-style ("Was ist die Bausumme?") which is technically a single-fact question but needs the protocol because of the multi-table corpus | Add "bausumme|baukosten|investitionskosten|gesamtkosten" as explicit triggers in the regex (these are domain-specific aggregation magnets even when phrased as point-questions). Same trick is already in `_EXPAND_NOUN_TRIGGER` in plan 16 T6. |
| Prompt grows too long, increases per-turn token cost | Static prompt was ~3.5k tokens; new structure trims redundancy in REGELN, adds ~400 tokens for HEURISTIKEN/WORKFLOW. Net ≤ 4k. Per-turn injection is ~80 tokens. Cache-stable across turns within a session. |
| Model still skips Workflow B even with the per-turn injection | T4 hard gate catches this; remediation is server-side enforcement (the deferred-out-of-scope item). |
| Q5 still hallucinates Bausumme numbers because the right table chunk is bottom-ranked | Q5 hallucination is a *chunk-content* problem (multiple cost tables in the corpus, agent picks the wrong one). T1+T2 force outline+read_section, which exposes the right Tabelle 2 on Teil B p.17 in document order — surfacing it bypasses the rerank pathology. If T4 still fails Q5, escalate to figure/table-aware retrieval (out of scope). |
