# 18.6 — Projektanalyse v2 rewrite

**Complexity:** ⚠️ Medium
**Prerequisites:** [18.2 Ingest pipeline](18.2.ingest-pipeline-replacement.md) (so PDFs live in GCS); independent of 18.3/18.4/18.5
**Companion:** [18.0 Master Overview](18.0.vertex-rag-migration-master.md)

---

## Goal

Replace v2's per-question parallel batch (~400 LOC of orchestration) with a **single multimodal Gemini 2.5 Pro streaming call**:

- Read all project PDFs from GCS
- Upload via Files API
- Build one prompt with `ANSWER_INSTRUCTIONS` + numbered question list
- Stream the response — Gemini answers all questions in one block

Net result: ~30 LOC for v2, no batching, no progress accumulator, no `_load_full_corpus`, no per-question Gemini call. Quality side-effect should be neutral or positive (model maintains coherence across questions in a single pass).

The no-v2-auto-escalation rule still applies: v2 is invoked only via the user typing `Projektanalyse v2 erstellen` or model-elected on a non-forced turn.

## Tasks

### Task 1 — Rewrite `stream_projektanalyse_v2`

**What:**

Replace the body of `backend/app/projektanalyse.py::stream_projektanalyse_v2` with the single-call pattern.

```python
import os
from google import genai
from google.genai import types
from google.cloud import storage
from backend.app.db import supabase
from backend.app.config import settings

_client = genai.Client(vertexai=True, project=settings.gcp_project_id, location=settings.gcp_location)


async def stream_projektanalyse_v2(*, template, chat_id, user_id):
    """Single multimodal Gemini call: all project PDFs + all questions in one prompt."""
    chat = await supabase.table("chats").select("project_id").eq("id", chat_id).single().execute()
    project_id = chat.data["project_id"]

    files_rows = await supabase.table("project_files") \
        .select("id, filename, gcs_blob_path") \
        .eq("project_id", project_id) \
        .eq("status", "ready") \
        .execute()

    if not files_rows.data:
        yield _sse_done(message="Keine Dateien in diesem Projekt.")
        return

    # Download PDFs from GCS to /tmp, upload to Files API
    storage_client = storage.Client()
    file_parts = []
    for row in files_rows.data:
        bucket_name, blob_path = _parse_gs_uri(row["gcs_blob_path"])
        local_path = f"/tmp/v2_{row['id']}.pdf"
        storage_client.bucket(bucket_name).blob(blob_path).download_to_filename(local_path)
        f = _client.files.upload(file=local_path)
        file_parts.append(f)
        os.remove(local_path)

    # Build single-shot prompt
    questions = template or DEFAULT_TEMPLATE
    question_block = "\n\n".join(f"## Frage {i+1}\n{q}" for i, q in enumerate(questions))
    prompt = f"""\
{ANSWER_INSTRUCTIONS}

Du erhältst die vollständigen Projektunterlagen als Anhänge. Beantworte ALLE folgenden Fragen
nacheinander im Markdown-Format. Halte für jede Frage exakt diesen Aufbau ein:

## Frage N
**Frage:** <wörtliche Frage>

**Antwort:** <deine ausführliche Antwort mit Zitaten in der Form (Dateiname, S. X)>

---

{question_block}
"""

    # One streaming call
    full_text = ""
    async for chunk in _client.aio.models.generate_content_stream(
        model=settings.vertex_rag_generation_model,
        contents=[*file_parts, prompt],
    ):
        if chunk.text:
            full_text += chunk.text
            yield _sse_delta(chunk.text)

    # Persist final assistant message
    await _persist_assistant_message(chat_id=chat_id, user_id=user_id, content=full_text)
    yield _sse_done()
```

**Files:**
- `backend/app/projektanalyse.py` — replace `stream_projektanalyse_v2` body
- `backend/app/projektanalyse.py` — DELETE `_answer_v2_sync`, `_load_full_corpus` (they are v2-only and no longer used)

**Validation:**
- `pytest backend/tests/test_projektanalyse_v2.py` — new test, mock Files API + generate_content_stream, assert single call with all files attached
- Manual: trigger v2 in the UI on a 3-PDF / 5-question test project → see streaming markdown response with all 5 answers, no progress overlay, total elapsed < ~60s

### Task 2 — Drop v2 progress events from frontend

**What:**

The current frontend shows `_Projektanalyse läuft… 3/8 — zuletzt: Kosten?_` based on `progress` SSE frames. v2 no longer emits those (it streams text directly). Remove the v2-specific progress overlay.

`components/chat.tsx` / `components/app-shell.tsx`:
- Where `progress.done / progress.total` is rendered, narrow the rendering to messages tagged as v1-projektanalyse only
- v2 messages should render via the normal text-streaming path (delta frames render character-by-character as in regular chat)

The frontend trigger for v2 (`text === "Projektanalyse v2 erstellen"`) stays — backend still routes the same way.

**Files:**
- `components/chat.tsx` — narrow progress overlay condition
- `components/app-shell.tsx:935-942` — review SSE meta/delta/done handling, no change expected (delta path already works)

**Validation:**
- Manual: trigger v2 → no progress percentage shown; markdown headings `## Frage 1`, `## Frage 2` appear as Gemini writes them
- Trigger v1 → progress overlay still shows (`1/8 — zuletzt: ...`)

### Task 3 — Files API cleanup is a non-issue (no action)

**What:**

Files API has a 48-hour TTL. We re-upload on every v2 invocation (cheap, fast). No cache, no cleanup logic needed. If a v2 call happens 49 hours after the previous one, the previous file refs would have expired — but we never reuse them, so nothing breaks.

If at scale this becomes wasteful, add caching later. For vanilla, do nothing.

**Validation:** N/A.

## Acceptance criteria

- [ ] v2 makes a single streaming Gemini call with all project PDFs as Files-API parts
- [ ] No `_run_batch`, `_answer_v2_sync`, `_load_full_corpus` invocations from the v2 path
- [ ] Frontend renders v2 response via the normal delta-streaming path (no progress overlay)
- [ ] v1 progress overlay still works (v1 unchanged from 18.5)
- [ ] Triggering v2 on a 3-PDF / 5-question project completes in well under 2 minutes
- [ ] Total cost per v2 run logged and within expected band
- [ ] No regression in v1 / chat tests

## Out of scope

- Caching uploaded Files API references across runs
- Splitting v2 into batches for projects exceeding 1000 PDF pages (Gemini 2.5 Pro limit) — handle when first project hits the limit
- Re-using v2 for chat (it's a separate flow, model/user-elected)
- Migrating from Files API to context caching API for further cost reduction
