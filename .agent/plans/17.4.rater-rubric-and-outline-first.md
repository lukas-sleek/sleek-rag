# 17.4 Rater Rubric Tightening + Outline-First Force-Retry

## Complexity: ⚠️ Medium

Plan 17.3 wired the binding iteration contract — `tool_choice="required"` on sufficiency-fail, retrieval-only tool list under force-mode, structured-error retry envelopes, no v2 auto-escalation. Live UAT against the 11 reference questions (LangSmith chat `a6500475`, 2026-04-28) shows the contract IS firing — sufficiency=insufficient triggers retries on Q3/Q6/Q10/Q11 — but the retries are shallow:

- Forced retry routes the model back to `search_chunks` with a near-duplicate query (Q3: `Projektleiter` → `Name Projektleiter`; Q11: `Ist in einer späteren Phase zu Detaillieren` → `detaillieren`). The structural tools (`list_document_outline`, `read_section`) almost never get called.
- When `read_section` IS called, it's invoked WITHOUT a prior `list_document_outline` (Q10: `read_section(section="8.3 Machbarkeitsstudie Phase 21")` — fabricated section name, wrong file).
- The rater greenlights 3-4 of the failures as `sufficient=true` despite obvious gaps (Q5: per-Etappe rows but no `39'114'000` Total; Q8: "no info found" rated sufficient on a question where the corpus does contain the 600/1350/3500 hour breakdown).

Per-question failure mode breakdown:

| # | Question | Mode | Pattern |
|---|---|---|---|
| 2 | Bauherren | rater-FP | sufficient=true on Hochdorf-only |
| 3 | Projektleiter | retry-shallow | reformulation, no outline(Teil B) |
| 4 | Termine | rater-FP | sufficient=true on Vorprojekt-only |
| 5 | Bausumme | rater-FP | sufficient=true on per-row, no Total |
| 6 | Drittprojekte | retry-shallow | 3 search calls, no outline, no fan-out |
| 7 | Rahmenbed. | rater-FP | sufficient=true on Vorprojekt-only |
| 8 | Bauprojekt-Stunden | rater-FP | sufficient=true on empty-content |
| 10 | Vermessung | guess-section | read_section without outline |
| 11 | "spätere Phase" | retry-shallow | reformulation, no outline |

Two architectural moves close all of these:

1. **Rater gets sharper rubric + chunk metadata.** Question-type detection inside the rater prompt (no regex outside the rater — the model classifies). For aggregation/total questions, the rubric explicitly requires ≥3 distinct entities OR explicit "only X exists" verification AND a Total when the question asks for one. Rater also sees `file_id`, `heading_path`, `block_type` so it can spot "all chunks are sub-rows of the same table; no headline row in the set."
2. **Forced retry routes to outline, not search.** When sufficiency fails AND we already did ≥1 `search_chunks` AND no `list_document_outline` was called this turn, force the next call to `list_document_outline` on the most-cited file_id. Plus `read_section(section=...)` rejects calls that don't have a prior outline call on the same file_id in the same turn — structural enforcement, not advisory prompt rules.

No v2 escalation. No always-on synonym expansion. No new tools. No SQL changes.

---

## Goal

After this plan, against the 11-question UAT:

- **≥9/11 with citations** (vs 17.3's 2/11 PASS).
- **Zero false-positive sufficiency verdicts** on aggregation/total questions where the corpus contains the answer (Q2/Q4/Q5/Q7/Q8 specifically).
- **Forced retries call `list_document_outline` ≥1×** on every failing-question trace where the model started with `search_chunks`.
- **Zero `read_section(section=...)` calls without a same-turn prior `list_document_outline`** — enforced via structured error envelope.
- The Q3 (Projektleiter), Q5 (Bausumme), Q6 (Drittprojekte), Q10 (Vermessung) traces should each show the workflow `search_chunks → sufficiency_fail → list_document_outline → read_section → final_answer`.

## Tasks

### T1. Stricter rater rubric — question-type aware

**File**: `backend/app/sufficiency.py`

Replace the current `_INSTRUCTION` constant with a multi-section prompt that:

1. Asks the rater to classify the question itself before scoring sufficiency:
   - `point` — single fact (Wer/Wie heisst/Was ist X?)
   - `aggregation` — list of entities (Welche/Alle Y / wer ist beteiligt)
   - `total` — sum/headline (Bausumme, Gesamtkosten, Gesamtaufwand, wie viele Stunden insgesamt)
   - `phrase` — verbatim phrase search (Steht der Satz X in den Plänen?)
   - `out_of_scope` — clearly outside the procurement scope (SIA-32+ Bauprojekt etc.)
2. Applies a type-specific rubric:

```
RUBRIC PER FRAGENTYP:

point: sufficient=true wenn der gesuchte Fakt in mindestens einem
  Chunk explizit benannt ist. NICHT sufficient: 'Name nicht
  enthalten', 'Formular ohne Daten', 'Rolle erwähnt aber Person
  fehlt'.

aggregation: sufficient=true nur wenn entweder (a) ≥3 distinkte
  Entitäten der gefragten Kategorie in den Chunks belegt sind,
  ODER (b) ein Chunk explizit aussagt, dass es nur X Einträge
  gibt und alle X enthalten sind. ALLEINSTEHEND 1-2 Entitäten
  bei einer 'welche'-Frage = insufficient.

total: sufficient=true nur wenn der HEADLINE-Wert (Gesamtsumme,
  Total, Gesamtaufwand) explizit in einem Chunk steht. Wenn nur
  Teilbeträge / Sub-Etappen / Tabellen-Sub-Zeilen vorhanden sind
  aber kein Total-Wert, dann insufficient — auch wenn die
  Teilbeträge sich aufsummieren liessen.

phrase: sufficient=true nur wenn ein Chunk die genaue Phrase
  oder eine semantisch identische Aussage enthält. Bei
  Negativ-Antwort ('Phrase nicht gefunden') verlange
  insufficient — der Agent soll dann via list_document_outline
  noch eine Datei explorieren statt sofort 'nichts gefunden' zu
  antworten.

out_of_scope: sufficient=true wenn die Frage erkennbar SIA 32+ /
  Ausführung / Bauprojekt-Phase betrifft und der Auftrag laut
  Chunks nur SIA 21/31 umfasst. Scope-Fallback ist KEINE
  Insuffizienz.
```

3. Removes the "Bias: lieber sufficient=true als false-negative" line. Replace with neutral: "Bewerte nüchtern. Eine `false`-Bewertung kostet maximal eine zusätzliche Retrieval-Runde; eine fälschliche `true`-Bewertung führt zu einer unvollständigen Antwort an den Nutzer."

4. Output schema unchanged: `{sufficient, missing, feedback}`. Adds an internal `question_type` field that's logged but not returned (helps debugging via LangSmith).

**Test**: extend `test_sufficiency.py` with one case per question type. Each test asserts the rater verdict is correct given an explicit chunk set:
- aggregation with 1 entity → insufficient
- aggregation with 3 entities → sufficient
- total with sub-rows but no headline → insufficient
- total with headline value → sufficient
- phrase not present → insufficient
- out-of-scope with scope-fallback content → sufficient

### T3. Forced retry routes to `list_document_outline`

**File**: `backend/app/routers/chats.py`, `backend/app/sufficiency.py`

When sufficiency fails, instead of `tool_choice="required"` (which lets the model pick any retrieval tool), pass `tool_choice={"type": "function", "function": {"name": "list_document_outline"}}` IF:

- ≥1 `search_chunks` was called this turn, AND
- 0 `list_document_outline` calls were made this turn, AND
- the rater verdict's `feedback` doesn't explicitly contradict (e.g., starts with "Synonym-Expansion ausreichend").

Otherwise (sufficiency fails after outline+section already explored), keep `tool_choice="required"` over the retrieval-only set.

The continuation-hint message (`build_continuation_hint`) gets a per-tool variant. When force-targeting outline, the hint reads:

```
SUFFICIENCY-CHECK: deine Antwort deckt die Frage nicht
vollständig ab. Fehlt: <missing>. Vorschlag: <feedback>.
Rufe als nächstes `list_document_outline` auf einer
wahrscheinlichen Datei auf (z.B. file_id=<top-cited file_id
prefix from current chunks>). Lies den Outline, identifiziere
die richtige Sektion, und rufe danach `read_section` mit dem
gefundenen Sektions-Namen auf. KEIN weiterer search_chunks-
Aufruf — der hat den richtigen Chunk nicht gerankt.
```

The `<top-cited file_id prefix>` is computed in `chats.py` from `collected_chunks` — pick the file_id that appears in the most retrieved chunks so far. This gives the model a concrete starting point instead of "explore some file".

**Code change in `chats.py`**:
- New helper `_top_cited_file_id_prefix(collected_chunks: list[RetrievedChunk]) -> str | None` (3 lines, Counter-based).
- New state: `outline_called_this_turn: bool` initialized False at turn start, set True when an outline executor runs.
- Sufficiency-fail branch: if `not outline_called_this_turn and any prior search_chunks ran`, set `force_tool_next_iter_target = "list_document_outline"` (new variable) and pass `tool_choice={"type":"function","function":{"name":"list_document_outline"}}`. Pass the file-id hint into `build_continuation_hint`.

**Test**: extend `test_chat_force_tool.py` — case where sufficiency=false after a search_chunks call, assert next chat.completions.create call has `tool_choice={"type":"function","function":{"name":"list_document_outline"}}` (not generic "required"). Case where outline already ran and sufficiency fails again → falls back to retrieval-only "required".

### T4. Outline-first hard gate in `read_section`

**Files**: `backend/app/tools/section.py`, `backend/app/routers/chats.py`

`read_section(file_id, section=...)` is currently advisory ("Section name must come from the outline" in description, not enforced). Make it enforceable.

**Plumbing**: the executor needs to know which `file_id`s have been outlined in the current turn. We thread a per-turn set through the executor signature. In `chats.py`, maintain `outlined_file_ids: set[str]` at turn-start; populate when `list_document_outline_executor` runs. Pass it into `read_section_executor` as a kwarg.

**Behavior in `read_section_executor`**:

```python
def read_section_executor(*, args, project_id, user_id, ref_offset,
                          outlined_file_ids: set[str] | None = None):
    section = (args.get("section") or "").strip()
    file_id_prefix = (args.get("file_id") or "").strip()
    # Resolve file_id_prefix to full UUID …
    full_file_id = resolve_file_id_prefix(file_id_prefix, ...)

    if section and outlined_file_ids is not None and full_file_id not in outlined_file_ids:
        return {
            "results": [],
            "_chunks": [],
            "error": {
                "code": "section_without_outline",
                "argument": "section",
                "guidance": (
                    f"Du hast `read_section(section='{section}')` "
                    f"aufgerufen, ohne vorher `list_document_outline` "
                    f"auf der Datei '{file_id_prefix}' auszuführen. "
                    f"Section-Namen müssen aus dem Outline stammen — "
                    f"Raten führt zu leeren Treffern. Rufe ZUERST "
                    f"`list_document_outline(file_id='{file_id_prefix}')` "
                    f"auf, lies die zurückgegebenen Section-Namen, und "
                    f"rufe `read_section` dann mit einem dieser Namen "
                    f"auf."
                ),
            },
        }
    # … existing logic
```

`page_from`+`page_to` calls (no `section`) bypass the gate — the model can still page-target without an outline.

This integrates with plan 17.3 T2's structured-error force-retry path: a `section_without_outline` error → `force_tool_next_iter = True`, the directive system message instructs the retry, and (combined with T3) the next call gets `tool_choice` pinned to `list_document_outline`.

**Test**: pytest in `test_tools_outline_section.py`:
- `read_section(section="X")` with empty `outlined_file_ids` → returns `section_without_outline` envelope.
- `read_section(section="X")` with `full_file_id` in `outlined_file_ids` → proceeds normally.
- `read_section(page_from=1, page_to=2)` with no section, empty outlined set → proceeds (no gate).

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| Rater overcorrects and rates correct answers as `insufficient` (extra retrieval rounds, latency cost) | Type-specific rubric is explicit about what "sufficient" means per type. Cap stays at one nudge per turn (existing 17.2 plumbing). Worst-case overhead = 1 extra Gemini call + 1 extra retrieval round per turn. |
| Forced outline call lands on the wrong file_id (most-cited ≠ correct file) | Outline returns the file's TOC compactly; even if the model needed Teil C, calling outline on Teil B costs ~few hundred tokens and the model can correct on the next iteration via a different `read_section` or another search. Better than the current "guess a section" failure mode. |
| `section_without_outline` gate breaks legitimate calls where the model already saw the outline in a prior turn (older message in history) | Acceptable. Outline is cheap to re-call (compact result). Cross-turn caching adds state we don't want. The gate triggers ONLY when the model fabricates a section name in the current turn. |
| Rater question-type classification is unreliable | The classification is internal to the rater's CoT, not a hard switch. If type is wrong, the rubric still asks "is the question answered factually?" and falls through to the general check. We log the type via LangSmith for debugging; trace audit will show if classification quality is acceptable. |
| Migration 0016 changing return type breaks the existing search.py mapping | Add the columns at the END of the RETURNS TABLE, after `rrf_score`. `_rpc_row_to_chunk` only reads named keys — extra fields are ignored. Backwards-compatible. |
| Rater context budget blown by 600-char excerpts × 30 chunks × added metadata | Metadata adds ~80 chars per chunk header (file_id, heading_path, block_type, chunk_index). Total: 30 × 680 ≈ 20k chars / ~7k tokens. Still fine for Gemini Flash 1M context. |
| Forced `list_document_outline` retry fires on a small file where outline isn't load-bearing (e.g., Teil C2 fee schedule) | Outline call is cheap (~few hundred tokens, no embeddings). If the outline shows few sections, the model can fall through to `read_section(page_from=..., page_to=...)` or another `search_chunks` next iteration. The retry isn't a one-shot kill; it's one of multiple iterations. |
