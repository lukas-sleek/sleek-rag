# 16. Hybrid Retrieval + Reranker — Close the Recall Gap Between Chat and Volltext

## Complexity: 🔴 Complex

Two new retrieval layers stacked on the existing pipeline (BM25 → RRF → cross-encoder rerank), plus a tightened chat-path prompt and a Projektanalyse v1 migration to the same retrieval pipeline so batch reports inherit the recall fix. Touches one Postgres migration with a backfill, one new client module, the `match_chunks_filtered` RPC, the `search_chunks` tool handler, the chat system prompt, and `projektanalyse.py:_answer_v1_sync`. Backwards-compatible by feature flag — old behavior reachable by setting `RETRIEVAL_MODE=vector_only` in `.env` until the new path is trusted; v2 (Volltext) stays as a manual escape valve.

---

## Background — what plan 14 + 15 left on the table

The agentic chat path (plan 14) does exactly one thing: cosine top-K over `match_chunks_filtered`. Plan 15 fixed `heading_path[]` so `section=` filters resolve deterministically. Neither plan addressed the **recall ceiling** of pure semantic retrieval.

UAT against the test user's "test" project (Teil A + Teil B + 2× C1) surfaced six failure modes that v2 (Volltext) handles for free but the chat path does not:

| Question | What v2 found | What chat missed | Why |
|---|---|---|---|
| Welche Bauherren? | SBB / Manor / KMU-Center / Balthasar / Kt. Luzern / VVL | All of them | Source chunks say **Grundeigentümer**, not Bauherr — synonym gap |
| Projektleiter? | Thomas Kieliger (Teil B Abb. 15) | Returned C1 form templates | Right answer lived in a figure caption with low term density; templates contained the literal word "Projektleiter" and won cosine |
| Termine? | Ausschreibungstermine (Teil A p.3) + Vorprojekt-Gantt (Teil B p.22) | Only the Gantt | One query embedding can't cover two clusters in two files |
| Bausumme? | CHF 39'114'000 headline + per-Etappe split | Per-Etappe only | Total row split off into its own small chunk by `chunk_size=500`; cosine ranked etappe-rich chunks higher |
| Drittprojekte? | PU Nord/Mitte/Süd, Bushof Ost, Renaturierung | Nothing | Source doc calls them **Schnittstellenprojekte** — synonym gap again |
| "späterer Phase zu detaillieren"? | "abschliessend zu definieren" + "Vertiefung bzw. Präzisierung" | Nothing | Paraphrase matches; pure cosine doesn't reliably rank paraphrase above lexical near-misses |

Common thread: **pure semantic, single-query, no rerank**. Three of six are synonym/lexical issues a BM25 channel would catch. Two are recall issues a wider top-K + rerank would catch. One is a multi-cluster issue a per-doc fan-out would catch.

User explicitly built v2 (Volltext) as the workaround. That's the comparison baseline this plan tries to retire.

## Goal

After this plan, both `search_chunks` (chat path) and Projektanalyse v1 (batch path) return results from a hybrid retriever (BM25 + cosine, fused via RRF) reranked by a managed cross-encoder. The chat path closes ≥80% of the v2-vs-chat gap on the six UAT questions above without changing the tool schema the model sees. The batch path closes ≥80% of the v2-vs-v1 gap on the same questions while running 11 parallel calls cheaper and faster than v2's full-corpus prompt.

Specifically:

1. New `tsvector` column on `document_chunks` with German FTS config + GIN index, populated automatically for new and existing chunks via a Postgres generated column.
2. New `match_chunks_hybrid` RPC: pulls top-N by cosine and top-N by FTS, fuses with **Reciprocal Rank Fusion** (RRF, k=60), respects all existing filters (`file_ids`, `block_type`, `page`, `figure_label`, `section`).
3. New ranking client (`backend/app/ranking_client.py`) — calls Google's **Vertex AI Ranking API** (`discoveryengine.googleapis.com/v1/projects/{PROJECT}/locations/global/rankingConfigs/default_ranking_config:rank`) using `semantic-ranker-default-004`. Authenticated via the **existing service-account JSON** we already use for Document AI + GCS. Same auth, same project, one new IAM role (`roles/discoveryengine.viewer`). Multilingual (25 languages incl. German), supports up to 200 records × 1024 tokens — comfortably more than our 30-doc rerank pass at ~500 chars/chunk. No new third-party dependency, no GPU, no separate API key.
4. `execute_search_chunks` pulls `pre_rerank_k` (default 30) from the hybrid RPC, reranks down to `top_k` (default 8). **Fail-open**: if the ranking call errors or times out, return the RRF order so the user still gets an answer.
5. Tightened `CHAT_SYSTEM_PROMPT`: ports the projektanalyse "PFLICHT-PRÜFUNG VOR 'Nicht in den Dokumenten gefunden'" rule, plus an explicit "before answering 'not found', try one synonym query AND one per-file fan-out" instruction.
6. **(Optional T6)** Server-side query expansion: when the user question contains fewer than ~6 content tokens or matches a "welche/wer/welcher" pattern, expand to 2–3 sub-queries via a fast Gemini call, run hybrid+rerank per sub-query, RRF the unions. Off by default behind a config flag — ship only if T1–T5 doesn't close the gap on Bauherren/Drittprojekte.
7. **Projektanalyse v1 routes through `execute_search_chunks`** with batch-tuned settings (`pre_rerank_k=80`, `top_k=15`) — replaces the legacy `_by_vector` per-question call. Same hybrid+rerank pipeline as the chat path, plus extra recall headroom because batch is already a multi-second operation.
8. **(Optional T8)** Per-question synonym fan-out + per-file sweep for the four aggregation questions in the Projektanalyse template (Bauherren, Termine, Drittprojekte, Bauzeit). Ship only if T7's UAT against the 11-question template still misses on those four.

## Why Vertex AI Ranking API and not a self-hosted reranker

We have access to two viable rerankers:

- **Google Vertex AI Ranking API** (`semantic-ranker-default-004`) — managed, multilingual, GA, billed per 1k requests, same service account as Document AI/GCS, no infra.
- **Self-hosted Qwen3-Reranker-4B** on llama-server `--rerank` — runs on the user's GPU box, OpenAI-compatible HTTP, free per-request but adds an external dependency this app currently does not have.

**Pick the managed one** for sleek-rag. Reasons:

1. **Auth + ops symmetry**: we already authenticate to GCP for Document AI Layout Parser and GCS staging. Adding `roles/discoveryengine.viewer` to the same service account is a one-line IAM change. The self-hosted path would introduce a second auth surface, a separate uptime concern, a network reachability assumption, and per-deploy SSH/secret plumbing.
2. **Latency**: Ranking API SLA is ~150–300 ms for 30 records — comparable to the local box for our query shape, without the GPU contention risk.
3. **Multilingual coverage**: `semantic-ranker-default-004` is tuned on 25 languages including German; we don't have to validate a German fine-tune story.
4. **Cost**: ~$1 per 1k rank calls (typical price tier). At our volume this is rounding error compared to the Document AI ingest cost we're already paying.
5. **Footnote on the alternative**: if a future deployment is air-gapped or wants to drop the GCP rerank line item, the abstraction in `ranking_client.py` is small (one HTTP call) — swap it for an OpenAI-compatible reranker shim later. Out of scope here.

We are **not** moving to Vertex AI Search or Vertex AI RAG Engine — those would require migrating chunk storage out of Supabase pgvector, losing RLS, the `chunk_images` table, and the `project_files` ↔ `document_chunks` join model that the rest of this app is built on. The Ranking API is the only piece of the Discovery Engine surface that's usable as a standalone primitive without buying into the full datastore.

## Architectural decisions

### Why RRF over weighted-sum fusion

RRF is rank-based, not score-based, so it doesn't need score calibration between cosine (similarity, [-1, 1]) and ts_rank (`ts_rank_cd`, unbounded positive). Two parallel queries each contribute `1 / (k + rank_i)` to a chunk's fused score; chunks that rank well in either channel float up. `k=60` is the canonical Anserini/Pyserini default and gives stable behavior with N up to ~100. No tuning needed.

Weighted-sum (`α·cos + (1-α)·bm25`) requires per-corpus α tuning and re-tuning when the embedding model or BM25 stats drift. RRF doesn't. Pick RRF.

### Why a separate `tsvector` column instead of `to_tsvector` in the query

Computing `to_tsvector('german', content)` per query is a sequential scan. With a stored generated column + GIN index, FTS is sub-millisecond. The table is small now (~hundreds of chunks per project) but this scales linearly with corpus size — pay the index cost once at ingest, never per query.

### German FTS config + heading boost

Postgres ships with a `german` text search config that does German stemming + stop words. Apply it to `content || ' ' || array_to_string(heading_path, ' ')` so heading terms count toward FTS — chunks under "1.3 FRAGEN" should match `?fragen` even if the body uses synonyms.

`tsvector` is a generated column on `(content, heading_path)`:

```sql
alter table public.document_chunks
  add column content_tsv tsvector
  generated always as (
    setweight(to_tsvector('german', coalesce(content, '')), 'B') ||
    setweight(to_tsvector('german', coalesce(array_to_string(heading_path, ' '), '')), 'A')
  ) stored;

create index document_chunks_content_tsv_gin
  on public.document_chunks using gin (content_tsv);
```

`setweight('A')` on heading_path means heading hits beat body hits at equal frequency — useful for "Drittprojekte" / "Schnittstellenprojekte" type lookups where the right answer is a section title.

Generated columns auto-populate on insert; **no separate backfill is needed for FTS** — the column is computed on the fly for existing rows the moment the migration runs (Postgres rewrites the table). For our row count this is a single-second migration.

### Hybrid RPC contract

`match_chunks_hybrid` replaces `match_chunks_filtered` as the primary retrieval RPC. Old RPC stays callable for now (keeps `RETRIEVAL_MODE=vector_only` working) but the tool handler defaults to hybrid.

```sql
create or replace function public.match_chunks_hybrid(
  p_project_id uuid,
  p_embedding vector(768),
  p_query text,
  p_top_k int default 30,           -- pre-rerank K, larger than before
  p_file_ids uuid[] default null,
  p_block_type text default null,
  p_page int default null,
  p_figure_label text default null,
  p_heading_prefix text default null,
  p_rrf_k int default 60
) returns table (
  id uuid,
  file_id uuid,
  project_id uuid,
  content text,
  page_start int,
  page_end int,
  figure_label text,
  block_type text,
  filename text,
  vec_similarity float,
  fts_rank float,
  rrf_score float
) language sql stable as $$
  with
  filtered as (
    select c.id, c.file_id, c.project_id, c.content, c.page_start, c.page_end,
           c.figure_label, c.block_type, c.embedding, c.content_tsv,
           pf.filename
    from public.document_chunks c
    join public.project_files pf on pf.id = c.file_id
    where c.project_id = p_project_id
      and (p_file_ids is null or c.file_id = any(p_file_ids))
      and (p_block_type is null or c.block_type = p_block_type)
      and (p_page is null or (c.page_start <= p_page and c.page_end >= p_page))
      and (p_figure_label is null or c.figure_label = p_figure_label)
      and (
        p_heading_prefix is null
        or (
          c.heading_path is not null
          and exists (
            select 1 from unnest(c.heading_path) h
            where h ilike p_heading_prefix || '%'
          )
        )
      )
  ),
  vec as (
    select id, 1 - (embedding <=> p_embedding) as sim,
           row_number() over (order by embedding <=> p_embedding) as rnk
    from filtered
    order by embedding <=> p_embedding
    limit p_top_k * 2
  ),
  fts as (
    select id, ts_rank_cd(content_tsv, plainto_tsquery('german', p_query)) as r,
           row_number() over (
             order by ts_rank_cd(content_tsv, plainto_tsquery('german', p_query)) desc
           ) as rnk
    from filtered
    where content_tsv @@ plainto_tsquery('german', p_query)
    order by r desc
    limit p_top_k * 2
  ),
  fused as (
    select coalesce(v.id, f.id) as id,
           coalesce(v.sim, 0)   as vec_similarity,
           coalesce(f.r, 0)     as fts_rank,
           coalesce(1.0 / (p_rrf_k + v.rnk), 0) +
           coalesce(1.0 / (p_rrf_k + f.rnk), 0) as rrf_score
    from vec v
    full outer join fts f on f.id = v.id
  )
  select f.id, f.file_id, f.project_id, f.content, f.page_start, f.page_end,
         f.figure_label, f.block_type, f.filename,
         u.vec_similarity, u.fts_rank, u.rrf_score
  from fused u
  join filtered f on f.id = u.id
  order by u.rrf_score desc,
           case
             when p_page is null then 1
             when f.page_start = p_page and f.page_end = p_page then 0
             when (f.page_end - f.page_start) <= 1 then 1
             else 2
           end
  limit greatest(1, least(coalesce(p_top_k, 30), 100));
$$;
```

Notes:
- We pull `p_top_k * 2` from each channel before fusion — RRF needs enough candidates to find rank-1 of one channel inside the top of the other.
- The page-bucket tiebreaker from `match_chunks_filtered` is preserved as a secondary sort, so exact-page hits still beat narrow-span at equal RRF score.
- If `p_query` is empty / yields no FTS hits, the RPC degrades to pure-vector via the `coalesce(...,0)` arms — no separate code path needed.
- Cap is bumped from 20 → 100 since we pre-rerank.

### Ranking client — Vertex AI Ranking API

New module `backend/app/ranking_client.py`. One public function:

```python
def rank(
    query: str,
    documents: list[str],
    top_n: int,
    *,
    timeout: float = 4.0,
) -> list[tuple[int, float]]:
    """Return list of (original_index, relevance_score) sorted by score desc,
    length ≤ top_n. Fails open: returns the input order with score=0 on any
    error/timeout."""
```

Implementation:

- **Auth**: reuse `service_account.Credentials.from_service_account_file(settings.gcp_service_account_json_path)` exactly as `_gcs()` does in `workers/ingest.py:36`. Scope: `https://www.googleapis.com/auth/cloud-platform`. Refresh via `google.auth.transport.requests.Request()` before each call (token TTL is 1h; for our QPS this is fine — caching is a follow-up if hot-path latency matters).
- **HTTP**: POST via `httpx.Client(timeout=settings.rerank_timeout_sec)` to:
  ```
  https://discoveryengine.googleapis.com/v1/projects/{settings.gcp_project_id}/locations/global/rankingConfigs/default_ranking_config:rank
  ```
- **Body**:
  ```json
  {
    "model": "semantic-ranker-default-004",
    "query": "<query>",
    "records": [
      {"id": "0", "content": "<doc 0>"},
      {"id": "1", "content": "<doc 1>"},
      ...
    ],
    "topN": top_n,
    "ignoreRecordDetailsInResponse": true
  }
  ```
  `ignoreRecordDetailsInResponse: true` strips title/content from the response — we only need `id` + `score` to reorder our local chunk list.
- **Response parsing**: `{"records": [{"id": "<int-as-str>", "score": float}, ...]}`. Map `id` back to the original index, return `[(int(id), score), ...]`.
- **LangSmith**: `@traceable(run_type="tool", name="rank")` so the rerank step is visible per-turn in the trace UI alongside `search_chunks` and the embeddings call.
- **Fail-open**: any exception, non-200, or timeout returns `[(i, 0.0) for i in range(min(len(documents), top_n))]` and emits one `log.warning(...)`. The chat path must keep working when the Ranking API is down — silently falls back to RRF order.

**Token budget**: Ranking API caps records at 1024 tokens each (~4000 chars). Our `chunk_size=500` chunks fit ~200 chars + heading prefix → ~80 tokens/record → well below the cap. No truncation needed.

Config additions in `backend/app/config.py`:

```python
rerank_model: str = "semantic-ranker-default-004"
rerank_timeout_sec: float = 4.0
retrieval_mode: str = "hybrid"  # "hybrid" | "vector_only" — escape hatch
pre_rerank_k: int = 30           # candidates pulled from RPC before rerank
```

No new API key, no new base URL. The Ranking API URL is fully derived from `settings.gcp_project_id`, which is already populated.

### Tool handler changes

`execute_search_chunks` (`backend/app/tools/search.py`) gets a small refactor:

```
1. Parse args, resolve file_ids prefixes (unchanged).
2. Embed query (unchanged).
3. If settings.retrieval_mode == "vector_only":
     call match_chunks_hybrid with p_query="" (degenerate to vector-only) and
     p_top_k = top_k → return as before. No rerank.
4. Else (hybrid):
     call match_chunks_hybrid with p_query=query and
     p_top_k = settings.pre_rerank_k (default 30).
     scores = ranking_client.rank(query, [c.content for c in chunks], top_n=top_k).
     reorder chunks by scores; trim to top_k.
5. Attach images (unchanged); build `ref` indices on the final post-rerank order.
```

`top_k` semantics from the model's perspective don't change — model asks for 8, gets 8. We just pull 30 internally, rerank, return 8. Pre-rerank K is configured via env, never via tool args.

Trace via LangSmith: existing `@traceable(run_type="tool", name="search_chunks")` wraps the whole call; the new `@traceable(... name="rank")` on the ranking client gives per-stage latency and score deltas in the trace UI.

### Page-bucket rule under RRF

Keep it as the secondary sort. RRF gives us the right global ordering; the page-bucket rule kicks in only when two chunks have identical RRF (same rank in both channels). Behavior on `page=N` queries should be indistinguishable from plan 14 except that synonym/exact-string hits now bubble into the candidate pool.

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| Ranking API latency adds noticeable per-turn cost | 4.0s timeout with fail-open. `semantic-ranker-default-004` typically returns in 150–300ms for 30 records; `semantic-ranker-fast-004` is ~3× faster if we need the latency budget. Easy env-var swap. |
| German FTS config misses words with foreign roots ("PU Nord" → tokens `pu` + `nord`, both noise) | The setweight('A') boost on heading_path catches the section-title path; in practice "Schnittstellenprojekt" is German enough that stemming nails it. If we see misses, switch the body weight to `simple` (no stemming) and keep heading on `german`. |
| RRF k=60 wrong for small candidate pool | k is the canonical default and behaves well for N ≤ 100. For our scale (<5k chunks per project) this isn't tunable. Don't expose it. |
| Generated `tsvector` column slows ingest inserts | One `to_tsvector` call per chunk at insert time. For 100 chunks/file it's <10ms total — invisible compared to the embedding round-trip. |
| Service-account token refresh blocks every rank call | Token TTL is 1h; per-call refresh is one HTTP round-trip to Google's token endpoint (~30ms). If trace shows this is hot-path-relevant, cache the refreshed credentials object module-level (same pattern as `_gcs_client` in `workers/ingest.py:34`). Optimization, not a blocker. |
| Ranking API quota exhausted under load | Free quota is generous for our QPS; if hit, fail-open kicks in and the user gets RRF order. Long-term mitigation is rate-limiting on our side, not in scope here. |
| Prompt addition makes the model over-call `search_chunks` (3-4 calls per turn) | `MAX_TOOL_ITERATIONS=4` already caps it. Each call is ~50ms RPC + ~200ms rank. Worst case: 4 × 250ms = 1.0s of retrieval per turn. Acceptable. |
| Generated-column migration locks the table | Generated columns are filled during the initial table rewrite triggered by `ALTER TABLE … ADD COLUMN GENERATED`. On a few thousand rows this is one transaction, sub-second. If the corpus grows beyond ~1M chunks we'd batch — not relevant now. |
| New IAM role missed at deploy | If `discoveryengine.viewer` is not granted, the rank call returns 403 → fail-open → chat works on RRF order. Visible in LangSmith; one-line fix. Not chat-breaking. |

## Out of scope

- Deleting Projektanalyse v2 (Volltext). Stays as the manual escape valve in case v1 (post-T7) has a regression on a specific question. Revisit removing it once v1 has run clean against several real-world reports.
- Deleting `_by_vector` from `retrieval.py`. After T7 it has no callers, but kept for one release as a safety net. Drop in a follow-up plan.
- Vertex AI Search / Vertex AI RAG Engine migration. Would require moving chunk storage out of Supabase pgvector and abandoning RLS, `chunk_images`, the `project_files` join model. Net negative for this app — only consider if the Supabase-side architecture is otherwise being replaced.
- Multi-vector / late-interaction retrievers (ColBERT-style). Bigger architectural shift — revisit only if hybrid + rerank still leaves gaps.
- Larger chunk sizes for tables (`enable_table_annotation=True` is set; we're not currently using the table-as-single-chunk output. Could be its own plan if Bausumme-style table answers are still flaky after this plan.)
- Removing prepended `#`-headings from `chunk.content` (separate decision; may help embedding quality).
- Dropping the legacy `match_chunks_filtered` RPC (drop in a follow-up after one full release of `match_chunks_hybrid` running clean).
- Air-gapped / GCP-free reranker fallback (swap `ranking_client.py` for an OpenAI-compatible reranker shim — small abstraction, deferred until needed).
