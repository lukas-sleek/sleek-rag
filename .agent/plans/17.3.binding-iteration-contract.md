# 17.3 Binding Iteration Contract — Force-Tool on Sufficiency-Fail, Force-Retry on Tool-Error, Auto-Escalate to v2

## Complexity: ⚠️ Medium

Plan 17.2 wired sufficiency check, structured error envelopes, and v2-as-escalation tool — but the live UAT (LangSmith chat `f0991bbb`, 2026-04-28) shows the model treating all three as advisory, not binding:

- **`sufficient=false` fired 3× across 6 questions, the model called zero follow-up tools.** Q3 even produced an empty answer string after the verdict came back insufficient.
- **Empty-`query` `search_chunks` call returned the `missing_required_argument` error envelope; the model surfaced "Bitte geben Sie an…" to the user verbatim** (Q4) — direct violation of `error.guidance`.
- **`run_projektanalyse_v2` was called zero times** across all 6 questions, including Q3 (Projektleiter) where v2's full-corpus answer is trivially correct.
- **`list_document_outline` and `read_section` were also called zero times** — the entire navigational layer from plan 17 is dead in production.

The fix is to stop relying on the model to read documentation and instead enforce the contract in the dispatcher: when sufficiency fails, force the next turn to be a tool call (`tool_choice="required"`); when a tool errors out, force a retry of that same tool with corrected args; when both retries and sufficiency loops have been exhausted, auto-call `run_projektanalyse_v2` server-side as the escalation.

This is one architectural shift: from **prompt-coercion** (17.1's regex injection — failed) and **doc-coercion** (17.2's tool descriptions + error guidance — failed) to **dispatch-coercion** (17.3 — the loop won't *let* the model emit prose when it should be retrieving). Google's documented function-calling guide explicitly supports this — `tool_choice` modes (`auto` / `required` / `{name}`) are designed for exactly this control surface.

---

## Goal

After this plan, against the same 11-question UAT:

- **Zero user-handoff phrasings** in any answer ("Möchtest du", "Bitte geben Sie", "bräuchte ich…").
- **Sufficiency=false → ≥1 follow-up retrieval tool call** in 100 % of traces where it fired.
- **Empty/invalid tool-args → automatic retry** in 100 % of cases (no error string ever reaches the user).
- **v2 escalation auto-fires** after 2 sufficiency loops + still insufficient, OR after 3 consecutive empty/error tool calls — visible as a `run_projektanalyse_v2` child run in the LangSmith trace.
- **≥9/11 questions pass with citations** (one full grade above 17.2's 2/11 PASS).

## Tasks

### T1. Force tool-call on sufficiency-fail via `tool_choice="required"`

**File**: `backend/app/routers/chats.py`

Right now (chats.py:530–556), when sufficiency fails we append a continuation hint as a system message and `continue` the loop. The next iteration calls `chat.completions.create` with no `tool_choice` override — the model is free to emit prose again, and that's exactly what's happening.

Change: when `sufficiency_already_nudged` is being set in the same loop step, also set a flag `force_tool_next_iter = True`. In the next loop iteration's `chat.completions.create` call, pass `tool_choice="required"` (OpenAI-compat — Gemini honors this; verified per the function-calling guide). After the next iteration fires its tool calls, reset the flag.

```python
# new state at top of _send_message_stream:
force_tool_next_iter = False

# inside chat.completions.create call:
create_kwargs = dict(
    model=settings.gemini_chat_model,
    messages=messages,
    tools=tools,
    stream=True,
)
if force_tool_next_iter:
    create_kwargs["tool_choice"] = "required"
    force_tool_next_iter = False  # consume the flag
stream = await asyncio.to_thread(
    lambda: gemini_client_untraced().chat.completions.create(**create_kwargs)
)

# inside the sufficiency-fail branch:
sufficiency_already_nudged = True
force_tool_next_iter = True  # next call MUST emit a tool call
```

The continuation hint stays — it tells the model *which* tool to pick — but the loop now refuses to accept "let me think out loud" as an answer when sufficiency said no.

**Test**: pytest with mocked Gemini stream returning prose on iteration N, tool call on iteration N+1; assert the second `chat.completions.create` call args contain `tool_choice="required"`.

### T2. Force retry on structured tool errors

**File**: `backend/app/routers/chats.py`, `backend/app/tools/search.py` / `outline.py` / `section.py`

Right now, structured errors (`{"results": [], "error": {"code": "missing_required_argument", ...}}`) get serialized into the `tool` message content and appended to messages — exactly like a successful tool result. The model reads the JSON and decides what to do; in Q4 it decided "ask the user."

Change: detect a structured error in the tool dispatch loop (chats.py:572–621) and treat it as a retry signal, not a final result. After dispatching all tools in a turn, count how many returned `error` envelopes:

- **All errored**: don't append the tool messages back into history with their error content as ground truth. Instead, append one synthetic system message: `"TOOL-RETRY: dein letzter Tool-Aufruf hatte fehlende oder ungültige Argumente ({code}). Korrigiere {argument} und rufe das Tool sofort erneut auf — keine Antwort an den Nutzer."` Set `force_tool_next_iter = True` for the next iteration. Increment a per-turn counter `tool_error_streak`.
- **Some errored, some succeeded**: append the tool messages normally (the successes will keep the loop moving), but if the model in the next iteration *still* doesn't issue a follow-up tool call and emits prose, the sufficiency check (T1) catches it.
- **`tool_error_streak >= 3`**: trigger T3's auto-escalation to v2 instead of looping further.

**Test**: pytest with `execute_search_chunks` returning the `missing_required_argument` envelope; assert next iteration's messages list contains the TOOL-RETRY system message AND `tool_choice="required"` is set; assert no error guidance text leaks into the SSE delta stream.

### T4. Tighten the empty-`query` envelope to be self-recovering

**File**: `backend/app/tools/search.py`

The Q4 trace showed the model called `search_chunks({})` — empty args entirely. The current envelope returns `error.guidance` telling the model to retry; with T1 + T2 in place that retry is now forced, but we can make it less likely in the first place.

Change the `search_chunks` tool description (already restructured in 17.2) to add one explicit example:

```
PARAMETER `query` (REQUIRED): always populate. Example transformations:
- "Welche Termine sind vorgesehen?" → query="Termine Meilensteine"
- "Was ist die Bausumme?" → query="Bausumme Baukosten Total"
- "Wer ist der Projektleiter?" → query="Projektleiter Name"
Never call `search_chunks` without query. Never call with query="".
```

The pattern transformation is what the LangSmith trace showed the model failing on — Q4's "Welche Termine" should produce `query="Termine"`, not no query at all. The example transformations remove the ambiguity.

**Test**: pytest assertion that `SEARCH_CHUNKS_TOOL["function"]["description"]` contains the three example transformations.

### T5. Increase chunk excerpt budget for sufficiency rater

**File**: `backend/app/sufficiency.py`

The rater currently sees 200-char excerpts (sufficiency.py:75). For Q3 (Projektleiter), the answer lives in a figure caption ("Abb. 15: Thomas Kieliger, Projektleiter Teilprojekt 2") — 200 chars truncates this kind of fact-bearing chunk before the relevant entity name appears, causing false-positive `sufficient=true` on incomplete coverage (the Q2 Bauherren miss is the same pattern).

Change `_format_chunks_for_rater`:
- Excerpt budget: 200 → 600 chars per chunk.
- Include `figure_label` AND `heading_path` in the chunk header (currently only `figure_label`). Heading context is what makes the rater able to say "this chunk is from section 'Projektorganisation' — entity name should be here".
- Token budget cap: hard-stop at 30 chunks rendered (~18k tokens at 600 chars each — still under Gemini Flash's context window with margin).

**Test**: existing sufficiency tests already assert verdict shape; add one new test that asserts the formatted chunk header includes both `figure_label` and `heading_path`, and excerpts are 600-char max.

## Out of scope (explicit)

- Replacing the sufficiency rater with a different model. The 200→600 char fix in T5 is the cheap win; a structured rubric (per-question schema like "list all entities expected") is a future plan.
- Changing the rerank pipeline or `pre_rerank_k`. Retrieval recall is fine; the LangSmith audit confirms 15 hits returned every non-empty call.
- Removing `run_projektanalyse` (v1). Still useful for the template-driven projektanalyse UI. v2 is the per-question escalation; v1 is the multi-question batch report.
- Adding parallel async tool dispatch. Sequential-with-loop is fine for our load and matches what the OpenAI-compat shim handles cleanly.
- Streaming the sufficiency rater's reasoning to the user. Internal-only.

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| `tool_choice="required"` makes Gemini emit nonsense tool calls (e.g., `search_chunks(query="")`) just to satisfy the constraint | T2's structured-error retry catches empty-query calls and forces correction. T4's tool description with example transformations primes the model to fill `query` with something sensible. |
| Auto-escalation to v2 fires too eagerly and blows up token cost on questions the deep-dive could answer | Trigger conditions in T3 are intentionally conservative (2 failed sufficiency rounds OR 3 tool-error streak OR 5 iterations with zero chunks). v2 cost is bounded — one call, ~100k tokens — not catastrophic. |
| `tool_choice="required"` triggers infinite loops if the model emits the same broken call every iteration | `MAX_TOOL_ITERATIONS=12` is the upper bound. T2's `tool_error_streak` triggers v2 escalation at 3, well below the cap. |
| Forcing tool retry hides legitimate model decisions to abstain ("not in corpus") | Scope-fallback is in the system prompt and triggers BEFORE retrieval (no chunks retrieved → sufficiency skips → no force). The model can still abstain on out-of-scope questions. The fix only forces tool calls in scenarios where retrieval already happened and was deemed insufficient. |
| Extending `stream_projektanalyse_v2` signature breaks the existing template-based UI flow | Keep template as the primary parameter; `question` is purely additive (optional kwarg). All existing call sites stay unchanged. |
| Sufficiency rater with 600-char excerpts hits Gemini context limits on 30-chunk turns | 30 × 600 = 18k chars ≈ 6k tokens content + 2k prompt = 8k total. Gemini 2.5 Flash supports 1M context. Safe. |
| `tool_choice="required"` not honored by Gemini's OpenAI-compat shim | Per Google's function-calling guide, Gemini supports `tool_choice` in OpenAI-compat mode. If not honored, the structured-error retry from T2 + the continuation hint from sufficiency is the fallback (degrades to current 17.2 behavior, no regression). |
