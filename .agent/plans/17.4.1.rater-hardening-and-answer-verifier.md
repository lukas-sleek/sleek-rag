### G1. Rater question-type hard rules

**File**: `backend/app/sufficiency.py`

The current rubric (post-17.4) does CoT classification but doesn't enforce structural cues from the question text. Q2 misclassified because "Welche Bauherren" matches "single entity" if Hochdorf appears prominently in chunks. Tighten the classification step:

Replace the `SCHRITT 1 — KLASSIFIZIERE DEN FRAGENTYP` block with explicit cue-based rules BEFORE the type definitions:

```
SCHRITT 1 — KLASSIFIZIERE DEN FRAGENTYP

Wende diese strukturellen Cues ZUERST an. Sie überschreiben deine
Intuition über den Inhalt der Chunks:

CUE-A (Aggregation, hart): wenn die Frage eines dieser Muster
  enthält, ist der Typ IMMER `aggregation`, unabhängig davon
  wieviele Entitäten in den Chunks stehen:
    • "welche [Plural-Substantiv]" (welche Bauherren, welche
      Termine, welche Drittprojekte, welche Elemente, welche
      Mandate)
    • "alle [Substantiv]"
    • "wer ist beteiligt", "wer sind die Beteiligten"
    • "Liste aller …", "auflisten"
    • Substantiv im Plural mit bestimmtem Artikel ("die
      Beteiligten", "die Termine")

CUE-B (Total, hart): wenn die Frage einen Headline-/
  Gesamtwert verlangt, ist der Typ IMMER `total`:
    • "Bausumme", "Gesamtkosten", "Gesamtaufwand"
    • "wie viele Stunden insgesamt", "Gesamtanzahl"
    • "Total", "Gesamttermin", "Endsumme"
    • "Gesamtprojektsumme"

CUE-C (Phrase, hart): wenn die Frage explizit nach einer
  Wortlaut-Suche fragt:
    • "steht der Satz / der Kommentar / die Phrase X in den
      Plänen/Dokumenten?"
    • Anführungszeichen um eine Mehrwort-Phrase
    • "wird Y wörtlich erwähnt?"

CUE-D (Out-of-scope, hart): wenn die Frage erkennbar
  SIA-Phasen ausserhalb 21/31 betrifft:
    • "Bauprojekt", "Ausführung", "Ausführungsprojekt"
    • "SIA 32", "SIA 33", "SIA 41", "SIA 51"

Erst wenn KEINER dieser Cues greift, klassifiziere als
`point` (Einzelfakt-Frage).

Hinweis: bei Q2 "Welche Bauherren sind beteiligt?" greift
CUE-A (welche + Plural Bauherren) → IMMER aggregation,
auch wenn nur 1 Entität in den Chunks ist.
```

The CUE listing is the load-bearing change. The rest of the rubric (per-type sufficiency rules) stays from 17.4.

### G4. Answer-correctness verifier

**Files**: `backend/app/answer_verifier.py` (new), `backend/app/routers/chats.py`

A second autorater that runs AFTER the model emits a final-text iteration AND sufficiency=true. Catches Q10-style inversions where the chunks contain the right evidence but the model paraphrased it wrong.

```python
# backend/app/answer_verifier.py
"""Answer-correctness verifier — second-pass autorater (plan 17.4.1 G4).

Sufficiency rates COVERAGE; this rates CORRECTNESS. Catches the case
where retrieved chunks contain the right evidence but the model paraphrased
or interpreted them wrong (e.g., Q10: chunks say "X is to be coordinated
by the offerer, executed by separate specialists" → model writes "X is
part of the offerer's contract").

Wired into the agent loop after sufficiency=true and before streaming.
Triggered only on `question_type ∈ {point, aggregation, total}` (the types
where inversion/contradiction is possible). Skipped on phrase / out_of_scope.

Cost: one Gemini call per chat turn that passes the sufficiency gate
(~500ms, ~3k tokens). Skipped if no chunks were retrieved.
"""
from __future__ import annotations

import json
import logging
from typing import TypedDict

from langsmith import traceable
from app.config import settings
from app.gemini_client import gemini_client
from app.retrieval import RetrievedChunk

log = logging.getLogger(__name__)

class VerifierVerdict(TypedDict):
    ok: bool
    issue: str | None  # ≤ 1 sentence describing the inversion/contradiction
    fix: str | None    # ≤ 1 sentence directive for the next iteration


_VERIFIER_INSTRUCTION = (
    "Du bist ein Korrektheits-Verifier für ein RAG-System. Du bekommst "
    "(1) eine Nutzerfrage, (2) den ENTWURF einer Assistenten-Antwort, "
    "(3) die Chunks (mit Excerpts), die der Assistent zur Erzeugung des "
    "Entwurfs gesehen hat. Deine Aufgabe: prüfe, ob der Entwurf das, was "
    "in den Chunks steht, KORREKT wiedergibt.\n\n"
    "PRÜFE GENAU DIESE FAILURE-MODES:\n"
    "1. INVERSION: der Entwurf behauptet das Gegenteil dessen, was in "
    "   den Chunks steht (z.B. 'Vermessung ist Teil des Auftrags' wenn "
    "   die Chunks sagen 'Vermessung wird in separaten Mandaten vergeben "
    "   und ist vom Anbieter nur zu koordinieren').\n"
    "2. FABRIKATION: der Entwurf nennt einen Wert/Fakt, der in den "
    "   Chunks NICHT steht, insbesondere bei Summen, Totals, Stundenzahlen.\n"
    "3. ENTITY-MIX-UP: der Entwurf weist eine Aussage der falschen "
    "   Person/Firma/Phase zu (z.B. 'TP1-Leiter Kieliger' wenn die Chunks "
    "   Kieliger als TP2-Leiter belegen).\n"
    "4. SCOPE-FEHLER: der Entwurf antwortet 'nicht gefunden' obwohl ein "
    "   Chunk die Antwort enthält, oder umgekehrt.\n\n"
    "Du sollst NICHT nach Vollständigkeit prüfen — das macht ein anderer "
    "Rater. Nur KORREKTHEIT der vorhandenen Aussagen.\n\n"
    "Bei `ok=false` MUSST du in `issue` (≤1 Satz) den konkreten "
    "Widerspruch nennen und in `fix` (≤1 Satz) sagen, wie die Antwort "
    "korrigiert werden soll (z.B. 'Antworte: Vermessung ist NICHT Teil des "
    "Auftrags, sondern in separaten Mandaten an Spezialisten vergeben').\n\n"
    "ANTWORTE AUSSCHLIESSLICH MIT JSON:\n"
    '{"ok": true|false, "issue": "..." oder null, "fix": "..." oder null}'
)


def _format_chunks_for_verifier(chunks: list[RetrievedChunk]) -> str:
    """Same shape as sufficiency rater (heading + 600 chars)."""
    # … reuse logic from sufficiency._format_chunks_for_rater …


@traceable(run_type="llm", name="answer_verifier")
def verify_answer(
    *, question: str, draft: str, chunks: list[RetrievedChunk],
    question_type: str | None = None,
) -> VerifierVerdict:
    """Returns ok=true if the draft accurately reflects the chunks. Fail-open
    on Gemini errors (we don't want the verifier blocking answers when
    upstream is flaky)."""
    if not chunks or not draft.strip():
        return {"ok": True, "issue": None, "fix": None}
    if question_type in {"phrase", "out_of_scope"}:
        # Phrase: success/failure is a coverage check, sufficiency already
        # handled it. Out-of-scope: pure prompt-emission, no synthesis.
        return {"ok": True, "issue": None, "fix": None}

    user_text = (
        f"Nutzerfrage:\n{question}\n\n"
        f"Entwurf-Antwort:\n{draft}\n\n"
        f"Retrieved chunks:\n{_format_chunks_for_verifier(chunks)}"
    )
    try:
        resp = gemini_client().chat.completions.create(
            model=settings.gemini_chat_model,
            messages=[
                {"role": "system", "content": _VERIFIER_INSTRUCTION},
                {"role": "user", "content": user_text},
            ],
            response_format={"type": "json_object"},
            extra_body={"reasoning_effort": "none"},
        )
    except Exception as exc:
        log.warning("answer_verifier: gemini call failed: %s", exc)
        return {"ok": True, "issue": None, "fix": None}

    raw = (resp.choices[0].message.content or "").strip()
    if not raw:
        return {"ok": True, "issue": None, "fix": None}
    try:
        parsed = json.loads(raw)
    except json.JSONDecodeError:
        return {"ok": True, "issue": None, "fix": None}
    ok = bool(parsed.get("ok", True))
    return {
        "ok": ok,
        "issue": parsed.get("issue") if not ok and isinstance(parsed.get("issue"), str) else None,
        "fix": parsed.get("fix") if not ok and isinstance(parsed.get("fix"), str) else None,
    }


def build_verifier_correction_hint(verdict: VerifierVerdict) -> str:
    parts = ["KORREKTHEITS-CHECK: dein Antwort-Entwurf scheint die Chunks falsch wiederzugeben."]
    if verdict.get("issue"):
        parts.append(f"Problem: {verdict['issue']}")
    if verdict.get("fix"):
        parts.append(f"Korrektur: {verdict['fix']}")
    parts.append("Formuliere die Antwort neu — ZITAT-PFLICHT: gib die belegende Stelle in deiner neuen Antwort wörtlich (oder als nahes Paraphrase) wieder, damit der Bezug zu den Chunks eindeutig ist.")
    return " ".join(parts)
```

**Subtask F8a (migration)**: New SQL function `chunks_on_page(p_file_id uuid, p_page int)` returning the same shape as `chunks_in_range`, ordered by `chunk_index`. Apply via `mcp__supabase__apply_migration`.

```sql
create or replace function public.chunks_on_page(
  p_file_id uuid,
  p_page int
)
returns table (
  id uuid,
  file_id uuid,
  project_id uuid,
  content text,
  page_start int,
  page_end int,
  figure_label text,
  block_type text,
  filename text,
  heading_path text[],
  chunk_index int
) language sql stable as $$
  select c.id, c.file_id, c.project_id, c.content, c.page_start, c.page_end,
         c.figure_label, c.block_type, pf.filename,
         coalesce(c.heading_path, array[]::text[]) as heading_path,
         c.chunk_index
  from public.document_chunks c
  join public.project_files pf on pf.id = c.file_id
  where c.file_id = p_file_id
    and c.page_start <= p_page and c.page_end >= p_page
  order by c.chunk_index;
$$;
```

## Risks & mitigations

| Risk | Mitigation |
|---|---|
| Verifier hallucinates inversions on correct answers (extra retrieval round, latency cost) | Fail-open on any error/non-JSON → ok=true. One nudge per turn cap. Verifier prompt explicitly says "Du sollst NICHT nach Vollständigkeit prüfen". The 4-failure-mode rubric is narrow. Live trace audit in §G6 catches over-firing. |
| Rater Q-type CUE rules fire on questions that are actually point-shaped despite plural noun (e.g., "Welche Phase wird angefragt?" — singular answer expected from plural-form question) | CUE-A explicitly examples "welche Bauherren / welche Termine / welche Drittprojekte / welche Elemente / welche Mandate". "Welche Phase" matches but Q1 is the existing UAT case — re-classify as `aggregation` and the existing rubric still passes if there are ≥3 phases or "only Phase X" verification. UAT will catch overfire; rubric tightens iteratively. |
| Chat-side "don't sum" rule causes refusal on legitimate-aggregation answers where adding 5 numbers IS expected | Rule is scoped to "Total-/Summen-Fragen". Per-Etappe enumeration ("Etappe 1: 1.9 Mio, Etappe 2: 16.4 Mio, …") still works — that's listing, not summing. Only the act of producing a Total from sub-rows is forbidden. |
| Verifier fires on every turn, doubles Gemini cost | Skipped on phrase/out_of_scope/empty chunks. Skipped if verifier_already_nudged. Cost cap: 1 extra Gemini call per turn that has chunks. Acceptable — it's the bug-fix layer. |
| G5 role-policy makes the model surface every named person on every "wer ist X" question (over-recall) | Role-policy rule scoped to "Rollen-Familie" — model has to match Projektleiter ↔ Teilprojektleiter ↔ Projektkoordinator etc. Random Bauleiter from another section don't match. Worst case: extra named persons surface; user can disambiguate. |
| F8 page-neighbors blows context budget on large pages | 30-chunk hard cap stays; rater's `_RATER_MAX_CHUNKS=30` caps rendering. |
| Migration 0017 conflict with 17.4's 0016 | 0017 is additive (one new function, no shape change to existing functions). 0016 must land first; 0017 builds on the heading_path/chunk_index columns. |
